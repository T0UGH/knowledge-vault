# OpenClaw 长任务执行与授权交互调研

## 背景

今晚这轮调研，核心问题不是单点工具，而是围绕一组连续问题展开：

1. 交互式 CLI 怎么处理更稳
2. `tmux` 适不适合菜单选择/TUI
3. `sudo` / 密码 / macOS 图形授权怎么处理
4. agent 怎么稳定执行长任务，尤其是能跑一整晚、至少 12 小时
5. 更大的问题：agent 不应该半路停下来频繁问用户
6. 对 OpenClaw 来说，应该接外部 workflow engine，还是优先把现有 subagent 体系补强

这篇笔记把今晚的判断统一收口。

---

## 一、交互式 CLI 的基本判断

### 结论

交互式 CLI 不应该只被看成“一条命令”，而应该区分：

- **短输入型交互**：一次 token / `y/n` / 一个 prompt
- **选择型/TUI 交互**：菜单、方向键、Tab、Enter、Space、多轮回退
- **持续长会话**：任务会运行很久，中途可能还有输入

### 默认工具选择

- **短输入型交互** → `exec + pty`
- **选择型/TUI/长交互** → `tmux`
- **长期后台交互式任务** → `tmux`

### 为什么 tmux 重要

`tmux` 真正解决的不是“让命令继续跑”，而是：

- 会话不中断
- 可以持续观察屏幕状态
- 可以发方向键、Tab、Enter 等按键
- 可以 detach/reattach
- 适合隔一段时间回来继续控制

对于菜单选择类 CLI、TUI 安装器、Claude Code 这种持续对话式终端，`tmux` 明显比一次性命令稳。

---

## 二、sudo / 密码 / 远程交互的判断

## 结论

真正的问题不是“tmux 能不能输入密码”，而是：

- 什么时候适合代输入
- 什么时候该工程化收口
- 如何避免长任务中途因为 sudo 卡住

### sudo 票据是最直接的临时方案

预制 sudo 票据：

```bash
sudo -v
```

长任务可续命：

```bash
sudo -v
( while true; do sudo -n true; sleep 30; done ) &
```

这适合：

- 一段时间内连续执行 root 操作
- 安装/运维类长流程
- agent 远程推进时减少中断

### 更长期的正确方向

不是让 agent 每次都想办法处理密码，而是：

- 高频固定 root 动作 → 做脚本封装
- 通过 `sudoers` 白名单免密放行少量可控脚本
- 不要给 `NOPASSWD: ALL`

也就是：

> 不要把“给 agent 全量 sudo”当方案，而要把“允许 agent 做的 root 动作”收口成明确原语。

## 三、macOS 授权弹窗的本质判断

### 结论

很多 macOS 授权问题，本质不是“密码输入”，而是：

- GUI 系统弹窗
- TCC 权限（Accessibility / Screen Recording / Automation / Full Disk Access）
- Keychain / 系统安全模型

### 区分两类问题

#### 1）terminal password
例如：
- `sudo`
- 普通 CLI asking for password

这种本质还是终端交互，可通过 `tmux` / `pty` 处理。

#### 2）macOS GUI 授权弹窗
例如：
- 系统权限弹窗
- 钥匙串弹窗
- 安装 helper tool 弹窗
- Touch ID / 系统设置级授权框

这种不是 shell stdin 问题，而是 GUI 授权流程。

### 正确方案

对于 macOS 图形授权，更靠谱的办法不是反复输密码，而是：

- **TCC 预授权 / 预配置**
- 高使用频率权限一次性放行
- 通过配置管理减少重复弹窗

### 一个关键修正

如果实际是通过 **Feishu channel** 驱动 OpenClaw，并不意味着应该给 Feishu 本身授权。

真正需要识别的是：

> **本机上实际执行和控制行为的权限主体是谁。**

也就是要区分：

- 消息入口（Feishu channel）
- 实际执行链（OpenClaw / runtime / shell / tmux / helper process）
- 真正触发 TCC/GUI 权限的 app / process

所以不能拍脑袋说“给 Terminal 开权限”就够了，应该在真实任务里识别实际权限主体。

## 四、“跑一整晚”这个问题的真正答案

### 先澄清

“长时间执行”至少有三种不同问题：

1. **进程别死**
2. **会话别断**
3. **任务别因为 agent 询问用户而阻塞**

前两个很好处理，第三个才是真正难点。

### 对“跑一整晚”的简单答案

- **交互式长任务** → `tmux`
- **纯非交互脚本** → `nohup`
- **长期常驻后台服务** → `launchd`（mac）

### 但这还不够

因为真正的问题是：

> agent 经常不是因为进程死了停下来，而是因为自己停下来问用户。

这说明“保活进程”只能解决表层问题，不能解决长期执行的主问题。

---

## 五、对“agent 半路停下来问用户”的调研结论

### 结论

要减少 agent 中途打断用户，真正有效的方案有三层：

1. **预先定义自动执行边界（approval policy / autonomy policy）**
2. **要求 agent 先推进可做部分，再汇总 blockers**
3. **真正需要用户决策时，进入可恢复状态，而不是阻塞等待**

### 关键判断

问题的根因通常不是：
- agent 不能跑

而是：
- 不知道哪些动作默认允许
- 不知道哪些风险该暂停
- 不知道遇到不确定时该继续还是停止

### 正确的行为模式

错误模式：
- 任务运行中遇到一点不确定就停在那等用户

更好的模式：
- 先完成不依赖用户决策的部分
- 把阻塞点结构化汇总出来
- 等用户回复后再恢复执行

也就是把：

- **阻塞式等输入**

改成：

- **异步交还控制 + 可恢复继续**

## 六、外部开源项目的判断

这轮调研里看了几个相关方向：

### 1）LangGraph
优点：
- durable execution
- persistence/checkpoint
- interrupt/resume
- 很贴合“agent 不要阻塞等用户”这个问题

判断：
- 是行业里很贴脸的正解之一
- 适合做可恢复 agent workflow

### 2）Temporal
优点：
- durable workflow
- 长任务编排
- 强恢复、重试、崩溃恢复

判断：
- 更像底层基础设施
- 太重，不适合作为 OpenClaw 当前第一步

### 3）AutoGen / CrewAI / OpenHands / AutoGPT
判断：
- 各有价值，但都不是最贴“12 小时长期执行 + 避免阻塞用户”的核心答案
- 真正的关键不在 agent 会不会做，而在任务运行时有没有可靠的 pause/resume 和 autonomy 边界

### 总结

如果只从“行业已有开源方案”角度看：

- **最贴脸的 agent workflow 方向**：LangGraph
- **最硬核的 durable orchestration 底层**：Temporal

但它们不一定是 OpenClaw 现阶段最优路线。

## 七、对 OpenClaw 的最终判断

### 结论

现阶段更合理的方向，不是急着把 LangGraph / Temporal 接进来，而是：

> **优先把 OpenClaw 现有的 subagent / session / thread 体系补强，让 subagent 真正成为长期执行单元。**

### 为什么

OpenClaw 其实已经有：

- session
- subagent
- messaging
- background execution
- thread 概念

缺的不是全新框架，而是：

- 长任务执行纪律
- 默认 autonomy policy
- 阻塞升级条件
- 轻量状态快照
- 主会话 / subagent 的边界

### 关键判断

真正的问题不在“subagent 能不能跑久”，而在：

1. subagent 太容易回头问主会话
2. 没有统一的“继续推进优先”策略
3. 缺少标准化 handoff / state snapshot
4. 主会话和 subagent 边界不够硬

### 更合理的 OpenClaw 内部路线

#### 主会话负责：
- 接收用户目标
- 定义边界
- 创建 subagent
- 只处理真正需要人拍板的问题

#### subagent 负责：
- 执行长任务
- 自行探索、重试、继续推进
- 分阶段汇报，而不是频繁打断
- 只有遇到真正 blocker 才升级

### 应该补的能力

1. **长任务默认下放到 subagent**
2. **定义默认 autonomy policy**
3. **收紧升级给主会话的条件**
4. **要求阶段性汇报，而不是阻塞式提问**
5. **维护轻量状态快照**
6. **优先 steer 现有 subagent，而不是每次重开**

### 一个轻量但足够有用的状态结构

长任务 subagent 至少应维护：

- goal
- current state
- just finished
- blockers
- next action
- done criteria

这已经足够支撑：
- 隔夜继续
- 换 agent 接手
- 主会话审阅当前状态

## 八、今晚的整体结论

### 1. 交互式 CLI
- 选择型/TUI/长交互 → `tmux`
- 短 prompt → `exec + pty`

### 2. sudo
- 临时连续 root 操作 → `sudo -v` + 票据续命
- 高频固定 root 操作 → 脚本 + `sudoers` 白名单

### 3. macOS 授权
- 真问题是 TCC/GUI 权限，不是单纯密码输入
- 应识别真实权限主体，而不是把消息入口误认为授权主体

### 4. 长任务执行
- “跑一整晚”表面上是进程保活，真正难点是 agent 不要中途阻塞等用户

### 5. 行业方案
- LangGraph / Temporal 是行业里的强方案
- 但对 OpenClaw 当前阶段来说，外接它们不一定比先补强 subagent 更划算

### 6. 对 OpenClaw 的推荐路线

> **先把 OpenClaw 现有 subagent 做成长期执行 worker，而不是先引入新的 workflow engine。**

这是当前收益/成本比最高的演进方向。

---

## 后续建议

如果下一步继续推进，最值得做的不是再泛泛研究框架，而是直接写一版：

### 《OpenClaw subagent 长任务执行协议（MVP）》

应该至少包含：

- 什么时候必须下放 subagent
- subagent 什么情况下不应询问用户
- 什么时候才允许升级回主会话
- 阶段性汇报格式
- 轻量状态快照结构
- steer 既有 subagent 的默认策略

这会比继续泛搜框架更接近真实落地。
