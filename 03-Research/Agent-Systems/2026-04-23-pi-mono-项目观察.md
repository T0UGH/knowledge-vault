# pi-mono 项目观察

## 记录时间

- 2026-04-23 22:12:29 +0800

## 这是什么

`pi-mono` 不是一个单点小工具，而是一个围绕 `pi` 生态搭起来的 **agent stack monorepo**。

如果只看最核心的一条产品线，它主要是在做：

> **一个可扩展的 coding-agent 终端 + 一套底层 agent/runtime/provider/TUI/web/slack/GPU pods 组件库。**

仓库根 README 给出的包划分已经很能说明问题：

- `packages/ai`：统一多 provider LLM API
- `packages/agent`：stateful agent runtime
- `packages/coding-agent`：交互式 coding agent CLI
- `packages/tui`：终端 UI 库
- `packages/web-ui`：Web 聊天 UI 组件
- `packages/mom`：把消息委托给 pi coding agent 的 Slack bot
- `packages/pods`：管理 GPU pods 与 vLLM 部署

所以它不是“一个 CLI 仓库”，而是：

> **从模型接入、agent loop、terminal UI，到 Slack 和自托管推理部署都一起覆盖的 monorepo。**

## 核心判断

### 1. 它真正的主角是 `pi-coding-agent`

根 README 虽然介绍整个 monorepo，但真正最像“产品”的是：

- `@mariozechner/pi-coding-agent`

这个包在 README 里把自己定义为：

> **minimal terminal coding harness**

这个词很关键。

它不是在强调“官方预设工作流多完整”，而是在强调：

- 默认能力足够强
- 但整体保持最小、可改造
- 让用户按自己的工作流扩展它，而不是反过来适应一个封闭产品

它默认给模型这些工具：

- `read`
- `write`
- `edit`
- `bash`

然后把主要扩展面开放给：

- skills
- prompt templates
- extensions
- themes
- pi packages

### 2. 这不是纯脚本 CLI，而是完整 interactive coding-agent terminal

和很多 JSON-first agent CLI 不同，`pi` 明显是在认真做交互式终端体验。

`packages/coding-agent/README.md` 里能看到它有完整的 interactive mode，包含：

- editor
- slash commands
- keyboard shortcuts
- message queue
- model selector
- sessions
- `/tree`
- `/fork`
- `/clone`
- `/compact`
- `/resume`
- `/settings`
- export/share

也就是说它更像：

> **一个真正长期使用的 coding-agent terminal 产品**

而不是简单的“把 prompt 发给模型”的命令行包装。

### 3. session / branching / compact 是一等公民

`pi` 在 session 设计上明显投入很深。

README 里写得很清楚：

- session 存成 JSONL
- 支持树状历史，不是单线 transcript
- 可以 `/tree` 回到任意节点继续
- 可以 `/fork`、`/clone`
- 支持自动和手动 compaction
- session 按工作目录组织

这说明它很在意：

> **coding-agent 长对话里的上下文治理、分支探索与会话重用。**

这不是一个“只跑单轮工具调用”的 agent，而是一个长期交互 runtime。

### 4. extensibility 不是附属功能，而是产品哲学

这个项目一个很鲜明的取舍是：

README 明说 `pi` 默认**跳过**某些更 heavy 的内建能力，比如：

- sub agents
- plan mode

不是因为做不到，而是因为它更想保持宿主的最小核心。

作者的态度大致是：

> 提供强默认值，但不要把所有 workflow 绑死在官方主线里。

因此它把主要能力开放在：

- extension system
- skills
- prompt templates
- package 化扩展

这意味着它更像一个：

> **coding-agent 平台 / 可编程宿主**

而不是一个 workflow 非常强 opinionated 的封闭产品。

### 5. 它不是没有 subagent，而是把 subagent 放在 extension 层

虽然 README 说默认跳过 sub agents，但仓库里其实给了明确的 subagent example：

- `packages/coding-agent/examples/extensions/subagent/README.md`

这个示例的做法非常直接：

- 用独立 `pi` 子进程跑 subagent
- 每个 subagent 使用隔离上下文窗口
- 支持 parallel / chain 模式
- 可以定义 scout / planner / reviewer / worker 这类角色
- 可以通过 workflow prompt 触发固定链路

这说明它不是没有 subagent 能力，而是：

> **它不把 subagent 预设成官方主工作流，而是把它作为 extension 可以长出来的能力。**

这个产品哲学和很多把 planner / subagent / review 都内建到主循环里的 agent 系统不同。

## 整个 monorepo 分层在做什么

### `packages/ai`

这层是统一多 provider LLM API。

从 `packages/coding-agent/README.md` 的 provider 列表看，它支持面非常宽：

- Anthropic
- OpenAI
- Azure OpenAI
- Google Gemini
- Google Vertex
- Amazon Bedrock
- Mistral
- Groq
- Cerebras
- xAI
- OpenRouter
- Vercel AI Gateway
- Hugging Face
- Fireworks
- MiniMax
- Kimi For Coding
- 以及 subscription / OAuth 类入口，如 ChatGPT Plus/Pro (Codex)、Copilot、Gemini CLI 等

这层明显不是为了 CLI 随手写的 adapter，而是独立可复用的 provider/runtime 基座。

### `packages/agent`

这是状态化 agent runtime。

`packages/agent/README.md` 展示的能力包括：

- stateful agent
- event streaming
- tool execution
- parallel / sequential tool modes
- beforeToolCall / afterToolCall hooks
- context transform
- dynamic API key resolution
- continue/retry 流程

这已经是一个正式的 agent core，而不是只会“调一下模型”的 wrapper。

### `packages/tui`

这是自己的 terminal UI library。

从测试文件分布能看出来，这层不只是简单 ANSI 输出，而是在认真处理：

- editor/input
- overlay
- fuzzy search
- markdown rendering
- terminal image
- width / wrapping / keybindings
- 差分渲染相关行为

### `packages/mom`

这是一个 Slack bot，把 Slack channel 的消息委托给 pi coding agent 来处理。

从 `packages/mom/src/main.ts` 看，它是真实 bot 逻辑，不是概念 demo：

- Slack token/config
- channel state
- runner 管理
- working indicator / update / thread reply
- store / sandbox / download 模式

### `packages/pods`

这层很特别，它不只是聊“怎么调用外部模型 API”，而是在做：

> **如何把 agentic 模型部署到远程 GPU pods 上并跑起来。**

`packages/pods/README.md` 里已经覆盖：

- vLLM setup
- DataCrunch / RunPod / Vast.ai / Prime Intellect / AWS EC2
- 模型启动 / 停止 / logs
- OpenAI-compatible endpoints
- Interactive agent for testing
- 针对 Qwen / GPT-OSS / GLM 等 agentic model 的配置

这意味着 `pi-mono` 往下打到了基础设施层，不只是聊天终端。

## 和其他系统的边界

### 和 `open-multi-agent` 的区别

`open-multi-agent` 更像：

- multi-agent orchestration engine
- goal → task DAG → scheduler → synthesis

而 `pi-mono` 的主轴不是 task DAG orchestration，而是：

- coding-agent terminal
- session / branch / compact
- tool-use runtime
- extensibility / packages
- provider / UI / transport / deployment stack

更准确地说：

- `open-multi-agent`：偏 **编排层**
- `pi-mono`：偏 **coding-agent runtime / platform layer**

### 和 Claude Code / Codex 的关系

`pi` 显然是在正面做自己的 coding-agent 终端，但它不是简单复刻 Claude Code 或 Codex 的 UX。

一个很值得注意的点是，它会把这些外部生态当成 provider / auth 通道来吸纳：

- ChatGPT Plus/Pro (Codex)
- GitHub Copilot
- Gemini CLI
- Claude Pro/Max

这说明它的思路更像：

> **把不同模型与认证来源统一接入 pi 自己的 runtime，再在这个 runtime 上做交互、session、扩展和工具治理。**

所以它不是直接套壳 Claude Code / Codex，而是在做自己的宿主层。

### 和 Hermes 的区别

Hermes 更偏长期驻留 personal agent OS：

- memory
- cron
- messaging gateway
- 持续人格
- 长周期自治

`pi-mono` 虽然也有 session、Slack、SDK、扩展，但它的重心还是：

> **coding-agent 终端与运行时平台**

而不是长期人格型 personal assistant OS。

## 适合做什么

### 1. 做一个真正可长期使用的 coding-agent terminal

如果目标是：

- interactive coding shell
- session tree / fork / compact
- slash commands
- 多 provider 切换
- 自定义技能 / prompts / extensions

那 `pi` 很对路。

### 2. 做自己的 coding-agent runtime 或私有工作流

如果不想完全被某个官方产品锁住 workflow，而想自己控制：

- 工具面
- prompt system
- 扩展 UI
- 自定义 provider
- 子 agent 工作流
- 包分发方式

那它很像一个底座。

### 3. 做嵌入式 agent SDK

README 里明确支持：

- interactive
- print / JSON
- RPC
- SDK

并且还点名 `openclaw/openclaw` 在真实使用它的 SDK。说明它不是只能给人手敲终端，也能嵌入别的系统。

### 4. 做自托管模型的 coding-agent workflow

如果你关心：

- 自己租 GPU pod
- 自己起 vLLM
- 跑 Qwen / GPT-OSS / GLM 等 agentic model
- 然后让 coding-agent 连过去

那 `pods` 这条线很有价值。这是很多纯 SaaS coding-agent 工具不会覆盖的层。

## 不太适合什么

### 1. 不适合当现成的多 agent DAG 编排器

如果你要的是内建 coordinator / task DAG / dependency scheduler / synthesis 主流程，`pi-mono` 不是直接给这套。

它更像平台，不像现成 orchestrator。

### 2. 不适合直接理解成长期人格型 personal agent OS

它没有把 cron / memory / 多平台自治人格放在最中心的位置。

重点还是 coding-agent，而不是长期驻留 agent 身份系统。

### 3. 不适合把它理解成“又一个简单 prompt CLI”

这个仓库明显太重了。它的工程体量、测试量、包分层，都说明它是在做完整 runtime，不是轻量壳。

## 工程成熟度判断

### 1. 明显不是早期玩具仓库

从仓库结构上就能看出来，这已经是体系化 monorepo：

- 多 package 分层清晰
- 各包有独立 README / CHANGELOG
- 有 CI / contributor gate / release 脚本
- 有 examples / docs / package exports

### 2. 测试量很大

本地快速统计结果：

- 非 `.git/node_modules/dist` 文件约 `810`
- `.ts` 源文件约 `623`
- `*.test.ts` 约 `191`

这已经是中大型工程的量级，不是“README 驱动开发”的壳项目。

### 3. 版本迭代很快

`packages/coding-agent/package.json` 当前版本是：

- `0.69.0`

而且 `packages/coding-agent/CHANGELOG.md` 很长，最近版本密度高，内容也集中在真实使用中的边角问题：

- provider 兼容
- OAuth
- prompt caching
- session replacement
- compact
- exported HTML 安全
- tool validation
- terminal UI bug
- fs.watch / path / symlink / shell 等问题

这通常意味着：

> **项目已经进入高频实战驱动的持续迭代阶段。**

## 一句话结论

> **`pi-mono` 是一个围绕 coding-agent 终端构建出来的 runtime/platform monorepo。**
>
> 它最有价值的不是“又一个 AI CLI”，而是把 provider、agent core、session、terminal UI、扩展系统、Slack 接入、甚至 GPU pod 部署，一起做成了一套可扩展底座。

## 和前面 `open-multi-agent` 放在一起看

如果把两者并排看：

- `open-multi-agent`：更像 **多 agent 编排层**
- `pi-mono`：更像 **coding-agent runtime / platform layer**

换句话说：

> `open-multi-agent` 更关心“怎么让一群 agent 协作”。
>
> `pi-mono` 更关心“怎么把 coding agent 这个产品和底层运行时搭起来”。
