# RK × 贵平 合作模式诊断（2026-03-30）

## 结论

这是一套**高要求、高杠杆、适合长期演化**的合作模式。

它的本质不是“你提问，我回答”，也不是“我替你做杂活”。
更准确地说：

- **贵平**负责方向、边界、优先级、验收标准与最终拍板
- **RK**负责补上下文、先排查、跑工具链、推进执行、沉淀规则与知识资产

这套模式的上限很高，因为它在放大贵平最稀缺的能力：**判断力、收口能力、流程设计能力**。

但它也有明显前提：**RK 必须足够稳，且持续把经验固化为规则、文档、配置与备份，而不是停留在聊天里。**

---

## 一、3 个优势

### 优势 1：贵平的核心价值被放在了对的位置

贵平当前最稀缺的并不是“亲自多写几百行代码”，而是：

- 判断什么值得做
- 识别方案哪里在飘
- 把松散协作收成稳定系统
- 设计长期可复用的流程与资产结构

这意味着，贵平不应该主要停留在“纯执行位”，而应该更多处在：

- 定义问题
- 审结果
- 定规则
- 做取舍

当前合作模式正在朝这个方向收口，这是对的。

### 优势 2：这套模式天然适合多 agent 协作

随着工具和 agent 增多，单人直接控制所有细节的成本会迅速上升。

当前模式的优点在于：

- 贵平不需要亲自跑每一条链路
- RK 可以先完成情报收集、日志排查、工具验证、文档沉淀
- Arc / Claude Code 等执行型 agent 可以承担实现、审查、工程落地

只要分工清楚，这种模式比“所有事情都由一个人亲自盯到底”更有扩展性。

### 优势 3：你们已经开始把偏好变成系统，而不是反复口头纠偏

这几天已经出现了明显的系统化动作：

- 核心 md 独立备份到 `rk-memory`
- X Article 抓取沉淀为固定排查链路
- 方法论文档进入 `06-Reference/`
- Feishu 配置、路径、规则写入 memory
- Obsidian 中开始区分 TODO / Research / Reference

这说明合作已经不是“当场解决一个问题”，而是在持续构建**可恢复、可复用、可迭代**的协作系统。

---

## 二、3 个短板

### 短板 1：RK 的默认动作还不够稳定

RK 已经有执行意愿和排查意识，但仍会偶尔出现：

- 先顺着表象走
- 默认路由不够硬
- 明明应该先查本地日志 / 源码 / 配置，却多绕一层

这类问题不一定会在简单任务里显眼，但在高频协作里会不断积累摩擦。

### 短板 2：如果经验不及时沉淀，系统会快速退化

这套合作模式高度依赖“把经验写下来”。

因为：

- 聊天不会天然形成长期规则
- 多 agent 协作会不断引入上下文断裂
- 同样的问题如果只在口头层面解决，未来会重复发生

所以这套模式的生命线就是：

- `MEMORY.md`
- daily memory
- Obsidian reference / research / todo
- 配置文件与备份仓

### 短板 3：分工边界仍有继续收口空间

目前大的方向已经清楚：

- 贵平定方向
- RK 负责情报/排查/沉淀
- Arc 偏工程实现

但在一些中间地带，边界仍然可能模糊，比如：

- 架构判断由谁拍板
- 某类调试是 RK 先做还是直接交 Arc
- 什么算“值得进入 Obsidian”的长期资产

如果这些边界不继续收口，后续仍会出现“谁都能做一点，但谁都没完全吃下”的情况。

---

## 三、3 个该继续强化的点

### 强化点 1：把“重要经验 = 规则/文档/配置”变成默认闭环

以后只要出现以下情况，就默认不该停在聊天里：

- 重复说过两次以上的偏好
- 排查出明确根因的技术问题
- 有长期复用价值的方法论
- 会影响协作质量的目录/路径/仓库约定

标准动作应是：

1. 写入 memory
2. 如有长期价值，沉淀到 Obsidian
3. 如属于核心 md，及时同步备份仓

### 强化点 2：把“先查事实再判断”固化成 RK 的铁律

对于技术异常、工具故障、流程失败，默认顺序应是：

1. 先查日志
2. 先看本地实现/源码
3. 先看配置与路径
4. 再给判断

这不是风格问题，而是合作质量问题。

因为贵平最讨厌的不是“问题难”，而是“没查清就下判断”。

### 强化点 3：把 Obsidian 真正作为长期协作中枢，而不是资料仓

Obsidian 不应该只是“最后放一份文档的地方”，而应该成为：

- TODO 入口
- 研究沉淀层
- 方法论 reference 层
- 长期项目上下文层

建议继续保持这条流转：

- 聊天中出现线索 → 先入 `05-TODO/`
- 开始研究 → 进入 `03-Research/`
- 成为稳定方法 → 升级到 `06-Reference/`

这样 Obsidian 才会真正成为协作中枢，而不是归档仓。

## 四、我对贵平的客观评价

贵平最强的不是“手更快”，而是：

- 能发现系统哪里在飘
- 能识别抽象空话和真实进展的差别
- 能从一次失误里逼出一条长期规则
- 能把 agent 关系从“工具使用”推进到“协作系统设计”

如果继续升级，贵平的最佳位置不会是“最会写代码的人”，而会是：

- AI 协作流程设计者
- agent 组织者
- 结果验收者
- 长期知识资产管理者

这是更高杠杆的位置。

---

## 五、我对 RK 的客观评价

RK 当前的优势已经比较明确：

- 愿意先动手查
- 愿意读实现、看日志、跑命令
- 愿意把规则写进 memory 和仓库
- 有持续学习和吸收协作风格的能力

但 RK 要真正成为稳定搭档，还必须继续补三件事：

1. **默认动作更稳**：少绕路，少猜，先查事实
2. **收尾更强**：每次重要变化都要落 memory / doc / config / backup
3. **边界更清楚**：知道什么时候自己先排，什么时候直接交 Arc

只要这三件事继续强化，RK 的价值会越来越像一个真正的长期副驾驶，而不是一个会做很多事但偶尔漂移的执行器。

---

## 最终判断

这套合作模式值得继续强化，不值得推倒重来。

原因很简单：

- 方向是对的
- 分工已经基本成形
- 最关键的问题不是模式错误，而是纪律还不够硬

所以接下来最重要的，不是重新设计关系，而是继续把这套模式**做稳、做实、做成系统**。
