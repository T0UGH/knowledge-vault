# Agent Super Evolution - 观点4：Skill 为什么比 Workflow 更重要，渐进式披露改变了什么

> [!summary]
> 固定流程式 workflow 并没有消失，但它已经不再是 agent 时代最优的高层能力组织单位。相比之下，**Skill 更接近 agent 原生工作方式**：先定义能力，再让 agent 根据目标动态编排路径。

## 一、Workflow 和 Skill 的根本区别

### Workflow 的核心思想：预先规定路径
Workflow 更像：

- 先做 A
- 再做 B
- 如果失败就走 C
- 成功就走 D
- 最后输出 E

它的本质是：

> **人先把流程图画好，再让系统照着走。**

这很适合：

- 稳定流程
- 规则清晰
- 分支有限
- 可预先穷举的大多数路径

### Skill 的核心思想：预先定义能力，不预先规定完整路径
Skill 更像：

- 定义一类能力模块
- 说明什么时候适用
- 说明怎么调用
- 指出边界和注意事项
- 附上可复用脚本与参考资料

它解决的是：

> Workflow 解决“路径编排”  
> Skill 解决“能力供给”

## 二、为什么固定 Workflow 在 Agent 时代开始不够了

### 1. 模型已经开始具备局部决策能力
以前必须预定义：

- 查哪个接口
- 拿哪些数据
- 怎样拼接结果
- 最后怎么产出

现在强模型已经能：

- 先理解目标
- 再决定查哪里
- 判断哪些信息相关
- 再选择调用哪个工具
- 再决定是否需要拆子任务

这时，如果还把完整路径写死，反而会束缚 agent 最大的优势。

### 2. 现实任务并不是固定流程图能覆盖的
文章中的很多案例，例如：

- 业务链路梳理
- 告警分析
- 抓包提炼技能
- 代码调研
- 旅游比价
- 区块链溯源
- 写长文
- 浏览器自动化修 bug

它们都有共同特点：

> **目标相对明确，但路径非常不固定。**

这类任务天然更适合 skill 化，而不是 workflow 化。

### 3. Workflow 的扩展性差
如果一个能力主要靠 workflow 表达，那么需求一变时通常要：

- 改节点
- 改分支
- 改条件
- 改异常处理
- 改状态流转
- 改上下游依赖

而 skill 化后的扩展方式通常只是：

- 多一个 skill
- 多一个 references
- 多一个脚本
- 多一种触发条件

这让扩展成本大幅下降。

## 三、Skill 真正厉害的地方：把“流程”降级成“能力原子”

文章里有句非常关键的话：

> Skill 的本质是构建原子化能力，Agent 自动根据原子化能力去编排决策。

### 什么是“原子化能力”
它不是定义整个世界，而只是定义：

- 一类事情怎么做
- 适用什么场景
- 需要什么工具
- 有哪些注意事项
- 有哪些可复用资产
- 什么信号下应触发

例如：

- PDF 处理 skill
- Feishu 文档 skill
- GitHub issue/PR skill
- 浏览器自动化 skill
- Obsidian markdown skill
- YouTube 字幕处理 skill

这些都不是完整 workflow，而是 agent 随时可调用的能力块。

### 为什么原子化能力特别适合 Agent
因为 agent 最擅长的是：

- 动态做选择
- 判断当前最适合的能力
- 遇到异常时换路
- 把多个小能力串起来

因此最适合喂给 agent 的，不是“严格按这 18 步做”，而是：

- 这里有若干能力模块
- 每个模块边界清晰
- 什么时候用、怎么用、注意什么都说明白
- 由 agent 根据目标动态编排

## 四、什么是“渐进式披露”，为什么它是关键

文章说：

> Skill 最大的意义是渐进式披露、按需加载，而非全量加载。

### 1. 如果没有渐进式披露
假设有 50 个 skill，如果全量塞进上下文，就会出现：

- token 爆炸
- 模型注意力分散
- 不相关信息被全部注入
- 规则互相干扰
- 模型更容易抓错重点

结果会变成：

> 不是 skill 越多越强，而是越多越乱。

### 2. 渐进式披露的真正含义

#### 第一步：只暴露“这里有哪些能力”
即只暴露：

- name
- description
- 触发条件

这是一个很轻的索引层。

#### 第二步：当某个 skill 被判断为相关时，再加载完整内容
这时才把：

- 详细说明
- 流程约束
- 脚本
- references
- 示例
- 注意事项

真正注入上下文。

#### 第三步：只在当前阶段使用它
而不是永久占据上下文。

### 3. 为什么这会带来根本变化

它改变了上下文的扩展方式：

- 技能数量可以增加
- 只要索引清晰、描述准确，平时不会压垮上下文
- 真正加载的是“当前任务真正相关的那一小块”

> [!important]
> **agent 的能力边界开始可以扩展，而不必线性增加上下文负担。**

## 五、为什么 Skill 比“长 Prompt”高级得多

很多新手仍停留在“写一个超长提示词文档”的阶段，这在小规模时有用，但不适合长期演进。

### 长 prompt 的问题

- 所有东西混在一起
- 很难复用
- 改一个地方要整体改
- 很难按场景分层
- 越写越像垃圾堆

### Skill 的优势

#### a. 可复用
一个 skill 可以在多个任务中复用。

#### b. 可维护
只改这个 skill，不必重写整套 prompt。

#### c. 可组合
多个 skill 可以被同一任务动态组合。

#### d. 可扩展
新增能力时，增加 skill 即可。

#### e. 可治理
skill 是独立目录、独立文档、独立 references，更容易治理。

## 六、Skill 不只是 `SKILL.md`

很多人以为 skill 就是一份 Markdown 说明，实际上那只是入口。

真正让 skill 从“提示词片段”升级成“能力模块”的，是同目录里的这些内容：

- scripts
- references
- templates
- examples
- schemas
- helper tools
- prompts
- docs
- snippets

也就是说：

> `SKILL.md` 负责说明“何时用、怎么用”  
> `references/` 等配套资产负责提供“真正可落地的能力支撑”

## 七、Skill 真正改变的是 Agent 的能力增长方式

以前如果想让系统会一件事，通常要：

- 写一个专门 workflow
- 写一个单独 bot
- 写一段新 prompt
- 造一个特定插件
- 做一套定制化自动化

而 skill 化之后，增长方式变成：

- 抽象出一个能力单元
- 给它清晰命名
- 写好触发说明
- 附上 references / scripts
- 让 agent 在需要时自己调用

> [!tip]
> Skill 的真正意义不是“更优雅”，而是：
> **它让 Agent 的能力积累第一次变得像软件工程，而不是像一次次临时魔改。**

## 八、Workflow 真的就没价值了吗？

也不是。

更准确的判断是：

> **Workflow 没消失，而是下沉了。**
> 它不再适合作为最高层能力组织单位，但仍然会存在于 skill 内部、工具内部、局部子流程内部。

### 仍然适合 workflow 的场景

#### a. 高确定性流程
例如：

- 发布流程
- 审批流
- 文档导出流
- 固定格式转换
- 数据清洗 pipeline

#### b. 技能内部的局部步骤
例如一个 PDF skill 内部，仍然可能包含稳定流程：

- 提取文本
- 判断编码
- 合并文件
- 输出结果

#### c. 高风险操作
例如支付、删除、生产变更等，高风险动作更适合固定流程约束。

### 正确关系应该是

- **Skill 是高层能力组织单位**
- **Workflow 是低层稳定过程封装单位**

## 九、这个观点对当前系统设计的启发

### 1. 应优先沉淀 skill，而不是堆长 prompt
适合 skill 化的事情包括：

- Obsidian 笔记整理
- Feishu 文档处理
- GitHub issue / PR 工作流
- 某类代码库链路调研
- 某类文章抓取与整理
- 浏览器抓包与接口抽取
- mailbox 处理协议
- 某类知识沉淀流程

### 2. 最该做的是“能力目录化”
不是按项目组织，而是按能力模块组织，例如：

- 搜索能力
- 写入能力
- 调研能力
- 浏览器能力
- 归档能力
- 总结能力
- 索引能力
- 修复能力

### 3. 要特别重视 skill 的 description 质量
因为渐进式披露能否工作，关键在于：

- name 是否清晰
- description 是否精准
- 触发信号是否明确
- 颗粒度是否合适

如果 description 写得差，agent 根本不会正确触发。

## 十、一句话总结

> [!summary]
> **Workflow 适合规定路径，Skill 适合供给能力；在 agent 已经具备局部决策能力的时代，最该沉淀的不是完整固定流程，而是可按需加载、可动态组合的能力模块。**
