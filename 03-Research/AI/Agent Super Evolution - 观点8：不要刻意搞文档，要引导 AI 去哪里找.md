# Agent Super Evolution - 观点8：不要刻意搞文档，要引导 AI 去哪里找

> [!summary]
> 真正高质量的知识库，不是“事实仓库”，而是：
> **索引系统 + 调研路径系统 + 高价值结论系统。**

## 一、为什么“拼命搞文档”这条路很容易走歪

很多人第一次做 AI 知识库，会天然走向：

- 把所有知识整理成文档
- 把系统介绍写全
- 把接口说明写全
- 把业务逻辑写全
- 把所有历史经验都存进去

这条路短期看起来勤奋，但长期很容易出问题。

### 1. 文档维护成本极高
尤其是在这些场景：

- 代码库一直变
- 接口一直改
- 依赖关系一直动
- 组织边界一直变
- 工具链一直更新

你今天写得再全，三周后也可能有一半不准。

### 2. 过期文档比没有更危险
没有文档时，agent 至少会去查；但如果有一份“看起来完整、实际上过期”的文档，agent 很容易直接信它。

> [!warning]
> **过期文档比空白更危险。**

### 3. “把一切都文档化”是在和变化做无效对抗
工程世界里的很多知识，本来就不是静态的，它们更适合：

- 动态获取
- 现场验证
- 从源码推导
- 从当前状态读取
- 从系统行为观察

## 二、这句话真正想表达什么

> 不要刻意搞文档，要引导 AI 去哪里找。

它不是在说：

- 文档没用
- 知识库没用
- 总结没用

它真正想说的是：

> [!important]
> **知识库最重要的功能，不是取代事实源，而是帮 AI 更快找到事实源。**

这就是成熟知识库和幼稚知识库的核心区别。

## 三、什么叫“引导 AI 去哪里找”

这其实是在强调一种更高级的知识表达方式：

不是直接告诉 AI “答案是什么”，而是告诉 AI：

- 去哪个仓库
- 先看哪个入口
- 搜什么关键词
- 哪个目录更可信
- 哪些文档只是背景
- 哪份配置才是真正生效的
- 哪个日志字段最关键
- 哪个脚本是标准入口
- 哪个服务代码才是事实源头

> **真正高质量的知识，不只是事实本身，更是“接近事实的路径”。**

## 四、为什么“调研路径”比“事实文本”更耐用

### 1. 事实会变，路径往往更稳定
例如：

- “这个 HTTP 请求最后一定打到 X 服务” —— 容易过期
- “从 gateway/controller 入口开始，沿 service → client → downstream trace 这条路径查，再结合日志字段和配置确认” —— 更稳定

路径更接近：

- 方法
- 搜索策略
- 认知骨架

### 2. 路径能让 agent 形成“自主求真”能力
如果知识库只给答案，agent 会依赖现成结论；
如果知识库给的是路径，agent 就会学会：

- 如何判断去哪查
- 如何缩小搜索空间
- 如何验证结论
- 如何在变化环境中重新逼近真相

> [!tip]
> 这其实是从“知识喂养”走向“知识导航”。

## 五、为什么这和 “Code Is Doc” 是同一个思想

所谓：

> **Code Is Doc**

更准确的理解不是“别写文档”，而是：

> **在工程系统里，代码、运行状态、配置、日志，往往才是最终事实源；知识库应该服务于 agent 更高效地抵达这些事实源。**

因此，文档更适合回到这些位置：

- 背景
- 总结
- 索引
- 路标
- 决策记录
- 方法沉淀

## 六、知识库到底应该存什么

如果知识库不是存“一切事实”，那更适合存这四类东西：

### 1. 索引
例如：

- 某类业务去哪个仓库查
- 某类排障从哪个文档开始
- 某类功能主要在哪几个模块
- 某类平台能力对应哪组 skill

### 2. 调研路径 / 方法路径
例如：

- 查 HTTP 链路时先看入口，再看 service，再看 downstream
- 查网站能力时先浏览器抓包，再筛请求，再抽有价值接口
- 查故障时先日志，再代码，再配置，再上下游依赖

### 3. 已沉淀的高价值结论
例如：

- 阶段性架构判断
- 某方案为什么被否决
- 某边界为什么这样定
- 某系统长期稳定成立的设计原则

### 4. 复盘与经验
例如：

- 反复踩坑后的经验
- 某类问题的排查套路
- 某类任务的执行协议
- 某类流程如何从一次性变成可复用流程

## 七、什么不该成为知识库的主内容

### 1. 高频变化的代码事实
例如：

- 某函数现在怎么实现
- 某 API 当前字段结构
- 某 class 当前所在文件

这些应尽量现查源码。

### 2. 大量原始日志和现场数据
它们适合作为附件、调查现场材料，但不适合做长期知识主干。

### 3. 大量未经提炼的对话转储
如果把所有对话、所有中间分析、所有临时总结都扔进去，知识库会迅速退化成垃圾场。

> [!warning]
> 知识库不是聊天记录坟场，而应该是提炼后的资产仓。

## 八、这条观点对 Obsidian / knowledge-vault 的直接含义

`knowledge-vault` 更适合定位成：

> **长期知识资产层，而不是事实镜像层。**

因此它更适合存：

- 研究笔记
- 精读文章
- 方法论
- 项目结论
- 协作协议
- 目录索引
- 调研模板
- 复盘
- skill 提炼前的冷却材料

而不适合变成：

- 代码事实全量镜像
- 日志倾倒仓
- 原始网页抓取垃圾池
- 所有 session 内容的无限转储库

## 九、这条观点对 OpenClaw / Agent 系统设计的含义

如果认同“引导 AI 去哪里找，比直接给答案更重要”，那么系统设计重点就会从：

- 塞更多记忆
- 塞更多文档
- 塞更多静态知识

转向：

- 提供更好的索引
- 提供更好的路标
- 提供更好的 skill 描述
- 提供更好的仓库边界
- 提供更好的检索入口
- 提供更好的调研路径模板

这会让整个系统更稳、更可扩展。

## 十、这是一种更成熟的知识观

幼稚的知识观是：

> 知识 = 一堆被写下来的事实

成熟的知识观是：

> 知识 = 索引 + 方法 + 结论 + 判断边界 + 接近真相的路径

前者会导致知识库越堆越重，后者会让知识库越用越灵。

## 十一、这条观点的边界

### 1. 不是说文档没价值
仍然非常有价值的文档包括：

- 架构设计文档
- 对齐文档
- 方案决策文档
- 研究总结
- 使用手册
- 协作协议

### 2. 也不是说一切都让 agent 自己现查
如果完全没有索引和总结，让 agent 每次从零开始，也会非常低效。

正确方式是：

> **文档负责抽象、索引和总结；现场负责提供最终事实。**

## 一句话总结

> [!summary]
> **一个成熟的知识库，不应该试图取代事实源，而应该持续降低 agent 抵达事实源的成本。**
