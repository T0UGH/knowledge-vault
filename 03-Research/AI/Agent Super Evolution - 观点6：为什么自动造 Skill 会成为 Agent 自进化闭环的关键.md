# Agent Super Evolution - 观点6：为什么自动造 Skill 会成为 Agent 自进化闭环的关键

> [!summary]
> 自动造 Skill 的真正价值，不是“自动生成技能文件”，而是把一次次临时成功，转化成可复用能力；把 agent 从执行器，推进到能自我扩展能力边界的系统。

## 一、为什么“自动造 Skill”会成为一个质变点

### 1. Agent 最大的问题不是不会做，而是“每次都重新学”
很多时候，一个强模型第一次做一件事，已经能做成七八分，例如：

- 调一个陌生网站
- 抓一篇文章
- 处理一个 PDF
- 自动化一个网页操作
- 生成一份格式化文档
- 走一遍某个业务分析流程

问题不在于做不出来，而在于：

> **下次再做类似事情时，它往往还得重新摸索一遍。**

这会带来巨大的重复成本：

- 重复试错
- 重复调 prompt
- 重复解释边界
- 重复查工具用法
- 重复踩同样的坑

### 2. Skill 的意义，就是把“学会一次”变成“以后都会”
所以自动造 skill 的真正价值，不是生成一个 Markdown 文件，而是：

> [!important]
> **把临时问题求解，转化为长期能力资产。**

如果没有这个转换，agent 再强，也只是：

- 会做很多事
- 但每件事都像第一次做

而一旦成功过程被 skill 化，就意味着：

- 第一次是探索
- 第二次开始是复用
- 第三次开始是优化
- 第 N 次开始是平台能力

## 二、为什么这条路比“收集 Prompt”高级得多

很多人做的是一种“伪能力沉淀”：

- 收藏 prompt
- 收藏模板
- 收藏脚本
- 收藏工作流图
- 收藏 AI 用法帖

这当然有一定价值，但和自动造 skill 不是同一层东西。

### Prompt 收藏的局限

- 脱离上下文
- 不知道什么时候触发
- 不知道适用于哪些边界
- 没有工具使用说明
- 没有失败处理
- 没有 references 资产
- 很难让 agent 自己发现并复用

所以 prompt 收藏更像“个人备忘录”。

### Skill 化的本质

一个 skill 真正成熟，至少意味着：

- 有清晰命名
- 有明确描述
- agent 知道什么时候该用
- 里面说明了怎么做
- 附带 references / scripts / examples
- 可以独立维护和迭代

这时它就不只是“人看得懂”，而是：

> **agent 也能把它当成能力对象来调度。**

## 三、自动造 Skill 的真正逻辑：从任务执行，走向能力提炼

文章中的链路大致是：

1. 浏览器自动化 / 抓包 / 调研
2. 找出真正有价值的接口或流程
3. 抽象出稳定模式
4. 调用 skill-creator
5. 沉淀成一个新的 skill

表面上看像是在“自动生成 skill 文件”，但它背后真正发生的是：

> **系统开始从“做事”升级到“理解自己是怎么做成这件事的”。**

### 做成一件事，不等于拥有能力
某次用 agent 成功完成：

- 一个网页自动化流程
- 一篇复杂文章整理
- 一个站点抓取
- 一个数据提取流程

这只说明“这次成功了”，不代表“这个能力已经属于系统”。

### 只有当成功过程被抽象、命名、固化、可触发，能力才真正属于系统
这就是 skill 的意义。

> [!tip]
> 自动造 skill 最厉害的地方，不是省写文档的时间，而是让成功经验完成一次升维：从个案，变成模块。

## 四、为什么这件事一旦成立，就形成“自进化闭环”

可以把它看作一个循环：

### 第一步：Agent 遇到新任务
先靠已有能力去探索和解决。

### 第二步：解决过程中暴露出稳定模式
例如：

- 某网站总是可以这样抓
- 某平台总是这套认证逻辑
- 某类文档总是这套处理方式

### 第三步：把稳定模式沉淀成 skill
变成新的能力模块。

### 第四步：下次类似问题直接调用 skill
不再从零摸索。

### 第五步：新 skill 又让 agent 可以更快攻克更复杂的新任务
于是又产生新的可沉淀能力。

这就形成真正的复利回路。

> [!important]
> 自动造 skill 的真正意义是：让 agent 从“会完成任务”，进化到“会扩大自己的能力地图”。

## 五、为什么这件事最考验工程判断

方向很对，但实现特别容易跑偏。

最典型的错误幻觉是：

> “既然 skill 很重要，那就让 AI 把看到的一切都自动生成 skill。”

这条路几乎一定会把系统搞烂。

### 1. 不是每个成功案例都值得 skill 化
很多任务能做成，只是因为：

- 这次场景刚好简单
- 这次路径碰巧可行
- 这次依赖恰好没变
- 这次输入很特殊
- 这只是一次性需求

如果不做抽象判断，直接自动 skill 化，就会产生：

- 过拟合技能
- 一次性技能
- 脆弱技能
- 重复技能
- 命名混乱技能
- 无法维护技能

### 2. 真正该被 skill 化的，必须具备稳定模式
一个能力至少满足下面条件，才值得沉淀为 skill：

#### a. 高频或高价值
不一定高频，但至少值得。

#### b. 输入输出边界相对清晰
知道：

- 输入是什么
- 输出长什么样
- 成功定义是什么

#### c. 过程里存在可复用模式
不是每次都完全不同，而是有稳定骨架。

#### d. 有明确工具依赖
例如：

- 一定会用某个 CLI
- 一定会用浏览器
- 一定会调某类 API
- 一定会读某种文件

#### e. 有足够稳定的 references
例如：

- 脚本
- schema
- 示例
- 文档链接
- 已验证路径

## 六、为什么“skill-creator”这种元技能特别重要

文章提到官方的 skill-creator，这件事特别关键，因为它意味着：

> **skill 本身也开始被 skill 化。**

### 1. 什么是元技能
元技能不是直接解决业务问题，而是帮助系统构建新能力。例如：

- skill-creator
- 总结为知识库的 digest skill
- 把临时经验沉淀成 rule 的 skill
- 自动索引与归档 skill

### 2. 为什么元技能是系统成熟的标志
如果系统只有业务技能，它只是一个功能集合。

如果系统开始出现元技能，说明它已经开始关注：

- 自己如何成长
- 自己如何沉淀
- 自己如何治理
- 自己如何扩展能力边界

这就有了“系统自我演化”的味道。

## 七、自动造 Skill 真正难的不是“生成”，而是“抽象”

很多人会把自动造 skill 理解成“让模型写 `SKILL.md`”。

但写文件其实是最不难的一步。真正难的是前面的抽象工作：

- 哪个模式值得抽象？
- 抽象粒度多大合适？
- 应该叫什么名字？
- 它和已有 skill 是否重叠？
- 触发条件怎么写？
- 边界到哪里？
- 哪些 references 要一起沉淀？
- 哪些步骤该让 agent 自由判断，哪些必须写死？

> [!warning]
> **自动造 skill 最难的地方不是生成技能文件，而是判断什么值得变成技能。**

## 八、这个观点对当前系统的直接启发

### 1. 现在最缺的，不是更多单次成功，而是把成功沉淀下来
当前系统里已经有很多成功个案，例如：

- mailbox 方案收口
- HappyClaw 落地
- OpenClaw 长期运行
- Obsidian 主仓建立
- 多 agent 分工
- 知识整理
- GitHub / Feishu / 浏览器自动化 / 文档沉淀

现在真正高杠杆的，不是继续增加更多单次尝试，而是把其中最稳定、最值钱的模式 skill 化。

### 2. 更适合做“元技能体系”
特别值得沉淀的，不一定只是业务技能，而是这些：

- 从临时成功中提炼 skill 的 skill
- 从会话中萃取知识入 vault 的 skill
- 从多 agent 协作结果中抽取 protocol 的 skill
- 从排障过程生成可复用排障模板的 skill
- 从新网站探索中生成站点能力模块的 skill

### 3. knowledge-vault 可以作为 skill 提炼的“冷却池”
更成熟的方式不是每次成功都立刻变 skill，而是：

#### 第一步：先把成功案例写进 vault
沉淀：

- 过程
- 输入
- 输出
- 坑点
- 关键判断
- references

#### 第二步：观察多次是否出现重复模式
如果同类问题一再出现，而且模式稳定，再考虑 skill 化。

#### 第三步：再由 skill-creator 或人工把它工程化
这比“当场自动造 skill”稳得多。

## 九、自动造 Skill 的边界和风险

### 1. 过度 skill 化
如果一切都 skill 化，会变成：

- 技能太多
- 粒度太碎
- 描述混乱
- agent 不知道该调哪个
- 维护成本爆炸

### 2. 伪技能泛滥
很多生成出来的 skill 本质只是：

- 一次性 prompt
- 一个 case 的步骤记录
- 没有边界的散文说明
- 没有 references 的空壳

这不是真 skill。

### 3. 缺乏治理会迅速腐化
自动造 skill 一旦开始，就必须配套：

- 命名规范
- 目录规范
- 去重机制
- 版本管理
- 评审机制
- 废弃策略

否则技能体系会比代码库腐烂得更快。

## 十、一句话总结

> [!summary]
> **一个 agent 系统真正开始产生复利，不是因为它会做更多任务，而是因为它能把做成的任务不断提炼成新的能力模块；自动造 Skill，本质上是在把“经验”转化为“系统能力”。**
