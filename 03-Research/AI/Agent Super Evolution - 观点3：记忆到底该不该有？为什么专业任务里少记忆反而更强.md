# Agent Super Evolution - 观点3：记忆到底该不该有？为什么专业任务里少记忆反而更强

> [!summary]
> 记忆不是越多越好，而是要看任务类型。对专业任务来说，错误的记忆常常比没有记忆更危险；对长期协作来说，没有记忆又会带来巨大摩擦。问题从来不是“要不要记忆”，而是：
> 
> **什么该记、什么不该记、什么时候该现查、什么时候该复用。**

## 一、为什么“记忆”容易被神化

很多人会天然觉得：

- 一个搭档越好用，是因为他越懂我
- 一个人越强，是因为记得越多
- 所以 Agent 也应该“记住越多越强”

这个直觉在社交协作里成立，但在工程任务里只成立一半。

因为工程任务更关心的是：

- 当前事实是否准确
- 当前环境是否变化
- 当前代码是不是还那个状态
- 当前接口是不是已经改过
- 当前问题是否真的和上次是同一个问题

记忆最大的风险是：

> [!warning]
> **它会让模型误以为自己“已经知道了”，从而减少重新查证。**

## 二、为什么专业任务里“没有记忆反而更强”

### 1. 专业任务追求的不是“连续感”，而是“真实性”

在专业任务里，模型最重要的不是像老朋友一样聊天，而是：

- 判断准
- 调研准
- 行动准
- 引用准
- 不偷懒
- 不基于旧事实自信发挥

所以“记忆少”反而会逼模型重新看现场。

这很像一个成熟工程师排障时的做法：

- 不先相信昨天的结论
- 先看现在的日志
- 先看当前代码
- 先看当前配置
- 先确认现场是不是仍然成立

> [!important]
> **现场优先于记忆。**

### 2. 记忆会污染专业判断

#### 副作用一：旧结论挤占当前事实
模型会更倾向于利用现成记忆，而不是重新调研，结果可能：

- 套用旧理解
- 错把相似问题当成同一个问题
- 用历史路径解释新现象
- 引用已经过时的代码结构

#### 副作用二：记忆制造“假熟悉感”
模型会表现得像：

- “这个我知道”
- “以前见过”
- “这应该就是那套逻辑”

但实际上，它可能只是抓住了某个相似 pattern，并没有真正验证。

## 三、为什么“给调研路径”比“给答案”更强

文章里有一句很值钱：

> 不要刻意搞文档，要引导 AI 在哪里找，这个更重要。

### 两种知识供给方式

#### 方式 A：直接给答案
例如：

- 这个服务就是这么设计的
- 这个接口一定走这条链路
- 这个逻辑就在某个模块里

#### 方式 B：给调研路径
例如：

- 先看哪个仓库
- 先搜哪个关键词
- 从哪个 HTTP path 入口查
- 哪份文档只是辅助，代码才是真相

更成熟的方式是 B。

### 为什么“给路径”更有价值

因为路径比答案更耐变化。

答案会过期，路径通常更稳定。

如果告诉 agent：

- “这个接口一定是 A 服务调 B 服务再调 C 服务”

这是一条可能很快过期的静态事实。

但如果告诉 agent：

- “从入口 controller 开始查，顺着 service → rpc client → downstream 调用链往下走，再结合日志字段和配置文件确认”

这是一条更稳定的方法路径。

> [!tip]
> 给答案是在喂鱼，给路径是在教捕猎。

## 四、“Code Is Doc”到底意味着什么

这句话不能机械理解成“文档没用”。

更准确的理解应该是：

> **在快速变化的工程环境里，代码是最终事实来源；文档和记忆只是辅助索引。**

这意味着：

- 文档的作用不是替代代码
- 记忆的作用不是替代调研
- 它们真正的价值是：
  - 缩短定位路径
  - 减少搜索范围
  - 更快接近真相

## 五、那是不是说记忆就不重要？

不是。

这篇文章真正成熟的地方，不是“反对记忆”，而是：

- **专业任务里，不要让记忆取代调研**
- **长期协作里，记忆非常重要**

这两句话必须同时成立。

## 六、哪些东西适合记忆，哪些不适合

### 1. 适合长期记忆的内容：偏“协作结构”

这些内容通常稳定、不容易过时，而且能显著减少协作摩擦。

#### a. 用户偏好
例如：

- 回复风格要简洁
- 哪类产出要同步进知识仓库
- 遇到“启动 Claude Code”默认走 tmux

#### b. 已经反复踩过的坑
例如：

- 某个启动方式会丢环境变量
- 某个项目在某台机器上 Bun 路径有坑
- 某个 group execution mode 会误触发 Docker 依赖

#### c. 高层决策与边界
例如：

- mailbox phase1 先不单独开某个适配层 repo
- 主会话注入只走主 session
- knowledge-vault 是统一主仓

#### d. 协作协议
例如：

- MT 负责收口与判断
- Hex 负责执行与排查
- 哪类任务先走哪个 agent

### 2. 不适合长期硬记的内容：偏“高变化事实”

这些内容最容易过时，应尽量现查。

#### a. 具体代码实现细节
- 某个函数现在怎么写
- 某个类在哪个文件
- 某个 API 当前返回结构是什么

#### b. 当前运行状态
- 现在哪个进程在跑
- 某个端口是不是在监听
- 当前数据库里某个字段值是多少

#### c. 高频变化的业务事实
- 某个需求现在走哪条链路
- 某个模块当前归谁管
- 某个接口本周是不是重构了

## 七、为什么“数字人才需要记忆”又成立

文章也强调：数字人才需要记忆，尤其是在日常生活和长期协作场景里。

### 1. 生活型场景的核心不是准确推导，而是长期陪伴
例如：

- 记得你今晚要做什么
- 记得你偏好哪种提醒方式
- 记得你之前想继续研究哪篇文章
- 记得哪些事情要之后跟进

这些都不是“去代码里现查”的问题，它们本质上就是长期协作连续性。

### 2. 这也解释了不同 Agent 角色的分工

#### [[Claude Code]] 更适合：
- 专业任务
- 明确目标
- 单 session 高质量执行
- 少记忆，重现场

#### [[OpenClaw]] / HappyClaw 更适合：
- 长期陪伴
- 跨天记忆
- 多渠道接入
- 提醒、跟进、收口
- 长期协作摩擦优化

也就是说：

> [!note]
> 一个偏“专家工作台”，一个偏“长期数字搭档”。

## 八、真正高级的做法：把“记忆”和“调研”分层

成熟做法不是“记住一切”，而是分层。

### 第一层：长期记忆层
记这些：

- 偏好
- 协作规则
- 决策边界
- 高价值坑点
- 稳定协议

### 第二层：知识索引层
记这些：

- 哪类知识去哪找
- 哪个目录可信
- 哪篇文档是入口
- 哪个仓库是事实源头
- 哪些 skill 可用于某类任务

### 第三层：现场调研层
每次现查：

- 当前代码
- 当前进程
- 当前日志
- 当前配置
- 当前 API 状态
- 当前页面结构

> [!important]
> 记忆帮助定位，调研确认事实。

## 九、这个观点对当前系统设计的直接启发

最该坚持的一条原则是：

> **把 Memory 记成“协作协议和关键决策”，不要把 Memory 变成“过期事实仓库”。**

### 应该写进长期记忆的
- 用户偏好
- Claude Code 启动铁律
- HappyClaw / Hex / OpenClaw 分工
- mailbox 阶段边界
- 已拍板的仓库边界
- 常见坑和验证过的启动路径
- 协作风格和输出纪律

### 不应该硬记进去的
- 某个 repo 里当前函数实现
- 某个服务当前代码位置
- 某个接口当前完整逻辑
- 某个网页今天的页面结构

这些都应该让 agent 去查。

## 十、一句话总结

> [!summary]
> **记忆的职责不是替 Agent 做判断，而是减少协作摩擦；事实判断这件事，仍然应该尽可能回到现场。**
