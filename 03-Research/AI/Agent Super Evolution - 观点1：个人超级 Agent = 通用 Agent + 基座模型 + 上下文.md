# Agent Super Evolution - 观点1：个人超级 Agent = 通用 Agent + 基座模型 + 上下文

> [!summary]
> 这一条公式的价值，不在于“新”，而在于它第一次把个人 Agent 的能力来源压缩成了一个足够清晰、足够可操作的工程抽象：
> 
> **个人超级 Agent = 通用 Agent + 基座模型 + 上下文系统。**

## 为什么这条公式重要

它帮助我们避开几个常见误区：

- 误以为某个 Agent 产品本身就是全部能力来源
- 误以为只要模型足够强，一切问题都会自动解决
- 误以为 prompt engineering 就是 agent engineering 的核心

这条公式真正做的，是把 Agent 的能力来源拆成三层：

1. **通用 Agent**：提供行动能力与运行时
2. **基座模型**：提供智力底盘
3. **上下文**：让模型真正“懂你、懂环境、懂目标”

## 一、通用 Agent：让模型从“会说”变成“能做”

通用 Agent 的典型代表包括：

- [[Claude Code]]
- [[OpenClaw]]
- Codex
- Cline
- Gemini CLI
- OpenCode

它们虽然外形不同，但本质上都在做同一件事：

**给模型提供行动能力的运行时。**

### 通用 Agent 提供什么

#### 1. 工具调用能力
让模型可以：

- 读写文件
- 搜索仓库
- 跑命令
- 调浏览器
- 请求 API
- 启动子 agent
- 调度外部系统

这意味着模型从“语言生成器”变成了“行动者”。

#### 2. 执行环境
它决定：

- 模型在哪个目录工作
- 能看到哪些文件
- 是否有 shell
- 是否可持久运行
- 权限边界在哪里

真实世界里，模型不缺“会说”，缺的是“能做”。

#### 3. 编排能力
包括：

- 单轮还是多轮推进
- 一个 agent 跑到底还是拆多个 agent
- 是否支持 hooks
- 是否支持任务切片
- 是否支持定时触发、事件驱动

同一个模型，放在不同 runtime / harness 里，表现可能差异极大。

#### 4. 人机协作界面
例如：

- CLI
- WebUI
- Feishu
- Telegram
- 微信
- PWA

这决定它是“一个工具”，还是“一个长期存在的数字搭档”。

### 为什么“通用 Agent”要单独拎出来

因为这提醒我们：

**不要把能力全部归因于模型。**

同样的模型，放在：

- 裸 API
- Chat UI
- Claude Code
- OpenClaw
- 精心设计的 harness

中，表现完全可能不是一个量级。

## 二、基座模型：决定智力底盘，但不是全部

文章说得很克制：

> 基座模型取决于大模型厂商，我们改不了。

这句话的真正含义是：

**你无法通过工程手段把一个弱模型硬抬成顶级模型。**

### 基座模型决定什么

- 推理深度
- 长任务稳定性
- 规划能力
- 代码 correctness
- 自我修正能力
- 审美能力
- 多步工具使用的鲁棒性

你可以通过 skill、harness、context 把模型用得更好，但很难跨越模型本身的核心上限。

### 这意味着什么

意味着很多问题不能靠“再补点 prompt”解决。

成熟的 agent 工程师应该：

- 先知道当前 SOTA 到哪
- 明白模型边界在哪
- 再围绕这个边界设计系统

### 但模型也不能被神化

虽然模型重要，但它不会自动长出：

- 你的习惯
- 你的目录结构
- 你的仓库规范
- 你的长期偏好
- 你的协作协议

所以模型决定底盘，但**不会自动决定协作质量**。

## 三、上下文：真正能长期积累的个人资产

这部分是整个公式最值钱的地方。

> [!warning]
> 对个人 Agent 来说，真正能长期积累的竞争力，不是模型，也不是框架，而是上下文系统。

### 什么叫上下文

上下文远不只是 prompt，它还包括：

- 角色设定
- 用户偏好
- 仓库结构
- 任务目标
- 规则约束
- 技能目录
- 工具说明
- 环境路径
- 历史决策
- 经验沉淀
- 领域知识
- 文档入口
- 调研路径

它本质上是一套：

**让模型在当前环境中不再像陌生人一样工作的外部信息系统。**

### 为什么上下文这么重要

因为模型本身不会自然积累长期协作记忆。每次新 session 打开时，如果没有外部化上下文，模型就会重新变成：

- 不认识你
- 不懂你的项目
- 不知道你的约定
- 不知道你的历史决策
- 不知道你的输出偏好

所以你必须把这些信息外化成：

- `CLAUDE.md`
- Rules
- Skills
- Memory
- Notes
- Knowledge Vault
- Tool docs

### 为什么说“一切工程化措施本质上都在构建上下文”

因为无论名字怎么变：

- prompt engineering
- rules
- skills
- MCP
- memory
- RAG
- docs

本质都在回答同一个问题：

**如何让模型在做决策时，拥有更完整、更正确、更可执行的信息基础。**

## 四、这条公式背后真正强的地方：它定义了增长路径

这不是一个静态公式，而是一个增长模型。

### 通用 Agent
短期可借用，不必自己从零造运行时。

### 基座模型
短期不可控，你只能选型，不能改造。

### 上下文系统
这是你真正能持续积累的资产：

- 更好的规则
- 更好的 skills
- 更好的知识组织
- 更好的命名
- 更好的流程
- 更好的记忆结构

所以：

> [!important]
> **个人超级 Agent 的真正护城河，不是底层模型，而是你围绕模型构建出来的上下文系统。**

## 五、这个观点对我当前工作的直接启发

结合我自己的工作体系，这条公式几乎可以直接映射为：

### 1. 通用 Agent 层
- [[OpenClaw]]
- HappyClaw
- [[Claude Code]]
- Hex 协作
- 各类 skills / CLI / tmux / browser automation

### 2. 基座模型层
- 不同任务要匹配不同模型能力
- 设计/架构/长程执行对模型要求更高
- 模型差异会显著影响工作流设计

### 3. 上下文层
真正值得长期投资的是：

- rules 收口
- 目录收口
- skills 收口
- memory 收口
- knowledge-vault 收口
- mailbox / heartbeat / inject 等协议收口
- 多 agent 协作边界收口

## 六、这条公式的边界

### 1. 它是高层抽象，不是落地方法
它告诉你“由哪三部分构成”，但没有直接回答：

- skill 粒度怎么定
- 知识库怎么拆
- 记忆怎么治理
- 多 agent 怎么分工

### 2. 它容易低估治理成本
真正难的是：

- 上下文维护
- skill 爆炸治理
- 版本同步
- 安全边界
- 自动化失误止损

### 3. 它默认你本人具备结构化能力
如果一个人不会：

- 定义问题
- 拆任务
- 总结规则
- 抽象复用模式

那即使知道“上下文重要”，也很难构出真正强的上下文系统。

## 一句话总结

> [!summary]
> **个人超级 Agent，不是某个神奇工具，而是你拿最强通用执行器和最强模型，再叠上你自己的规则、技能、记忆、知识组织之后，形成的一套可持续进化的协作系统。**
