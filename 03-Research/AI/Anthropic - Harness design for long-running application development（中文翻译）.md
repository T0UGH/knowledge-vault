# Anthropic - Harness design for long-running application development（中文翻译）

> [!info]
> 原文链接：<https://www.anthropic.com/engineering/harness-design-long-running-apps>
> 
> 作者：Prithvi Rajasekaran（Anthropic Labs 团队）
> 
> 本文为中文整理翻译版，适合在 [[Obsidian]] 中回看与做摘录。

## 引言

在过去几个月里，作者一直在研究两个彼此关联的问题：

1. 如何让 Claude 产出高质量的前端设计
2. 如何让 Claude 在没有人工干预的情况下构建完整应用

这项工作起源于此前两个方向上的探索：

- frontend design skill
- long-running coding agent harness

在这些早期工作中，团队通过 prompt engineering 和 harness design，已经把 Claude 的表现提升到明显高于 baseline 的水平，但两个方向最后都遇到了上限。

为了突破这个上限，作者寻找一种新的 AI 工程方法，希望它既适用于：

- 由主观审美定义的设计任务
- 也适用于可通过正确性和可用性来验证的软件开发任务

作者从 [[Generative Adversarial Networks]]（GAN，生成对抗网络）中获得灵感，设计出一种多 agent 结构：

- **generator（生成者）**
- **evaluator（评估者）**

但要想构建一个能**稳定打分**、而且还带有“审美判断”的 evaluator，首先必须设计出一套评价标准，把像“这个设计好不好？”这样的主观问题，转化成具体可评分的项目。

随后，这些方法又被应用到长时间 autonomous coding（自主编码）上，并继承了此前 harness 工作中的两点经验：

- 把构建任务拆成更可处理的块
- 用结构化 artifacts 在不同 session 之间传递上下文

最终形成的是一个 **三 agent 架构**：

- planner
- generator
- evaluator

这个架构可以在持续数小时的 autonomous coding session 中，产出内容丰富的全栈应用。

## 为什么朴素实现会失效

团队此前已经展示过：对于长时间运行的 agent 编码任务来说，**harness 设计会显著影响效果**。

在更早的一次实验中，他们使用：

- 一个 initializer agent，把产品 spec 拆成任务列表
- 一个 coding agent，一次实现一个 feature
- 再通过 artifact 在不同 session 之间传递上下文

更广泛的开发者社区也逐渐收敛到了类似认识。比如 “Ralph Wiggum” 方法，就是通过 hooks 或脚本让 agent 保持持续迭代。

但仍然有些问题始终存在。对于更复杂的任务，agent 随着时间推移仍然会逐渐“跑偏”。作者在拆解这个问题时，观察到了两个典型失败模式。

### 1. 上下文越来越长，模型逐渐失去一致性

模型在长任务中，随着 context window 被填满，往往会逐渐失去连贯性。有些模型还会出现一种所谓的 **“context anxiety（上下文焦虑）”**：当它们觉得自己快接近上下文限制时，会开始提前收尾、草率结束工作。

一种有效解法是 **context reset**：

- 彻底清空上下文窗口
- 启动一个新的 agent
- 通过结构化 handoff，把前一个 agent 的状态和下一步计划传给后一个 agent

这样能同时解决：

- 上下文膨胀导致的一致性下降
- 上下文焦虑导致的过早结束

这和 **compaction（压缩）** 不同。压缩是把前面的对话在原地总结，然后让**同一个 agent** 继续工作。虽然 compaction 保留了连续性，但它并没有给 agent 一个真正的“干净起点”，因此上下文焦虑依然可能存在。

作者提到，在之前的测试中，**Claude Sonnet 4.5** 的上下文焦虑非常明显，以至于单靠 compaction 不足以支撑强健的长任务表现，因此 context reset 变成了 harness 设计中的关键部分。

不过这也带来了代价：

- 更高的编排复杂度
- 更多 token 开销
- 更高延迟

### 2. 自我评估能力差

这是团队此前没有充分解决的另一类问题。

当一个 agent 被要求评估**自己刚完成的成果**时，它往往会非常自信地称赞自己的工作——即便在人类看来，这个结果其实很普通。

这个问题在主观任务上尤其明显，比如设计，因为这里没有像软件测试那样的二元可验证标准。一个布局到底是“精致”还是“平庸”，本质上是判断题；而模型在给自己打分时，往往会偏乐观。

但即使在那些**有可验证结果**的任务上，agent 也依然会表现出判断力不足，影响任务完成质量。

因此，把：

- **执行工作的 agent**
- 和 **负责评估的 agent**

拆开，就成为了一个很强的杠杆。

当然，仅仅拆开并不会自动消除“宽松打分”的倾向，因为 evaluator 本身仍然是一个 LLM，它对 LLM 生成内容也天然偏宽容。

但事实证明：

**把一个独立 evaluator 调教成“怀疑型评审”，远比让 generator 对自己苛刻更可行。**

一旦这种外部反馈存在，generator 就有了可以反复迭代的明确目标。

## 前端设计：让主观质量变得可评分

作者先从前端设计开始实验，因为在这个领域，自我评估问题最明显。

如果没有任何干预，Claude 通常会倾向于生成那种：

- 安全
- 可预测
- 功能上没问题
- 但视觉上毫无亮点

的布局。

### 两个关键洞见

#### 第一，审美虽然不能完全压缩成一个分数，但可以借助评分标准来改进

个人审美当然会有差异，但如果使用一组编码了设计原则和偏好的评分标准，结果还是可以被改善的。

例如：

- “这个设计美吗？”——很难稳定回答
- “这个设计是否符合我们定义的优秀设计原则？”——就具体得多

后者会给 Claude 更清晰的评判坐标。

#### 第二，把前端生成和前端评分拆开，能形成反馈回路

只要 generator 和 evaluator 分离，evaluator 的反馈就能持续把 generator 推向更强输出。

### 四个评分维度

作者定义了四条标准，并同时给到了 generator 和 evaluator：

#### 1. 设计质量（Design quality）
设计是否像一个完整的整体，而不是零散部件的拼装？

优秀设计意味着：

- 颜色
- 字体
- 布局
- 图像
- 细节

这些元素组合起来，形成清晰的氛围和身份感。

#### 2. 原创性（Originality）
是否存在明显的“定制化决策”？还是只是模板布局、库默认样式、AI 常见套路的复用？

一个真正的人类设计师，应该能看出其中有意识的创意选择。如果只是照搬 stock component，或者充满典型 “AI slop” 痕迹，比如白卡片配紫色渐变，那这里就不合格。

#### 3. 工艺（Craft）
技术执行质量如何，例如：

- 字体层级
- 间距一致性
- 配色和谐
- 对比度

这更像能力检查，而不是创意检查。大多数实现默认都还可以；如果这里失败，说明连基本功都没做好。

#### 4. 功能性（Functionality）
不看美感，只看可用性：

- 用户能否理解界面在做什么
- 能否找到关键操作
- 能否不靠猜测就完成任务

### 权重设置

作者特别强调了：

- 设计质量
- 原创性

而相对弱化了：

- 工艺
- 功能性

原因是：Claude 在工艺和功能性上默认已经表现得不错。模型通常天然就有足够的技术能力来完成这些部分。

但在设计感和原创性上，它经常只能做出“平庸但不出错”的东西。

因此，这些标准里明确惩罚高度泛化的“AI 味”设计，并通过提高设计与原创性的权重，逼模型承担更多审美风险。

### evaluator 的校准方法

作者使用了 few-shot 示例，并配上详细评分拆解，来校准 evaluator。

这样做的效果是：

- evaluator 的判断更贴近作者偏好
- 不同迭代轮次之间的评分漂移更少

### 实现方式

作者使用 [Claude Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) 搭建这个 loop，因为这样编排起来更直接。

流程如下：

1. generator 根据用户 prompt 先生成一个 HTML/CSS/JS 前端
2. evaluator 带着 Playwright MCP 去实际操作这个页面
3. evaluator 针对每个维度打分，并输出详细批评意见
4. 这些反馈再回流给 generator，作为下一轮迭代输入

在实践中，evaluator 会自己导航页面、截图、仔细研究实现，然后再给出评估。

作者通常会让它迭代 **5 到 15 轮**。每一轮 generator 都会根据 evaluator 的批评，把设计推向更鲜明的方向。

因为 evaluator 不是在看静态截图，而是在**实际操作页面**，所以每轮都需要真实墙钟时间。完整跑下来，最长会到 **4 个小时**。

作者还要求 generator 在每一轮评估后做一个策略决策：

- 如果分数趋势不错，就继续沿当前方向打磨
- 如果这个方向不行，就换一个完全不同的审美方向

### 效果与观察

跨多次实验来看，evaluator 的评分通常会随着迭代而提升，直到平台期，但理论上仍然还有提升空间。

有些生成结果是渐进式优化，另一些则会在某一轮突然发生明显的审美转向。

作者还观察到，评分标准本身的措辞，会以他没有完全预料到的方式影响 generator。

例如，在标准里加入类似：

> 最优秀的设计应该达到博物馆级别

这样的表述后，设计风格就明显朝某种特定视觉收敛，说明评分标准背后的 prompt 语言，实际上直接影响了输出气质。

虽然整体上分数通常会越来越高，但这个曲线并不总是线性上升。在一些案例里，后面的版本整体更好，但作者反而更喜欢中间某一版。

另外，实现复杂度也往往随着轮次增加而提升——因为 generator 会在 evaluator 的批评下，越来越倾向于尝试更大胆、更复杂的方案。

即使是第一轮，相比完全没有 prompt 的 baseline，结果也明显更好。这说明：**仅仅是这些标准和配套语言本身，就已经能在没有 evaluator 反馈之前，先把模型从“泛化默认值”推离出去。**

### 一个典型案例：荷兰艺术博物馆网站

作者曾让模型做一个荷兰艺术博物馆网站。

到第 9 轮时，它已经做出一个相当干净的深色主题虚构博物馆首页，看起来很精致，但大体仍在作者预期之内。

但到了第 10 轮，它完全推翻了之前方案，把网站重新想象成一种**空间体验**：

- 用 CSS perspective 渲染出一个 3D 房间
- 地板是棋盘格
- 画作自由悬挂在墙面上
- 导航方式不再是滚动或点击，而是通过“门洞”在不同展厅之间穿梭

这是作者此前从单轮生成中从未见过的创造性跃迁。

## 扩展到全栈开发

在这些发现基础上，作者把这种受 GAN 启发的模式应用到了全栈开发中。

generator-evaluator 这个循环天然适配软件开发生命周期，因为：

- code review
- QA

本质上就和设计评估器扮演同样的结构性角色。

### 架构设计

在更早的 long-running harness 中，团队已经解决了多 session 编码的一致性问题，方法是：

- 一个 initializer agent
- 一个一次只做一个 feature 的 coding agent
- session 间 context reset

其中 context reset 是关键突破点，因为那套 harness 用的是 Sonnet 4.5，它有明显的 context anxiety。

但 **Opus 4.5** 基本消除了这种行为，因此作者这次完全去掉了 context reset。整套 agent 直接作为一个连续 session 跑完整个构建过程，依赖 Claude Agent SDK 的自动 compaction 去处理上下文膨胀。

在这个基础上，作者构建了一个三 agent 系统，每个 agent 都负责解决过去实验中观察到的一个特定缺口。

#### Planner

过去的 long-running harness 需要用户事先给非常详细的 spec。作者希望把这一步自动化，于是加入 planner agent。

它接受一个 **1 到 4 句话** 的简单 prompt，然后扩展成一份完整产品 spec。

作者要求它：

- 在 scope 上更有野心
- 关注产品上下文和高层技术设计
- 不要过早下到细粒度实现

原因是，如果 planner 一开始就在底层技术细节上判断失误，这些错误会顺着 spec 一路传递到后续实现中。

更聪明的做法是：

- 对最终交付物做约束
- 但把具体怎么实现，留给后面的 agent 在过程中自己决定

作者还要求 planner 主动寻找机会，把 AI 功能融入产品 spec 中。

#### Generator

之前 “一次一个 feature” 的方式在 scope 管理上效果很好，所以这次也沿用了类似模式，要求 generator 以 **sprint** 的方式工作，每次从 spec 中取一个 feature 来实现。

每个 sprint 使用的技术栈包括：

- React
- Vite
- FastAPI
- SQLite（后来换成 PostgreSQL）

generator 还被要求在每个 sprint 结束时做一次自评，再把结果交给 QA。此外它还有 git 做版本控制。

#### Evaluator

此前 harness 产出的应用常常“看起来很厉害”，但实际一用仍然有真 bug。所以 evaluator 的职责就是抓这些问题。

它用 Playwright MCP 像真实用户一样去点击、操作运行中的应用，测试：

- UI 功能
- API endpoint
- 数据库状态

然后根据：

- 发现的 bug
- 一套参考前端实验改造过来的评分标准

来给这个 sprint 打分。

这些标准覆盖：

- 产品深度
- 功能性
- 视觉设计
- 代码质量

每一项都有硬阈值，只要其中任何一项低于阈值，这个 sprint 就判失败，generator 会收到详细反馈。

### Sprint contract：写代码前先约定“完成标准”

在每个 sprint 开始之前，generator 和 evaluator 会先协商一个 **sprint contract**，也就是先约定：

**这个 sprint 里的“完成”到底意味着什么。**

这么做的原因是：产品 spec 被故意保持在高层，不想过早规定实现细节；但与此同时，又需要一个桥梁，把高层用户故事转换成可测试的实现标准。

流程是：

1. generator 先提出：
   - 这轮准备做什么
   - 怎样验证它是否成功
2. evaluator 审核这个 proposal，确认 generator 做的是正确的事情
3. 双方来回迭代，直到达成一致

### Agent 之间如何通信

它们通过文件沟通：

- 一个 agent 写文件
- 另一个 agent 读文件
- 然后在原文件里回应，或者写新文件让前一个 agent 再去读

随后 generator 就按照已经达成一致的 contract 去实现，再把工作交给 QA。

这种方式的好处是：

- 保持实现忠于 spec
- 又避免过早把实现细节写死

## Harness 的实际运行

第一版 harness 使用的是 **Claude Opus 4.5**，并把完整 harness 与单 agent 系统做了对比。

作者之所以选 Opus 4.5，是因为那时它是最强的 coding model。

测试使用的 prompt 是：

> 创建一个 2D 复古游戏制作器，具备关卡编辑器、精灵编辑器、实体行为、以及可试玩测试模式等功能。

下表展示了两种 harness 的运行时间和成本：

| Harness | 时长 | 成本 |
| --- | --- | --- |
| Solo | 20 分钟 | 9 美元 |
| Full harness | 6 小时 | 200 美元 |

完整 harness 的成本高了 20 多倍，但输出质量差异一眼可见。

### 单 agent 结果的问题

作者原本期待的是一个这样的界面：

- 可以构建关卡及其组件
- 包括 sprite、entity、tile layout
- 然后点击 play 真正运行关卡

单 agent 的初始结果看上去似乎符合预期。

但深入点几下后，问题暴露出来了：

- 布局浪费空间，固定高度面板让大部分视口空着
- 工作流很僵硬
- 当用户想给关卡填内容时，系统提示要先建 sprite 和 entity，但 UI 没有引导这个顺序
- 更关键的是：游戏实际上坏了

实体虽然出现在屏幕上，但根本不响应输入。进一步读代码发现，entity 定义和运行时之间的连接是断的，而且界面上完全看不出问题出在哪。

### 完整 harness 的结果

再看 harness 版本：它从同样的一句话 prompt 开始，但 planner 把它扩展成了一份包含 **16 个 feature、分布在 10 个 sprint** 的 spec。

它做出的东西远远超出了单 agent 的范围。除了核心编辑器和试玩模式，还规划了：

- sprite 动画系统
- 行为模板
- 音效与音乐
- AI 辅助的 sprite 生成器
- AI 辅助关卡设计器
- 带分享链接的游戏导出能力

作者还给 planner 提供了 frontend design skill，让它在 spec 中一并定义应用的视觉设计语言。

在每个 sprint 中，generator 和 evaluator 都会协商一个 contract，明确：

- 这轮具体要实现什么
- 如何验证它已完成

### 相比单 agent 的提升

这个应用一打开，就比 solo 版本明显更精致、更顺滑。例如：

- canvas 用满了整个视口
- 面板尺寸合理
- 界面有统一视觉身份
- 风格和 spec 中定义的设计方向一致

当然，solo 里一些笨拙感还是存在。比如工作流依然没有明确告诉用户：先做 sprites 和 entities，再往关卡里填充。这更像是基础模型产品直觉上的短板，而不是 harness 本身主要要解决的问题。

但随着作者继续使用各个编辑器，新版本的优势就更明显了：

- sprite editor 更丰富
- 工具面板更干净
- 颜色选择器更好用
- 缩放控制也更顺手

因为作者要求 planner 主动把 AI 功能织入 spec，应用里还自带 Claude 集成，可以通过 prompt 直接生成游戏的不同部分，这大大加速了工作流。

### 最大差异：play mode 终于真的能玩了

最关键的区别在于试玩模式。

作者真的可以控制自己的实体并开始玩游戏。虽然物理效果还有些粗糙——例如角色跳到平台上时会和平台有些重叠，这在直觉上不太对——但最核心的东西终于工作了，而 solo 版本根本没做到。

当然，用了一会儿后也还是能碰到一些 AI 自动生成关卡的局限。例如有一堵很高的墙，怎么跳都跳不过去，导致流程被卡死。

这说明 harness 还可以在常识判断和边界情况上继续加强。

### Evaluator 是如何发挥作用的

从日志里能明显看出来，evaluator 把实现牢牢拉回了 spec。

每个 sprint，它都会沿着 sprint contract 里的测试标准，用 Playwright 去操作运行中的应用，只要看到任何和预期行为不一致的地方，就会报 bug。

这些 contract 非常细。比如仅仅 **Sprint 3**，就有 **27 条标准** 覆盖 level editor。而 evaluator 报出来的问题又足够具体，generator 不需要额外调查就能直接修。

文中给了几个例子：

#### 例 1
**合同标准：** 矩形填充工具支持点击拖拽填充选中的 tile 区域  
**评估结果：** 失败。工具只在拖拽起点和终点放 tile，并没有真正填满区域。`fillRectangle` 函数虽然存在，但没有在 `mouseUp` 时正确触发。

#### 例 2
**合同标准：** 用户可以选择并删除已经放置的实体出生点  
**评估结果：** 失败。`LevelEditor.tsx:892` 的 Delete 键处理逻辑要求 `selection` 和 `selectedEntityId` 同时存在，但点击实体时只会设置 `selectedEntityId`。条件应该改成 `selection || (selectedEntityId && activeLayer === 'entity')`。

#### 例 3
**合同标准：** 用户可以通过 API 调整动画帧顺序  
**评估结果：** 失败。`PUT /frames/reorder` 路由被定义在 `/{frame_id}` 路由后面，导致 FastAPI 把 `reorder` 当成 `frame_id` 整数解析，返回 422。

### 把 evaluator 调到这种水平并不容易

开箱即用的 Claude，并不是一个好的 QA agent。

在早期实验中，作者经常看到它明明识别出了真实问题，结果又自己说服自己“这不算什么大问题”，最后还是把工作给过了。

它还倾向于浅尝辄止地测试，不太会主动挖边界情况，所以很多隐蔽 bug 都会漏掉。

作者的调参方式是：

1. 去看 evaluator 的日志
2. 找出那些 evaluator 判断与自己不一致的案例
3. 更新 QA prompt，专门修这些偏差

这个循环做了好几轮，evaluator 的评分方式才终于接近作者认为“合理”的水平。

即使如此，这套 harness 的输出仍然暴露了模型 QA 能力的边界：

- 小型布局问题
- 某些交互直觉上不顺
- 深层嵌套功能里还有没测出来的 bug

这些都说明，验证能力还有进一步提升空间。

但和 solo 版本相比——那个版本甚至连应用最核心功能都跑不起来——提升已经非常明显。

## 继续迭代 harness

第一版结果很鼓舞人心，但它也：

- 过于庞大
- 很慢
- 很贵

因此下一步自然是：**在不损伤性能的前提下，尽量简化 harness。**

这背后有一个一般性原则：

> harness 里的每一个组件，都隐含着一个假设：模型自己做不好这件事。

这些假设必须不断被质疑和压力测试，因为：

- 它们可能本来就是错的
- 就算当下成立，也会随着模型进步很快过时

这和 Anthropic 在《Building Effective Agents》中讲的原则一致：

**尽可能先用最简单的方案，只有在必要时才增加复杂度。**

作者一开始尝试过大刀阔斧地删减 harness，并加上一些新的创意想法，但没能复现原始版本的表现。而且这样做会让人很难判断：到底哪些部分是真正承重的。

因此后来改为更系统的方法：

- 一次只移除一个组件
- 观察它对最终结果的影响

### Opus 4.6 发布后的影响

在做这些迭代时，Opus 4.6 发布了。这进一步强化了“减少 harness 复杂度”的动机。

因为很有理由相信：4.6 比 4.5 需要更少的 scaffolding。

Anthropic 在发布博客里提到：

> Opus 4.6 规划更谨慎，能持续做 agentic 任务更久，在大型代码库中更可靠，并且在 code review 和 debugging 上更强，更能发现自己的错误。

它的长上下文检索能力也有明显提升，而这些恰好都是原来 harness 想要补足的能力。

## 去掉 sprint 结构

作者做的第一步简化，就是**彻底移除 sprint 结构**。

sprint 结构原本的作用，是把工作拆块，让模型能够更连贯地推进。但考虑到 Opus 4.6 的提升，作者有理由相信模型现在可能已经可以原生处理这类分解工作，不再需要这种外部拆分。

作者保留了：

- planner
- evaluator

因为这两个依然明显有价值。

如果没有 planner，generator 会显著缩 scope：它会直接拿着原始 prompt 开始写，结果产出的应用就没有 planner 做出来的 spec 那么丰富。

移除 sprint 后，作者把 evaluator 改成 **在整个运行结束后只做一次整体评估**，而不是每个 sprint 都评一次。

因为模型更强了，所以 evaluator 是否值得保留，开始取决于任务是否已经落进模型的“单独完成能力范围”之内。

在 4.5 上，这个边界很近：很多任务都处在 generator 单独做会比较吃力的边缘，所以 evaluator 常常能抓到关键问题。

到了 4.6，这个边界被推远了。原本需要 evaluator 才能做稳的任务，现在很多已经落入 generator 可以自己做好的一侧。

因此：

- 对那些已经在模型能力范围内的任务，evaluator 变成额外开销
- 但对那些仍然在模型能力边界附近的任务，evaluator 依然能提供真实收益

所以一个实际结论是：

**要不要 evaluator，不是固定答案。** 只有当任务超出当前模型“单打独斗”的稳定能力范围时，它才值得这个成本。

### 同时也加强了 AI feature 的构建能力

除了结构上的简化，作者还加了一些 prompting，去提升 harness 在每个 app 中构建 AI 功能的能力——尤其是让 generator 正确构建一个“能通过工具驱动 app 自身功能”的 agent。

这部分也需要反复迭代，因为相关知识太新，Claude 的训练数据覆盖不够充分。但经过足够调参后，generator 已经能够比较正确地搭出 agent。

## 新版 harness 的测试结果

为了测试新版 harness，作者使用了这样一个 prompt：

> 在浏览器中用 Web Audio API 构建一个功能完整的数字音频工作站（DAW）。

这一轮依然又长又贵：

- 大约 4 小时
- token 成本约 124 美元

其中大部分时间都花在 builder 上。即使去掉了 sprint 分解，builder 依然能连续稳定工作 **两个多小时**。

阶段成本大致如下：

- Planner：4.7 分钟，0.46 美元
- Build Round 1：2 小时 7 分钟，71.08 美元
- QA Round 1：8.8 分钟，3.24 美元
- Build Round 2：1 小时 2 分钟，36.89 美元
- QA Round 2：6.8 分钟，3.09 美元
- Build Round 3：10.9 分钟，5.88 美元
- QA Round 3：9.6 分钟，4.06 美元
- 总计：3 小时 50 分钟，124.70 美元

### 运行结果如何

和之前一样，planner 把一句话 prompt 展开成了一份完整 spec。从日志看，generator 在这些事情上做得不错：

- 规划 app
- 规划 agent 设计
- 把 agent 接进去
- 自己先测试一轮
- 再交给 QA

但 QA 依然抓出了真实问题。

在第一轮反馈里，它指出：

> 这是一个很强的应用，设计完成度高，AI agent 扎实，后端也不错。主要失败点在于功能完整性：虽然 app 看上去很厉害，AI 集成也工作得很好，但若干核心 DAW 功能只是展示性质，并没有真正可交互的深度。比如：clip 不能在时间线上拖拽/移动，没有 instrument UI 面板（如 synth 旋钮、鼓机 pads），也没有效果器的可视化编辑器（如 EQ 曲线、压缩器表盘）。这些不是边角问题——它们是让 DAW 可用的核心交互，而且 spec 明确要求了这些能力。

在第二轮反馈里，QA 又抓到几项缺口：

> 仍然存在的问题：
> - 音频录制仍然只是 stub（按钮会切换，但并没有真的采集麦克风）
> - 没有实现 clip 边缘拖拽调整长度，也没有 split clip
> - 效果器可视化仍然只是数字 slider，不是真正图形化的 EQ 曲线

这说明：即使模型更强了，generator 自己还是会漏掉细节，甚至会把功能做成 stub；QA 在补“最后一公里”上依然很有价值。

### 最终产物的水平

根据 prompt，作者原本期待的是一个程序，能够让用户：

- 创作旋律
- 编和声
- 做鼓点
- 把这些排成一首歌
- 并且中间还有一个集成 agent 帮忙

最终结果虽然离专业音乐制作软件还有明显距离，agent 自己写歌的能力也远称不上优秀，而且 Claude 本身也不能真正“听音频”，这使得 QA 在音乐审美层面的反馈闭环不够强。

但最终的 app 已经具备了一套真正可用的音乐制作核心原语：

- 可工作的 arrangement view
- mixer
- transport
- 都能在浏览器中运行

更重要的是，作者确实可以通过 prompt 从头拼出一段歌曲片段：

- agent 设置 tempo 和 key
- 铺旋律
- 做鼓组轨道
- 调 mixer 电平
- 加 reverb

也就是说，最核心的作曲和制作原语已经存在，而且 agent 可以通过工具 autonomously 驱动它们，端到端完成一个简化版音乐制作流程。

## 接下来会怎样

随着模型继续提升，我们大致可以预期：

- 模型能持续工作更久
- 能处理更复杂的任务

这会带来两种看似相反、但实际上同时成立的变化。

### 一方面

随着模型变强，包在模型外面的 scaffold 可能会越来越不重要。有些过去必须靠 harness 才能解决的问题，等下一代模型出来后，也许模型自己就能搞定。

### 另一方面

模型越强，越有空间设计出更强的 harness，让它完成 baseline 根本做不到的复杂任务。

因此，这项工作带来的几个经验值得长期保留：

1. 针对你正在构建的模型，持续做实验
2. 认真读 trace，观察它在真实问题上的行为
3. 持续调它，直到达到你想要的结果
4. 对复杂任务，可以通过拆解任务、给问题的不同侧面配专门 agent，来挖额外收益
5. 新模型出来后，应重新审视 harness：
   - 去掉那些已经不再承重的部分
   - 增加新能力，让模型达到之前做不到的层级

作者最后的核心判断是：

**随着模型进步，有意思的 harness 组合空间并不会缩小。它只是会迁移。** 而 AI 工程师真正有趣的工作，就是不断找到下一种新组合。
