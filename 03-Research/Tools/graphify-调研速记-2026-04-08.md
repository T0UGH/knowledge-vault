# graphify 调研速记

## 结论

**不建议把当前 knowledge-vault 直接迁移到 graphify。**

更合理的定位是：

> **Obsidian 继续做主知识库，graphify 作为结构化索引层 / 图谱导航层。**

它适合增强理解，不适合替代当前以人为中心、长期演化的笔记系统。

## graphify 的核心思想

graphify 的核心不是“写笔记”，而是：

- 从原始材料里抽节点与关系
- 对关系做聚类与分析
- 输出结构地图，让 agent 先看结构、后读原文

它的典型流水线是：

- detect
- extract
- build_graph
- cluster
- analyze
- report / export

所以它更像：

- 代码库地图
- 研究资料地图
- 混合语料的结构压缩层

而不是长期笔记系统本体。

## 为什么不适合替代当前 Obsidian

### 1. 它不是为长期写作工作流设计的

graphify 的主要输出是：

- `graph.json`
- `graph.html`
- `GRAPH_REPORT.md`
- 可选 `obsidian/` 派生输出

这套输出更像自动生成的派生物，不像主编辑面。

### 2. 它的 Obsidian 导出是“图谱视图”，不是现有 vault 模型

graphify 可以导出 Obsidian vault，但本质是：

- one note per node
- wikilinks 连接
- canvas 辅助查看
- graph view 配置

这适合浏览图谱，不适合直接承载当前 knowledge-vault 里的人类组织方式，例如：

- Projects / Research / Concepts 分层
- 日记、状态交接、总结稿
- 人工收口后的判断与边界

### 3. 它是解释层，不是真身层

graphify 会把材料重新解释成：

- EXTRACTED
- INFERRED
- AMBIGUOUS

这非常适合做结构化理解，但不适合直接取代原始笔记真身。

## graphify 更适合怎么用

### 适合作为增强层的目录

- `03-Research` 下的专题目录
- 某个项目下的混合资料目录（md / pdf / png / jpg）
- 需要 agent 快速理解结构的材料集

### 不适合作为迁移目标的内容

- daily notes
- 任务管理
- 状态交接
- 人工总结稿
- 长期记忆类笔记

## 推荐方案

### 方案判断

- **不建议**：把 knowledge-vault 整体迁移到 graphify
- **建议**：保留 knowledge-vault 为真身，按专题用 graphify 做旁路增强

### 推荐工作方式

对单个研究专题或项目材料目录运行 graphify，生成：

- `GRAPH_REPORT.md`
- `graph.html`
- `graph.json`
- 可选 `obsidian/`

然后在主 vault 中只保留一篇人工 note：

- 这个专题是什么
- graphify 看到了什么结构
- 我的结论是什么
- 接下来该继续查什么

## 关于“突然的小调研”放哪里

当前更建议采用一个简单规则：

### 默认放置原则

- **工具/产品/项目的快速调研**：放 `03-Research/Tools/`
- **明确属于某个进行中项目**：放对应 `02-Projects/<项目名>/`
- **只是临时收集、还没判断归属**：先放 `01-Inbox/`

### 对这次 graphify 调研的判断

这次更像：

- 工具研究
- 知识管理/agent 能力评估
- 尚未进入明确实施阶段

所以当前放在：

- `03-Research/Tools/`

是合适的。

## 一个后续可沉淀的规则

这类“突发小调研”不要一开始就追求完美归档，先按下面规则收：

1. **先收在最接近的问题域目录**
2. 文件名带日期，降低重名和重构成本
3. 如果后续进入持续推进，再从 `03-Research` 升格迁到 `02-Projects`

也就是说：

> 先让信息可找回，再决定它值不值得升级为项目资产。

这个规则比一开始纠结最终归属更稳。
