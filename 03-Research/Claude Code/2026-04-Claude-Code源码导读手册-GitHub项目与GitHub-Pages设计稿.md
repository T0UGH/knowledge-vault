# Claude Code 源码导读手册 GitHub 项目与 GitHub Pages 设计稿

## 这篇看什么

这不是新的源码共读正文，而是为下一阶段做准备：

> **把现有的 Claude Code 共读系列，整理成一个适合公开发布的 GitHub 项目，并挂一个 GitHub Pages 文档站。**

当前判断很明确：

- 重点不是先搭站
- 而是先把现有 `00-90` 篇的结构整理好
- 把它从“写作顺序累积出来的研究笔记”重组为“公开可读的源码导读手册”

所以这份文档的核心，是先把方向、边界、目录重组思路和实施顺序定下来。

## 先给主结论

如果这份设计稿只先记一句话，我会留这个版本：

> 当前最合理的方向，不是直接把 90 篇原样搬上 GitHub Pages，而是先把它们重组为一套 **文档手册型的 Claude Code 源码导读手册**：90 篇全文公开，但主导航不再按旧编号顺序展开，而是按新的卷/章结构组织；正文允许轻度修订；GitHub Pages 只是展示层，真正的第一优先是先完成内容结构重构。**

再压缩一点，就是：

- **先重组结构**
- **再公开全文**
- **最后再搭站**

一句最短版：

> **这不是“先做个站”，而是“先把 90 篇整理成一套手册，再让它上站”。**

---

## 一、项目目标

这次要做的，不是一个随手搭的归档仓库，而是一个能长期公开展示、可持续维护的项目。

核心目标有三个：

1. **把 90 篇从写作顺序重组为阅读顺序**
2. **把项目形态从研究日志提升为公开可读的源码导读手册**
3. **为后续 GitHub 仓库 + GitHub Pages 部署准备稳定内容骨架**

这轮不追求的事情也要明确写清：

- 先不做品牌化首页
- 先不做复杂交互
- 先不做课程产品
- 先不重写全部正文
- 先不讨论视觉包装

这轮就专注一件事：

> **先把现有的 Claude Code 共读内容整理成一套能公开发布的手册结构。**

---

## 二、目标读者

首要目标读者不是已经把 Claude Code 源码翻熟的人，而是：

> **对 Claude Code 感兴趣，但还没深入看过源码的人。**

这会直接决定手册的结构和写法：

- 目录必须更像导读，而不是资料堆
- 不能假设读者已经知道 `QueryEngine`、`compact`、`MCP`、`teammate runtime`
- 每一卷都要说明“这部分在讲什么”“建议从哪开始”“读完应得到什么”
- 正文可以保持技术密度，但入口必须更友好

次级读者还包括：

- 已在使用 Claude Code / coding agent 的工程师
- 想快速查某个内部机制的读者
- 未来想拿这套内容转文章、做分享、做索引的人

所以这套手册最终应同时满足两点：

1. **新读者能进入**
2. **老读者能检索**

---

## 三、为什么不能直接把 90 篇原样挂上去

现有 `00-90` 系列已经足够厚，也足够有价值，但如果直接原样上 GitHub Pages，会有几个明显问题。

### 1. 写作顺序不等于阅读顺序

这 90 篇不是一开始就按公开手册目录设计出来的，而是按当时的研究重点、理解收益和专题推进节奏一步步长出来的。

所以它天然带有两种阶段混合的痕迹：

- 前半段更像探索期
- 后半段更像专题期

这对内部持续写作没问题，但对新读者不够友好。

### 2. 原编号体系对公开阅读不友好

旧编号很适合内部连续积累，但它会把“历史写作顺序”误导成“最佳学习顺序”。

这会带来几个问题：

- 新读者容易迷路
- 早期探索期痕迹会被放大
- 整体更像研究日志，而不是成型项目

### 3. 缺卷级导读和阅读路径

目前单篇之间已经有不少局部衔接，但仍然缺：

- 每一大块为什么值得读
- 哪些是前置、哪些是进阶
- 新人最快从哪几篇进入
- 整套内容的最短路径和完整路径

所以如果直接原样挂站，读者大概率会感受到：

> 内容很强，但不够“可进入”。

---

## 四、推荐方案

当前最合理的方案是：

> **文档手册型结构 + 全量公开全文 + 正文允许轻度修订 + 旧编号退居映射层。**

这套方案的核心原则是：

1. **重做结构，不推倒重写**
2. **新站主导航不按旧编号展开**
3. **旧编号保留，但不再充当站点主结构**
4. **先稳定内容骨架，再谈技术实现**

为什么不选“只归档不整理”？

- 因为对新读者不够友好
- 站点会更像资料堆，不像项目

为什么不选“全面重写成书”？

- 工作量太大
- 会严重拖慢上线
- 很容易掉进无止境重写

所以最稳的就是中间路线：

> **结构上重组，正文上轻修。**

## 五、新目录结构（第一版）

当前建议把现有内容重组为 6 卷。

### 卷一：先建立 Claude Code 的基本地图

目标：先让读者知道 Claude Code 是什么系统，它的核心抽象有哪些。

建议纳入：

- 00 学习路线
- 01-04 Skill 起步
- 05-17 Tool / Agent / Skill 基础层
- 18-23 Agent 主链与定义层

这一卷读完，读者应该至少知道：

- Claude Code 不是普通 CLI，而是 agent runtime
- skill、tool、agent、fork、prompt 各处在哪一层

### 卷二：一条用户请求是怎么跑完整个系统的

目标：把 Query 主链真正串起来。

建议纳入：

- 38-48 Query / 主循环主链

这一卷读完，读者应知道：

- 用户输入怎么进入系统
- message 怎么组装成请求
- system prompt / context 怎么并进模型调用
- tool call 怎么接回主循环

### 卷三：Claude Code 为什么能处理长上下文和长会话

目标：把上下文治理和恢复机制单独讲清楚。

建议纳入：

- 49-60 上下文治理 / 长会话管理
- 61-63 Session / Resume 恢复链

这一卷读完，读者应知道：

- compact / microCompact / snip 各在干嘛
- transcript 有没有被真改写
- session 怎么存下来、读回来、再继续跑

### 卷四：外部能力和扩展点是怎么接进来的

目标：解释 Claude Code 为什么能长出外部能力，以及这些扩展点分别处在哪一层。

建议纳入：

- 64-68 MCP
- 69-72 Hooks
- 73-78 Plugin

这一卷读完，读者应知道：

- MCP、hooks、plugin 不是同一类东西
- 它们各自在 runtime 哪一层
- Claude Code 的扩展能力为什么不是拼凑的

### 卷五：执行边界、安全和权限控制

目标：解释 Claude Code 为什么不是“想跑什么就跑什么”。

建议纳入：

- 79-84 Permissions

这一卷读完，读者应知道：

- tool execution 和 permission decision 怎么接
- BashTool 为什么最复杂
- path rules / settings / policy limits 各自负责什么

### 卷六：多 agent 协作运行时

目标：把 team / swarm 这一层讲成一个独立系统。

建议纳入：

- 85-90 Team / Swarm

这一卷读完，读者应知道：

- team 为什么不是简单多开 agent
- leader / teammate / mailbox / shutdown 为什么构成 swarm runtime

## 六、旧编号怎么处理

旧编号不能丢，因为它已经进入：

- 现有 Obsidian
- 飞书文档
- 历史聊天语境
- 后续检索和回溯

但旧编号也不能继续当主导航，不然公开阅读还是会乱。

所以更合理的做法是：

### 1. 站点主结构用“卷 / 章 / 节”

公开站和项目主页优先按新目录组织。

### 2. 旧编号保留为 secondary id

例如：

- 页面正文里保留“原编号：CC-42”
- 索引页保留“旧编号 → 新目录”的映射表

### 3. 第一阶段不强制改文件名

为了降低整理成本，第一阶段可以保留现有文件名，只在：

- frontmatter
- 索引页
- 站点导航

里引入新结构。

等结构稳定后，再决定是否做第二轮重命名。

## 七、正文编辑策略

这轮采用：

> **轻度修订正文，不大改写。**

### 允许做的事

- 改标题，使其更适合公开阅读
- 补导语，说明这篇在整卷里的位置
- 补过渡段，说明和前后篇的关系
- 统一术语表达
- 删除少量明显重复段落
- 补必要的术语解释或前置提醒

### 这轮不做的事

- 不重写整篇结构
- 不大规模合并拆分正文
- 不改变核心判断和主要论证逻辑
- 不为了“像出版物”而抹掉源码细节

目标不是把研究味全去掉，而是：

> **让它更像公开导读手册，但仍然保留源码共读的硬核密度。**

## 八、站点至少要有什么页面

在技术选型之前，信息架构要先定清楚。

至少建议有这些页面：

### 1. 项目首页
回答：

- 这是什么项目
- 适合谁读
- 建议从哪里开始
- 现在覆盖了什么

### 2. 总目录页
展示：

- 6 卷结构
- 每卷简介
- 建议入口

### 3. 旧编号映射页
服务于：

- 老读者
- 历史检索
- 飞书/Obsidian 追踪

### 4. 最短阅读路径页
至少给：

- 最短路径
- 标准路径
- 完整路径

### 5. 分卷导读页
每卷至少包含：

- 卷目标
- 前置知识
- 本卷目录
- 推荐阅读顺序

### 6. 正文章节页
每篇至少建议补：

- 所属卷/章
- 原编号
- 上一篇 / 下一篇
- 推荐前置篇目

## 九、实施顺序

如果真做这个项目，最稳的顺序不是先搭站，而是：

### 第一步：整理新目录
产出：

- 6 卷结构确认
- 各卷收录范围确认

### 第二步：建立映射表
产出：

- 旧编号 → 新结构映射
- 需轻修文章清单
- 明显断裂/重复位置清单

### 第三步：补索引层内容
产出：

- 首页草稿
- 总目录页
- 分卷导读页
- 阅读路径页

### 第四步：轻修正文
产出：

- 标题统一
- 导语补齐
- 术语和过渡修订

### 第五步：最后再选站点技术栈
等内容骨架稳定之后，再决定：

- Docusaurus
- MkDocs Material
- Astro Starlight

这一阶段不先锁技术。

## 十、当前这版设计稿还缺什么

这版已经够定方向，但如果要直接进入执行，还缺几项必须补实的东西：

1. **正式目录表**
   - 现在只有 6 卷结构
   - 还缺到“每篇文章 → 新卷 / 卷内顺序 / 是否需桥接页”的粒度

2. **轻修规则要更具体**
   - 哪些篇必须补导语
   - 标题统一到什么格式
   - 术语首次解释怎么做
   - 什么算“明显重复段落”

3. **要明确允不允许新增桥接内容**
   - 比如卷导读、术语页、最短路径页、必要过渡页

4. **需要阶段 gate 和验收标准**
   - 什么叫“目录结构确认”
   - 什么叫“可以进入建站”

5. **要定义 91 篇以后怎么并入**
   - 新文章继续走旧编号
   - 还是直接进入新手册结构

也就是说，这篇更准确地说是：

> **方向已经对了，但还需要补成可执行版 spec。**

## 术语补充 / 名词解释

### 文档手册型

指项目主要形态更像源码导读手册，而不是博客流或研究日志流。

### 旧编号映射

指保留 `00-90` 这套历史编号，但不再把它作为站点主导航。

### 轻度修订

指允许调整标题、导语、过渡、术语解释和少量重复内容，但不重写主内容。

### 结构重组

指把现有写作顺序重新排列为新的公开阅读顺序。
