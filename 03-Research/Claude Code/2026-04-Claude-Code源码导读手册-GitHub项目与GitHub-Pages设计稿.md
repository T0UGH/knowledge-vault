# Claude Code 源码导读手册 GitHub 项目与 GitHub Pages 可执行设计稿

## 这篇看什么

这不是新的源码共读正文，而是把下一阶段要做的项目先定成一份**可执行 spec**。

目标很明确：

> **把现有的 Claude Code 共读系列，从按写作顺序积累出来的 `00-90` 篇，重组为一套适合公开发布的源码导读手册，并为后续 GitHub 仓库 + GitHub Pages 文档站提供稳定骨架。**

这份文档不是只讲方向，而是要回答下面这些真正会卡住执行的问题：

- 新目录到底怎么拆
- 90 篇怎么映射进新结构
- 哪些文章要轻修，修到什么程度
- 允不允许新增桥接页
- 什么情况下才算“可以进入建站”
- 91 篇以后怎么并进，不让新旧体系再裂一次

## 先给主结论

如果这份 spec 只先记一句话，我会留这个版本：

> 当前最合理的方案，不是直接把 90 篇原样搬上 GitHub Pages，也不是推倒重写成一本新书，而是先把它们重组为一套 **文档手册型的 Claude Code 源码导读手册**：站点主结构按新卷/章组织，旧编号退居映射层；90 篇全文公开；正文只做轻度修订；允许新增少量桥接页；等目录、映射表、导读层和编辑准则稳定之后，再进入具体站点技术实现。**

再压缩一点，就是：

- **先重组结构**
- **再补导读和桥接层**
- **再轻修正文**
- **最后再搭站**

一句最短版：

> **先把 90 篇整理成一套可读手册，再让它上站。**

---

# 一、项目目标

这次要做的，不是一个资料仓库，也不是一个品牌首页优先的展示站，而是一个：

> **面向公开阅读的 Claude Code 源码导读手册项目。**

核心目标有三个：

1. **把 90 篇从写作顺序重组为阅读顺序**
2. **把项目形态从研究日志提升为公开可读的手册**
3. **为 GitHub 仓库 + GitHub Pages 建立稳定内容骨架**

本阶段明确不追求：

- 不先做品牌视觉
- 不先做复杂交互
- 不先做课程产品
- 不重写全部正文
- 不在目录未定前先选技术栈

这轮只专注一件事：

> **先把内容结构整理对。**

---

# 二、目标读者

首要目标读者是：

> **对 Claude Code 感兴趣，但还没深入看过源码的人。**

因此这套手册必须满足：

- 目录像导读，不像资料堆
- 不假设读者已经理解 `QueryEngine`、`compact`、`MCP`、`teammate runtime`
- 每一卷都要明确“这部分讲什么、建议从哪开始、读完得到什么”
- 正文可以保留技术密度，但入口必须更友好

次级读者包括：

- 已经在用 Claude Code / coding agent 的工程师
- 想快速查某个内部机制的读者
- 后续想转文章、做分享、做二次研究的人

所以最终手册需要同时满足：

1. **新读者可进入**
2. **老读者可检索**

# 三、为什么不能直接把 90 篇原样挂上去

现有 `00-90` 已经足够厚，也足够有价值，但如果直接原样公开，会有 4 个明显问题。

## 1. 写作顺序不等于阅读顺序

这 90 篇不是一开始按手册目录规划出来的，而是按阶段性研究重点一路长出来的。

所以它天然混着两种气质：

- 前半段更像探索期
- 后半段更像专题期

对内部积累没问题，对新读者不够友好。

## 2. 原编号体系对公开阅读不友好

旧编号适合内部连续积累，但容易把“历史写作顺序”误读成“最佳学习顺序”。

结果就是：

- 新读者容易迷路
- 早期探索期痕迹被放大
- 站点看起来更像研究日志而不是成型项目

## 3. 缺卷级导读和阅读路径

当前局部衔接已经不少，但仍缺：

- 每一大块为什么值得读
- 哪些是前置，哪些是进阶
- 新人最快从哪几篇进入
- 整套内容的最短路径与完整路径

## 4. 术语暴露方式还不够手册化

最近的共读流程已经开始补“名词解释”，但要做公开手册，还需要更系统地处理：

- 每卷的前置假设
- 高密度术语第一次出现时怎么解释
- 哪些解释放页内，哪些放术语页

---

# 四、推荐方案

推荐采用：

> **文档手册型结构 + 全量公开全文 + 正文轻度修订 + 旧编号退居映射层 + 允许新增少量桥接页。**

核心原则：

1. **重做结构，不推倒重写**
2. **新站主导航不按旧编号展开**
3. **旧编号保留，但只作为历史索引层**
4. **允许少量新增桥接页，不等于重写正文**
5. **先稳定内容骨架，再谈技术实现**

为什么不选“只归档不整理”？

- 对新读者不够友好
- 站点更像资料堆，不像项目

为什么不选“全面重写读本”？

- 工作量过大
- 会显著拖慢上线
- 很容易从整理滑向无止境重写

所以最稳的是：

> **结构上重组，导读层补齐，正文上轻修。**

- compact / microCompact / snip 在干什么

## 卷四：外部能力和扩展点是怎么接进来的

- 64-68 MCP
- 69-72 Hooks

读完应得到：
- MCP、hooks、plugin 不是一类东西
- 它们各自在 runtime 哪一层

## 卷五：执行边界、安全和权限控制
目标：解释 Claude Code 为什么不是“想跑什么就跑什么”。

收录范围：
- 79-84 Permissions

- tool execution 和 permission decision 怎么接

目标：把 team / swarm 这一层讲成一个独立系统。

收录范围：
- 85-90 Team / Swarm

读完应得到：
- team 为什么不是简单多开 agent
- leader / teammate / mailbox / shutdown 为什么共同构成 swarm runtime

# 七、桥接内容策略

这次重组**明确允许新增少量非原编号页面**。

这些页面不算“重写正文”，而是为公开阅读补必要骨架。

建议允许新增的页面类型：

## 1. 卷导读页
每卷必须有。

至少包含：
- 这一卷讲什么
- 适合谁读
- 前置知识
- 推荐阅读顺序
- 读完会得到什么

## 2. 总目录页
全站必须有。

至少包含：
- 6 卷结构
- 每卷简介
- 推荐入口

## 3. 最短阅读路径页
建议有。

至少给：
- 最短路径
- 标准路径
- 完整路径

## 4. 术语页
建议有一页总术语表，主要承接高频内部词：
- QueryEngine
- compact / microCompact / snip
- MCP
- hooks
- plugin
- teammate runtime

## 5. 必要过渡页
仅在卷内明显跳跃时新增，数量应严格控制。

### 新增桥接页数量约束

第一阶段建议控制在：
- 卷导读页：6 页
- 总目录页：1 页
- 阅读路径页：1 页
- 术语页：1 页
- 其他过渡页：最多 2 页

总新增页建议不超过 **11 页**。

这样可以补骨架，但不至于把项目变成另一轮大写作。

# 八、正文轻修编辑准则

这部分必须明确，不然不同文章会修得尺度不一致。

## 1. 标题统一规则

新站展示标题优先采用：

- 面向公开阅读的完整标题
- 避免过于聊天化、口语化、探索式标题
- 保留技术判断，但减少“我在半路发现了什么”的痕迹

允许保留原标题作为：
- frontmatter 里的 `original_title`
- 或旧编号映射页中的历史标题

## 2. 导语补充规则

以下文章**必须补篇前导语**：
- 每卷第一篇
- 承担桥接作用的文章
- 存在明显上下文依赖的文章
- 当前标题不足以自解释的文章

导语长度建议：
- 80-180 字

导语至少回答：
- 这篇在全局里解决什么问题
- 建议在什么位置读

## 3. 术语解释规则

- 高频核心术语优先在**第一次出现处**做简短页内解释
- 若该术语横跨多卷，统一再放进总术语页
- 不要求每篇都做 glossary dump
- 原则是**降低理解阻力，而不是把每篇写成词典**

## 4. 重复内容处理规则

“明显重复段落”判定标准：
- 与紧邻文章重复解释同一概念，且没有新增判断
- 与卷导读页的职责重复
- 仅仅因为历史写作顺序不同而导致的重复开场白

处理方式：
- 优先删重复开场
- 其次改成一句链接式指向
- 不轻易删核心论证段落

## 5. 上一篇 / 下一篇 规则

进入新手册主链的文章，正文页必须补：
- 所属卷
- 原编号
- 上一篇
- 下一篇
- 推荐前置篇目（如有）

## 6. 哪些文章可以只做最轻修

以下类型可只做最轻修：
- 已经结构完整的专题收口篇
- 标题已经清楚、上下文依赖低的单篇
- 主要用于检索而非主路径阅读的补充篇

# 九、验收标准

这部分必须可检查，不能只写方向口号。

## 阶段一：目录结构确认
通过条件：
- 6 卷结构固定
- 90 篇全部落入“主链 / 补充 / 桥接参考”三类之一
- 没有未归属文章
- 每卷有明确目标陈述

## 阶段二：映射表完成
通过条件：
- 90 篇都有旧编号 → 新结构映射
- 标注出每篇是否需轻修
- 标注出需弱去重、需桥接说明的文章
- 新增桥接页清单已确定

## 阶段三：导读层完成
通过条件：
- 首页草稿完成
- 总目录页完成
- 6 个卷导读页完成
- 阅读路径页完成
- 术语页草稿完成

## 阶段四：正文轻修完成
通过条件：
- 所有主链文章完成标题/导语/导航补齐
- 重复内容处理完成
- 术语解释规则已落实
- 至少抽查 10 篇，风格一致性可接受

## 阶段五：可进入建站
通过条件：
- 新读者第一次打开首页，能明确知道从哪开始
- 任一主链文章页面都能知道自己处于哪一卷
- 老读者能通过旧编号定位到对应正文
- 内容骨架不再需要大调整

只有阶段五通过，才进入技术选型与建站。

# 十、阶段 gate

为了防止过早跳去建站，这里明确 gate：

## Gate A：不完成映射表，不讨论站点侧边栏

## Gate B：不完成卷导读与阅读路径，不进入视觉/首页包装

## Gate C：不完成正文轻修抽查，不进入 GitHub Pages 技术实现

## Gate D：不定义 91+ 并入规则，不宣布结构稳定

这些 gate 的作用只有一个：

> **先把内容结构做对，再谈展示层。**

# 十一、91 篇以后怎么并入

这件事必须现在定，不然后面又会裂。

建议规则如下：

## 1. 新文章继续保留连续编号
例如继续写 91、92、93……

因为旧编号体系已经进入历史语境，不值得硬砍。

## 2. 但新文章写完后，不直接按编号挂到站点主导航
而是必须先回答：
- 它属于哪一卷？
- 是主链新增、卷内补充，还是桥接文？
- 是否改变原卷结构？

## 3. 新文章并入流程
每新增一篇，至少更新：
- 旧编号映射页
- 所属卷目录页
- 如有必要，更新阅读路径页

## 4. 如果新主题明显超出 6 卷
例如未来真的写出一整条新大线，才允许新增“卷七”。

新增卷的门槛建议是：
- 至少已有 4 篇以上稳定相关内容
- 能形成独立阅读目标
- 不只是一个补充专题

# 十四、风险与约束

## 风险 1：整理滑向重写
控制方式：

- 桥接页数量受限
- 不改核心论证结构

控制方式：

- 必须标注是否主链 / 补充 / 参考

控制方式：
- 允许新增少量桥接页
- 卷导读必须承担阅读引导责任

## 风险 4：旧编号彻底失效
控制方式：
- 保留旧编号映射页
- 每篇保留原编号字段

## 风险 5：91+ 再次把体系拉裂
控制方式：
- 明确并入规则
- 任何新文必须先归卷再上站
