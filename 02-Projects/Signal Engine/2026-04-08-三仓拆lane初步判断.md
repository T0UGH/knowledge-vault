# 2026-04-08｜三仓拆 lane 的初步判断

## 这次要记的结论

今晚继续围绕 `github-watch`、`signal-engine` 和三条重点仓（`anthropics/claude-code`、`openclaw/openclaw`、`openai/codex`）怎么改做了进一步判断。当前最值得记住的不是实现细节，而是下面三点：

1. **把三条仓库继续塞在一条 `github-watch` lane 里，长期看不对，应该拆成三条 repo-specific lane。**
2. **OpenClaw 这条线现在日报效果还不错，不是因为 collector 很先进，而是因为它自己的 CHANGELOG 足够强。**
3. **如果要改，第一步最好先拆采集层/collector，让三条线先各自产出更对路的 signals，再决定 signal schema 和 editor 怎么跟进。**

---

## 一、为什么现在应该考虑拆成三条 lane

前面已经判断过：Claude Code、OpenClaw、Codex 不是同一种仓库类型。

今晚进一步确认之后，这个结论更稳了：

### Claude Code
更像：
- 产品发布线
- 主要有效信息在 release / changelog 正文里

### OpenClaw
更像：
- 平台集成 / 多运行时平台线
- 关键变化集中在 changelog 与 PR 主题面

### Codex
更像：
- 底层 runtime / execution substrate 线
- 关键变化主要在 merged PR / commit 流，不在 README/CHANGELOG

这意味着：

> **三条线的“有效观察面”本身就不一样。**

如果继续让它们共享一条 `github-watch` lane，只会出现越来越多的 repo-specific if/else：

- Claude Code 要求强制展开 release body
- OpenClaw 要求 changelog-first，再补 PR
- Codex 要求 PR/commit-first

久而久之，`github-watch` 会从“通用 lane”变成一坨越来越难维护的 repo 特判逻辑。

所以从 signal engine 视角看，更合理的方向是：

> **把这三条重点仓直接拆成三条独立 lane。**

仍然共用一套 runtime / artifact / editor 流水线，但 collector 逻辑分开。

---

## 二、OpenClaw 这条线现在到底是怎么抓的

今晚又专门回看了当前 daily 报里 OpenClaw 的来源，避免误判。

### 结论很明确

当前 OpenClaw 不是单独一条 lane，它还挂在：

- `github-watch`

里，和：

- `anthropics/claude-code`
- `openai/codex`

一起被抓。

### 当前 `github-watch` 的真实抓法

配置在：
- `~/workspace/daily-lane/config/lanes.yaml`

脚本在：
- `~/workspace/daily-lane/lanes/github-watch.sh`

它当前只抓三类信号：

1. `release`
2. `CHANGELOG`
3. `README`

具体说：

- release：`gh api repos/<repo>/releases`
- changelog：直接抓 `CHANGELOG.md / CHANGES.md / HISTORY.md`
- readme：`gh api repos/<repo>/readme`

它**没有**抓：

- merged PR
- commits
- issues
- discussions

### 2026-04-08 当天 OpenClaw 实际抓到了什么

当天 OpenClaw 只产出了两条 signal：

- `openclaw__openclaw__changelog__2026-04-08.md`
- `openclaw__openclaw__readme__2026-04-08.md`

没有 release signal。

也就是说，**当天日报里 OpenClaw 能写得不错，主要靠 changelog 那条。**

## 三、为什么 OpenClaw 现在 daily 效果看起来还不错

这是今晚最关键的一个校正。

之前可能会直觉觉得：

- `github-watch` 这条 lane 对 OpenClaw 抓得还行

但细看之后，更准确的判断应该是：

> **不是 collector 很强，而是 OpenClaw 自己的 CHANGELOG 很强。**

### 直接证据

2026-04-08 那条 `openclaw CHANGELOG updated` signal 本身信息密度非常高，里面直接就有大量日报级素材：

- `openclaw infer ...`
- Gemma 4 / Google compatibility 细化
- memory / dreaming / memory-wiki
- ClawHub 安装 SHA-256 校验
- allowlist owner gate
- Feishu docx workspaceOnly
- Browser SSRF / Gateway auth rotation / node approvals
- sessions compaction checkpoint / branch restore
- subagent `lightContext: true`

这些内容本身已经接近“经过第一轮整理的更新摘要”。

### 所以为什么 editor 最终能写得像样

不是因为 collector 深入抓到了 OpenClaw 的真实底层变化，
而是因为：

1. OpenClaw changelog 自己写得足够好
2. editor prompt 又强制要求重点 repo（包括 OpenClaw）必须单独成节
3. editor 再把 changelog 里的高密度信息二次整理成日报 prose

因此，OpenClaw 当前看起来“质量还行”，更像是：

> **高质量 changelog 喂给了一个还算能整理的 editor。**

不是因为这条 lane 的 repo 观察策略已经设计好了。

## 四、这对后续拆 lane 的启示

### OpenClaw 不一定需要像 Codex 那样重改

Codex 现在的问题非常明确：
- 真正该看的内容在 PR / commit
- 而现有 `github-watch` 基本没抓到

所以 Codex 必须大改。

但 OpenClaw 不完全一样。

因为 OpenClaw 至少有一个天然优势：
- changelog 本身质量高

所以它的未来 lane，不一定需要一步到位做成纯 PR-first，
更像应该是：

> **`changelog-first + PR-assisted`**

也就是：

1. 优先抓 changelog 正文
2. 再抓 PR 流补强：
   - 安全 / 权限
   - gateway / browser
   - agents / sessions / runtime
   - integrations
3. README 继续只做低优先级补边

这样改造成本会比 Codex 低，而且更符合 OpenClaw 自身的 repo 结构。

### Claude Code / OpenClaw / Codex 未来 lane 的直观定位

目前比较稳的判断可以先记成：

- **Claude Code**：`release-first`
- **OpenClaw**：`changelog-first + PR-assisted`
- **Codex**：`merged-PR-first + commit-second + release-third`

---

## 五、今晚对改造范围的一个重要收口

用户提出拆成三条 lane 后，我给出的收口建议是：

> **先把目标缩成最小可落地版本：先拆 collector/lane，先让三条线各自产出更对路的 signals；暂时不急着同时改 editor/report 流水线。**

这个收口很重要，因为否则很容易一次性把问题做太大：

- collector 变
- signal schema 变
- editor prompt 变
- orchestrator lane 发现逻辑也变

如果一步全动，排查会变得很乱。

### 更稳的推进顺序应该是

#### Phase 1
- 拆 collector / lane
- 三条 repo-specific lane 独立产 signal
- 继续共用 runtime / signal artifact / editor 流水线

#### Phase 2
- 根据新 signal 的信息密度，再决定 signal schema 要不要扩 richer fields
- 例如 PR body summary、validation block、theme tag 等

#### Phase 3
- 最后再决定 orchestrator / editor 是否要做 repo-aware 特化

也就是说，今天的设计直觉已经比较明确：

> **先拆采集，不先拆写作。**

---

## 六、一句话收束

今晚这波讨论后，最值得保留的判断是：

> **三仓拆成三条 lane 是对的，但不是因为“想要更细”，而是因为三条仓本来就不是同一种观察对象；其中 OpenClaw 当前看起来效果不错，主要靠 changelog 质量高，所以它未来更适合走 `changelog-first + PR-assisted`，而不是直接照搬 Codex 的 PR-first 方案。**
