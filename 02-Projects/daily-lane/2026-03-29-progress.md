# Daily Lane Progress

这份笔记只回答三件事：

1. `daily-lane` 现在已经干到哪里了
2. 当前有哪些关键结论已经拍定
3. 下一步最值得继续推进什么

---

## 一、当前项目状态（一句话）

`daily-lane` 已经从“概念讨论”进入到：

> **底层 lane / signal 召回层已经基本成形，接下来应从“继续扩 lane”切换到“开始长日报内容层”。**

也就是说，项目现在不再只是一个想法，也不只是几份 spec；已经有一批 lane 代码、真实 try-run、真实 signal 产物，以及对上层日报机制的第一轮收敛。

---

## 二、目前已经完成了什么

## 1. 底层系统方向已经定了

已经明确：这不是一个“日报生成器”，而是一个：

- **lane-first**
- **signal-first**
- **文件优先**

的情报系统。

当前 Phase 1 的最小闭环是：

```text
lane -> signal -> signal pool
```

这层的职责是：

- 召回原始信号
- 尽量无损保留
- 不在 signal 层做 AI 筛选、候选裁剪、推荐排序

这个方向已经比较稳，不是还在摇摆的状态。

---

## 2. lane 层原则已经拍定

lane 层当前已经明确几个硬原则：

- lane 是第一公民
- lane 层尽量是**稳定脚本**
- signal 层只负责收集，不负责编辑判断
- 不在召回层过早引入 AI 决策
- 同一对象同一天出现多个异质 signal，也应分别保留，不在 signal 层合并

这意味着后面继续扩 lane 时，不需要再反复讨论“要不要在 lane 层做智能筛选”这类问题了。

---

## 3. 五条 lane 已经基本落地

截至当前，已经明确进入系统的五条 lane 是：

- `github-watch`
- `product-hunt-watch`
- `github-trending-weekly`
- `x-feed`
- `x-following`

其中最关键的一点是：

- 原先的 `x-watchlist`
- 已根据贵平的产品决策
- 收口成了 `x-following`

这个变化不是技术人员自作主张，而是产品判断已经落到实现里。

---

## 4. 五条 lane 的实现状态

### `github-watch`
状态：**已跑通 Phase 1 样板 lane**

当前承担：

- 长期跟踪型 GitHub signal 收集
- `release / changelog / README` 变化记录

它已经证明：

- lane-first 的目录结构成立
- signal pool 结构成立
- `state/` 快照机制成立

### `product-hunt-watch`
状态：**已有实现，也已有真实历史产物**

当前作用：

- 证明非 GitHub 来源也可以接进同一套 lane / signal 骨架

注意：这条 lane 在最近一轮独立 fresh 验证时，受 `PH_API_TOKEN` 环境限制，没有完成新的 try-run 闭环；但历史产物是存在的。

### `github-trending-weekly`
状态：**实现已完成，独立 try-run 通过**

当前作用：

- 承担 GitHub 新项目 / 趋势发现入口的一部分
- 用 GitHub Trending weekly 总榜 + README 形成独立 signal

这条 lane 已证明 discovery 型 GitHub 来源也能纳入当前系统。

### `x-feed`
状态：**实现已完成，真实产物存在，独立 try-run 通过**

当前语义：

> 算法推荐流里“我被看到了什么”

也就是推荐 tab 的被动曝光记录。

### `x-following`
状态：**实现已完成，真实产物存在，独立 try-run 通过**

当前语义：

> 我关注的人最近发了什么

它已经替代旧的 `x-watchlist` 方案，且这个替换有真实产品理由：

- 逐账号 search 太慢
- following timeline 更自然
- 全量收录更符合 signal 层“尽量不裁剪”的原则

---

## 5. 代码与验收层面已经不是纸面状态

目前这套系统已经不只是“有设计文档”，还包括：

- lane 脚本
- 真实 try-run
- data repo 中的真实 signal 产物
- 一轮独立 review / 验收

这意味着项目已经从：

- idea
- spec
- impl design

进入到：

> **可以开始讨论“上层怎么消费这些 signal”**

的阶段。

---

## 6. 日报上层机制也已经有第一轮收敛

现在已经不只是底层 lane 在长，日报 pipeline 的第一版机制也已经形成共识：

- 第一版**不要先做规则系统**
- 不要先把筛选权固化成显式规则
- 先采用**双 agent 机制**

即：

- **Agent 1**：筛选 / 归纳 / 成稿
- **Agent 2**：审核 / 打分 / 打回
- **最多只回环 1 次**

这个决定很关键，因为它说明上层日报不是先做 DSL，而是先把“编辑闭环”跑顺。

---

## 三、当前已经拍定的关键结论

以下这些，我认为已经属于当前阶段的稳定结论：

### 1. `daily-lane` 的底层不是日报系统，而是 signal system

日报只是上层消费层之一，不应反向污染 lane / signal 设计。

### 2. lane 层尽量用稳定脚本，不引入 AI 决策

AI 更适合放在 signal 之上的内容处理层。

### 3. X 相关 lane 已经分成两条不同语义

- `x-feed` = 推荐流 / 被动曝光
- `x-following` = 关注流 / 主动关注对象输出

这两个不能混为一谈。

### 4. `x-watchlist -> x-following` 已经是正式产品决策

后续文档、代码、验收都应按 `x-following` 理解，不应再把旧方案当默认方向。

### 5. 日报第一版先用双 agent 编辑闭环

不是先做规则引擎，不是先做复杂工程化评分系统。

### 6. 当前阶段的主问题，已经不再是“还要不要继续扩 lane”

而是：

> **怎么把现在已有的 signal pool 长成真正可交付的日报内容层。**

---

## 四、当前还没完成、但已经看得见的问题

## 1. 底层 lane 有了，但上层“消费协议”还没落地

现在 signal 已经开始积累，但还缺一层稳定的：

- 日报输入组织方式
- 每条 lane 如何提供给日报层
- 哪些 signal 要进入当天候选池
- 怎么保留 lane 语义进入最终 briefing

换句话说：

- 召回层已经开始成形
- 内容生产层还没正式落代码

---

## 2. `product-hunt-watch` 还缺一次 fresh 验证闭环

这不是说它没实现，而是最近一轮独立验收里，受环境变量影响，没有拿到 fresh try-run evidence。

所以它当前更准确的状态是：

- 有代码
- 有历史成功产物
- 但还值得补一次新鲜验证

---

## 3. 当前 data repo 提交流程不适合并发 lane 同时写入

这不是当前 blocker，但已经暴露了工程边界：

- 如果多个 lane 并发往同一个 data repo 工作区写
- commit 边界会互相污染

这说明后面如果要并行跑 lane，需要明确：

- 要么串行跑
- 要么独立工作区

但这属于运维/执行层问题，不是当前主线问题。

如果把项目阶段压成一句更像项目经理口径的话，我会写成：

> `daily-lane` 的 Phase 1 召回层已经基本立住：五条 lane 已经成形，至少四条拿到了独立 fresh try-run 证据，X 相关 lane 也完成了从 watchlist 搜索模式到 following timeline 模式的产品收口。项目现在应该从“继续铺底层 lane”转向“让现有 signal pool 产出真正可用的日报内容”。

我建议下一步不要再分散开做很多事，而是聚焦一件最关键的：

## 下一步主目标：把“日报内容层”长出来

也就是正式进入：

> **signal pool -> briefing / daily report**

但这里第一步仍然不应太重。

## 七、建议的下一步顺序

### Step 1：先采用“两个 subagent + 一个 cron”的最小形态

当前对日报内容层的最新收敛是：

- **Subagent A / Editor**：负责从当天各 lane signal 中做筛选、归纳、成稿
- **Subagent B / Reviewer**：负责审核、打分、决定是否打回
- **一个 cron**：负责每天定时触发、调度两个 subagent、控制最多一次回环，并完成最终交付

也就是说，cron 不是写日报的人，而是：

> **日报流程的 orchestrator**

这个结构已经足够支撑第一版，不需要再额外引入更复杂的 agent 网络、规则引擎或状态机系统。

### Step 2：cron 的第一版输入已经拍定

日报内容层第一版里，cron **只喂各 lane 的 `index.md`**。

不先做：

- 全量 signal 打包输入
- 复杂 lane bundle
- 额外中间层整理

具体展开方式是：

- **Editor** 看 `index.md` 后，按需自己回查原始 signal 文件
- **Reviewer** 也不是只看草稿，而是按需自己回查原始 signal 文件

也就是说，cron 第一版只负责把“入口目录”交给两个 agent，而不是替它们预先整理完整材料。

这版成立的前提是：

> **各 lane 的 `index.md` 必须足够像导航入口，而不是裸文件名列表。**

### Step 3：日报最终产物和结构也已经拍定

第一版先只做：

- **完整 markdown 日报**

暂时不急着先上：

- Feishu 摘要版
- 顶部重点层
- 额外压缩稿

而且结构已经明确：

- **按 lane 展开**
- **不要顶部重点**

这意味着第一版日报更像：

> **按 lane 组织的完整 briefing**

而不是跨 lane 压缩过的晨报摘要。

### Step 4：编辑原则也已经拍定

同一个 lane 下面的内容：

- 可以一条 signal 对应一条日报项
- 也可以由 Editor 把多个相关 signal 合成一条日报项

但有一个更高优先级约束：

> **不能为了变短而丢信息。**

当前默认策略应是：

- 宁可长一点
- 宁可多保留原始变化
- 不把关键细节洗掉
- 让贵平能看到更完整的信息，而不是过度抽象后的总结

### Step 5：在正式跑日报前，先重做各 lane `index.md`

因为现在 cron 第一版只喂 `index.md`，所以 index 不能再只是归档目录，而必须成为：

> **agent 可导航入口**

这一步的最新收敛是：

- lane 层仍然只是**稳定脚本**
- index redesign 不允许引入 AI
- index 采用：**公共底座 + lane 自定义附加字段**
- 导航提示必须从现有 signal 字段或正文里用脚本稳定摘取，不允许 AI 生成

目前的判断是：

- `x-feed` / `x-following` 勉强可用，但导航性仍偏弱
- `github-watch` / `product-hunt-watch` 明显不够用
- 所以在日报内容层真正跑起来前，index 补强应视为前置工作

### Step 6：再去验证 Editor / Reviewer 的最小工作流

也就是：

- Editor：基于各 lane `index.md` 自己找 signal、筛选、归纳、出完整 markdown 草稿
- Reviewer：基于 `index.md` + 草稿 + 原始 signal 做真实审核、打分、打回一次
- cron：硬控“最多一次回环”，第二版无论是否通过都结束

先验证这个闭环是否真的改善日报质量，再谈后续工程化。

### Step 4：等闭环跑顺后，再决定哪些判断值得固化

届时再看：

- 哪些 reviewer 判断值得规则化
- 哪些 lane 约束需要显式化
- 哪些输入结构需要程序化整理

而不是现在就先上规则系统。

---

## 八、下一步不建议优先做什么

当前阶段，我不建议优先做：

- 继续新增很多 lane
- 提前做规则 DSL
- 提前做复杂排序 / 评分系统
- 提前做大而全的 orchestrator 架构设计
- 把 lane 层又拉回 AI 筛选逻辑

这些都容易把注意力从真正的问题上带偏。

当前真正的主问题已经不是“能不能抓到更多东西”，而是：

> **抓到的这些东西，怎么形成一份真正有用的日报。**

---

如果只保留一句最短建议：
