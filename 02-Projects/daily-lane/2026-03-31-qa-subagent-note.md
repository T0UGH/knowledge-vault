# Daily Lane QA Subagent Note

记录时间：2026-03-31
状态：**仅记录，暂不实施**

## 背景

今天讨论了是否要在现有分工之外，引入一个新的长期角色，专门负责质量把关、测试核对和代码 review。

当前已有的稳定分工大致是：

- RK：需求收敛、产品判断、流程设计、验收口径定义
- Arc：工程实现、技术判断、代码修改、链路跑通

新增角色的候选定位不是“第三位什么都参与的同事”，而是一个**看证据的 QA**。

---

## 当前结论

倾向于：

> **可以引入一个常驻身份的 QA subagent，但它应当是“固定角色、按需调用”，而不是全天在线、随时插话的第三同事。**

这里“常驻”的含义应该是：

- 有固定身份
- 有固定 prompt
- 有清晰边界
- 有稳定调用时机
- 有固定输出格式

而不是：

- 持续围观所有对话
- 在设计、实现、验收三个阶段都频繁发表意见
- 变成流程噪音源

---

## 建议角色定义

### 角色定位

这个 QA 的本质应该是：

- 不负责实现
- 不负责拍方案
- 不负责润色表达
- 只负责看证据、做验收、指出未满足项

### 核心价值

它的价值不在“再提一些想法”，而在：

- 防止“嘴上说做完了，但没有验证”
- 防止“局部改了，但最终产物没变”
- 防止“看起来差不多，其实没满足 spec / plan / 用户原话”

---

## 建议职责范围

### 1. 验收 / Verification

对照以下材料做核对：

- 用户原始要求
- spec
- implementation plan
- 当前真实产物

目标是回答：

> **这件事是不是真的做到了？**

### 2. 测试与结果核对

重点看：

- 命令是否真的运行过
- rerun 是否真的产生了目标输出
- 标题、链接、sources、输出文件是否真的符合要求
- 最终可交付物是否和“实现者的口头描述”一致

### 3. 代码 Review

这里的重点不是泛泛而谈的“代码优雅性”，而是：

- 是否偏离需求
- 是否漏改关键链路
- 是否引入明显风险
- 是否只修了表面、没修透传链路

---

## 不建议承担的职责

这个 QA 不应该做：

- 主实现
- 主设计
- 需求再定义
- 风格化建议泛滥
- 没看证据时凭经验猜问题

它应该避免退化成“爱发表意见的 reviewer”。

---

## 建议工作方式

### 推荐形态

更合适的形态是：

> **长期存在的固定 QA 角色，按需被调用。**

而不是一个持续插入主会话的在线第三人。

### 推荐介入节点（待后续再定）

候选节点包括：

1. Arc 说“做完了”之后做一次验收
2. 宣称“修好了/完成了”之前做一次 verification
3. 高风险改动（cron、外发链路、生产脚本、配置）必须额外过 QA
4. 当结果看起来不对，但问题还说不清时，让 QA 只看证据做判定

目前这个“默认什么时候介入”还没有最终拍板，后续需要单独定规则。

---

## 建议输出格式

这个 QA 的输出应尽量固定、克制、证据化，避免散文式 review。

建议格式：

- `status: pass | issues`
- `checked_against:` 对照的 spec / plan / 用户要求
- `evidence:`
  - 具体文件
  - 具体命令
  - 具体输出
- `findings:`
  - 已满足项
  - 未满足项
- `verdict:`
  - 可交付 / 不可交付

原则：

- 没看文件，不下结论
- 没跑验证，不说通过
- 没证据，不做判断
- 不代写实现

---

## 当前判断

如果要引入第三角色，最值得尝试的不是“多一个会提意见的同事”，而是：

> **一个专门防止假闭环的证据型 QA。**

这个方向和贵平当前对协作质量的要求是契合的：

- 不接受“差不多”
- 要看真实产物
- 要看验证证据
- 要区分“说完成了”和“真的完成了”
