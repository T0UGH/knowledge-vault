# 以后如何给 Codex 派源码解析任务

## 一句话结论

如果后面还要继续让 Codex 写源码解析，最重要的不是“让它开始写”，而是：

> **先把文章的主问题、目标读者、层级位置、相邻边界和证据链钉死，再让它写。**

否则很容易得到一篇信息很多、也不算错，但边界模糊、层级混乱、像“又一篇文章”而不是“书的一章”的稿子。

---

## 为什么不能直接让 Codex 开写

如果只给一句类似：

- 给 Codex 写一篇源码解析
- 再写一篇这种风格的
- 让 Codex 跑这篇

那 Codex 最容易做出来的东西是：

- 会讲很多信息
- 结构表面完整
- 但一篇里同时想讲四件事
- 容易把主循环、对象定义、扩展层、工程判断混在一起
- 与上一篇 / 下一篇边界不清

所以真正要做的，不是直接让它写，而是先替它完成任务定义。

---

## 派单前必须先补的五件事

### 1. 这篇到底回答什么问题

不要说“讲 QueryEngine”或“讲某个源码文件”，而要钉成：

- QueryEngine 为什么是主循环入口？
- 一次请求是怎么进入 QueryEngine 的？
- QueryEngine 和 query 主循环怎么分工？
- 为什么它不是普通请求包装器？

也就是：

> **每篇源码解析都必须是单问题表达。**

一篇文章最怕“一次讲四件事”。

---

### 2. 目标读者是谁

至少先判断它是写给谁：

- 第一次看这个系统的人
- 已经用过，但不懂内部的人
- 想做同类系统的工程师
- 只想查一个源码点的人

这个判断直接决定写法：

- 新人向：先立问题，再讲对象，再讲链路
- 工程师向：更快进入结构、边界和工程取舍
- 检索向：少铺垫，多给调用链和定位图

---

### 3. 这篇属于哪一层

在开始写前，必须先定它属于哪一层：

- 系统全景层
- agent turn 主链层
- 执行层
- 上下文 / 状态层
- 扩展层
- 多 agent 协作层
- 控制层 / 产品层

如果层级不先定，Codex 很容易写成：

- 一半讲对象定义
- 一半讲主循环
- 一半再讲扩展
- 最后再来一段产品判断

看起来都对，但整体已经跑偏。

---

### 4. 这篇和相邻文章怎么分工

在派单前，要先写清：

- 这篇不讲什么
- 这篇只讲什么
- 它和上一篇怎么切边界
- 它怎么把读者送到下一篇

很多源码解析的问题，不在单篇质量，而在相邻篇都在重新解释同一个东西。

所以要把“边界”当成任务的一部分写进去。

---

### 5. 这篇最低必须讲出的证据链是什么

不要只给标题，要给证据框架。

至少包括：

- 必读源码文件
- 最低必须讲出的调用链 / 数据链
- 旧文或旧材料锚点
- 本篇禁止越界的话题

这样 Codex 才不会只做“文件介绍”或“泛泛解释”。

---

## 正确的执行顺序

### 第一步：先让 Codex 出骨架，不是直接铺正文

更稳的做法是先让它交：

1. 文章主问题
2. 文章结构（每节一句职责）
3. 必须讲出的调用链
4. 与上一篇 / 下一篇的边界
5. 本篇禁止越界的话题

也就是说：

> **先交骨架，再交正文。**

---

### 第二步：先审骨架，再允许展开正文

审骨架时，重点看四件事：

1. 有没有单一主问题
2. 有没有越界
3. 结构是不是按读者理解顺序排的
4. 证据链是不是够硬

骨架没过，正文越多，返工越大。

---

### 第三步：再让 Codex 展开正文

展开正文时，要求它：

- 不要从空话开场
- 每段都服务主问题
- 讲源码时不是“文件介绍”，而是“系统职责 + 调用位置 + 为什么这样设计”
- 结尾要自然导流下一篇，而不是空泛总结

---

### 第四步：最后做 reader review / friction review

最后还要补一轮“读者阻力检查”，重点看：

- 哪些地方作者自己懂，但读者不一定懂
- 哪些术语太密
- 哪些段落信息正确，但没有把读者带进去
- 哪些地方属于结构上能读，但阅读体验仍有摩擦

Codex 往往能把信息铺出来，但不一定天然懂“读者阻力”。

---

## 如果是整套系列，不要按“想到哪篇写哪篇”推进

如果后面不是单篇，而是继续做一整套源码解析，那就不能再按“哪个文件有意思就写哪个”。

更稳的顺序应该是：

1. 先定卷级或阶段级结构
2. 再列映射表和缺口清单
3. 再补导读层
4. 最后才派单篇

否则最后只会得到很多高质量文章，但不会得到一套真正可连续阅读的书。

---

## 我会怎么给 Codex 下任务

以后更理想的派单方式，应该像这样：

> 你要写的不是一篇泛泛的源码介绍，而是一篇只回答一个问题的源码解析文章。
> 目标读者是已经用过该系统、但不理解内部实现的人。
> 本篇属于某一层（例如执行层），不负责解释主循环总览，也不负责扩展层。
> 请先基于指定源码文件建立结构化骨架：
> 1. 主问题
> 2. 文章结构（每节一句职责）
> 3. 必须讲出的调用链
> 4. 与上一篇 / 下一篇的边界
> 5. 本篇禁止越界的话题
> 骨架确认后再展开正文。

这比一句“写一篇关于 xxx 的源码解析”强很多。

---

## 建议固定成长期模板

以后如果要频繁让 Codex 处理这类任务，可以直接固定成一个模板。

### Codex 源码解析派单模板

1. 文章标题
2. 一句话主问题
3. 目标读者
4. 所属层级 / 卷
5. 上一篇 / 下一篇
6. 本篇不讲什么
7. 必读源码文件
8. 最低必须讲出的调用链
9. 旧材料锚点
10. 先交骨架，再交正文
11. 最后做读者阻力审查

这个模板一旦固定，后面输出质量会稳很多。

---

## 最短版总结

如果只保留最短版，我会写：

> 以后让 Codex 写源码解析，不该直接让它开始写，而应该先替它把问题、边界、层级位置和证据链钉住。先让它交骨架，审过之后再展开正文。这样出来的才更像“书的一章”，而不是“又一篇不错的文章”。
