## 一、卷一重写的核心判断

新卷一不应该继续承担“工具卷开头”或“零散源码导读入口”的职责。

它应该被重建成：

> **Claude Code 系统地图卷 / 全景导论卷**

也就是说，卷一的任务不是先把某个局部讲透，而是先让读者脑中形成一张稳定的系统地图：

- Claude Code 是什么系统，不是什么系统
- 用户输入、主循环、工具、上下文、扩展、agent 分别在什么位置
- 后面几卷为什么按那个顺序展开

读完卷一后，读者应该获得的是：

1. 一套可站住的总图
2. 一组核心对象的边界感
3. 对后续各卷阅读顺序的心理预期

而不是某一个源码文件的细节掌握。

---

## 二、卷一的写作原则

### 1. 少讲工程细枝末节，先讲清系统分层
卷一不需要细抠太多实现细节。它要做到的是：
- 把每一部分说清楚
- 把对象关系讲明白
- 把动态主线和静态结构立起来

### 2. 少篇幅、少文章
卷一不应再拆成十几二十篇碎片。应控制为少数大章节，形成更强的导论感。

### 3. Mermaid 图承担主解释任务
卷一应该大量使用 Mermaid 图，而且图不是装饰，而是结构解释的主承载体。

### 4. 先建立地图，再把读者送进后续各卷
卷一的最佳效果不是“局部讲得很深”，而是“后面每一卷为什么存在，现在已经被读者预先理解了”。

---

## 三、建议的新卷一篇数

我当前建议：

> **6 篇**

原因：

- 5 篇略紧，容易把几个本应分开的台阶挤在一起
- 7 篇开始有重新变碎的风险
- 6 篇最适合形成“全景导论卷”的节奏：既完整，又不会散

所以当前默认建议是：

> **新卷一 = 6 篇大章节**

## 五、卷一不该做什么

### 1. 不要变成 Tool 细节卷
不要一开始就把读者拖进：
- `Tool.ts`
- `BashTool.tsx`
- FileRead / FileEdit / ToolExecution

这些应该放到后续执行能力卷里深入。

### 2. 不要变成源码遍历卷
卷一不是“我先带你按目录逛一圈源码”。

### 3. 不要把每个模块都讲深
卷一的任务是立图，不是先抢后面几卷的活。

### 4. 不要缺少阅读导航感
卷一读完，读者必须知道：
- 下一卷为什么存在
- 后面每一卷大概会继续讲什么

## 六、我当前推荐的卷一结构节奏

### 节奏建议

1. 先立系统定位
2. 再立核心对象地图
3. 再立动态主线
4. 再立执行能力层
5. 再立上下文与持续工作层
6. 最后立扩展地图和全书导航

这套顺序的核心，是：

> **先让读者知道它是什么，再让读者知道它怎么动，再让读者知道它怎么持续和怎么扩展。**

---

## 七、对现有文章的态度

新卷一可以：

- 参考现有文章里的判断和素材
- 复用部分段落或图
- 复用已经成立的结论

但不需要：

- 强绑定现有卷一文章顺序
- 强行保留原来的拆分粒度
- 因为“已经写过”就硬塞回卷一

换句话说：

> **新卷一应该按新目标重写，而不是按旧文章拼装。**
