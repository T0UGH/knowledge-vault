# Claude Code 源码导读手册这波写书经验与踩坑复盘

## 一句话结论

这波最深的体感不是“单篇怎么写”，而是：

> **技术写书最难的不是把单篇写深，而是先把整本书的认知坡度排顺。**

结构没排顺，越写越乱；结构一旦排顺，单篇改写、导读页补齐、GitHub Pages 站点化，都会顺很多。

---

## 这波最值钱的经验

### 1. 先定“卷要回答什么”，再定文章放哪

这是最关键的一条。

最容易犯的错，是先看手里已有多少文章，再想怎么归档。这样做出来的往往不是书，而是资料重排。

更有效的顺序是：

1. 先定这一卷到底回答什么问题
2. 再决定哪些旧文能服务这个主问题
3. 不服务的就挪走、降级、补桥接，甚至直接退出主链

也就是说：

> **卷的主问题，必须先于旧材料的归档冲动。**

---

### 2. 写书和写系列文章，是两套完全不同的组织逻辑

系列文章可以容忍：

- 探索痕迹
- 重复开场
- 同一概念多次重新解释
- 写作顺序驱动阅读顺序

但书不行。

书要求的是：

- 每卷有独立主问题
- 每篇有单一职责
- 相邻篇之间能自然导流
- 重复内容要被压掉
- 读者不能靠“跟着作者当时研究路径走”来理解

所以这次最重要的转变，不是“继续写文章”，而是把旧系列从**研究顺序**切换成**阅读顺序**。

---

### 3. 卷级导读和 index 不是装饰，而是正文的一部分

以前很容易把 README、卷导读、总览页当成外围包装。

这波推进下来，更明确的判断是：

> **导读页不是外壳，它就是书的阅读控制层。**

它承担了正文不方便承担、但又必须有人承担的工作：

- 解释这一卷在全书里的职责
- 降低新读者进入门槛
- 给相邻章节切边界
- 指出建议阅读路径

没有这些页，正文即使很强，读者也容易迷路。

---

### 4. 弱去重比整篇重写更划算

这次一个很实用的收获是：

> **很多问题不是文章不行，而是相邻文章的职责没切开。**

高性价比的做法通常不是大重写，而是：

- 改导语
- 明确这篇不解决什么
- 明确这篇只解决什么
- 给上一篇 / 下一篇加分工关系
- 删掉少量重复开场

也就是做“弱去重”。

它的价值在于：

- 成本低
- 不破坏原文核心判断
- 能快速改善连读体验
- 不容易滑进“整理着整理着重新写了一本”的坑

---

### 5. 站点化必须后置，内容信息架构先行

这波也再次验证了一条：

> **不要太早陷进建站、主题、样式、导航形式；先把内容骨架做对。**

真正决定项目能不能立住的，是：

- 目录结构稳不稳
- 卷级主问题清不清楚
- 导读层是否成立
- 旧资产与新入口的关系有没有讲明白
- 后续新文章并入规则有没有定死

这些没定前就上站，只是把混乱上墙。

---

### 6. 旧资产保留，但退出主导航，是很好的过渡策略

内容项目从 v1 升级到 v2 时，最容易走两个极端：

- 舍不得旧内容，导致新旧混在主导航里
- 一刀砍掉旧内容，导致历史索引和旧链接全断

这次更稳的做法是：

- `guidebookv2/` 作为正式入口
- `guidebook/` 继续保留
- 但旧内容不再占主导航

这个策略兼顾了三件事：

- 新读者不会迷路
- 老读者还能回看旧索引
- 项目自己保留回滚余地

---

## 这波比较硬的踩坑

### 1. 最容易高估“我脑子里已经清楚了”

这是第一大坑。

作者自己熟，很容易觉得：

- 这两篇明明不一样
- 这一卷和下一卷边界我自己知道
- 读者应该也能顺着看懂

但真落到目录和导航上，问题就会暴露：

- 两篇都在重新定义同一个东西
- 同一概念在不同卷抢职责
- 卷标题说的是 A，正文实际在讲 B
- 首页、README、站内总览三套口径不一致

所以很重要的一条教训是：

> **你脑子里清楚，不等于结构已经清楚；边界必须被显式写出来。**

---

### 2. 旧编号体系很容易反客为主

旧内容一多，就会不自觉地继续按历史编号思考。

这样会带来几个问题：

- 写作顺序误导阅读顺序
- 目录为了迁就旧编号而弯掉
- 新卷主问题被旧材料反向定义

所以更稳的做法是：

> **旧编号适合做索引，不适合继续做主结构。**

它可以保留映射价值，但不该继续占主导航中心位置。

---

### 3. 卷一最容易沦为大杂烩

这是技术书里特别常见的坑。

因为大家总想把所有前置认知都塞进卷一，结果卷一就会混入：

- 总览
- 术语表
- 对象介绍
- 主循环预告
- 扩展系统预告
- 方法论预告

最后什么都沾一点，但没有真正立稳。

所以一个重要原则是：

> **卷一最需要克制。它不是装下所有前置知识，而是只负责给出最小但稳定的总图。**

---

### 4. 相邻篇最容易“都对，但连起来不对”

长项目里一个很隐蔽的坑是：

- 单篇看都没问题
- 每篇也都有信息量
- 每篇都能自圆其说

但连起来看，会出现：

- 节奏断裂
- 重复解释
- 视角忽高忽低
- 上一篇刚立住的概念，下一篇又换说法
- 读者不知道为什么此刻该看这一篇

所以真正的单位不是“单篇质量”，而是：

> **相邻篇关系质量。**

这也是为什么后面会需要弱去重执行表——它本质上不是审单篇，而是在审篇与篇之间的摩擦。

---

### 5. 太早追求文采，会把结构问题盖住

这种项目很容易写着写着就开始追求：

- 更漂亮的总结句
- 更有气势的结尾
- 更顺滑的表达

但如果结构和边界还没钉牢，文采越强，越容易把问题遮住。

更稳的顺序应该是：

1. 先把结构钉死
2. 再把职责切清
3. 再做弱去重
4. 最后再磨语言

否则就容易写出“好看，但不能做骨架”的文章。

---

### 6. 只盯正文，不同步 README / 首页 / 导航，会导致公开口径分裂

这次在切 GitHub Pages 与 README 时，这个问题很明显：

- 站点已经切到 v2 了
- 但 README 还是旧六卷口径
- 首页入口还导向旧 guidebook
- 仓库首页和站内口径不一致

这类问题不难修，但很容易被忽略。外部读者看到时，会立刻感觉割裂。

所以一个很实用的检查项是：

> **每次结构升级，都要同步检查四层：站点导航、站点首页、仓库 README、仓库结构说明。**

---

## 如果把这波经验压成几条“写书原则”

我会留下面这 8 条：

1. 卷先于文
2. 主问题先于材料归档
3. 读者理解坡度先于源码目录
4. 导读页不是包装，是正文的一部分
5. 弱去重优先于整篇重写
6. 旧资产要保留，但不能继续占主结构
7. 单篇质量不等于连读质量
8. 内容升级时，README / 首页 / 导航必须一起改

---

## 如果再来一轮，我会更早做的事

### 1. 更早建立“每卷一句话职责”

比如：

- 卷一：这是什么系统
- 卷二：一轮怎么跑
- 卷三：怎么真正做事
- 卷四：怎么持续工作
- 卷五：怎么长能力
- 卷六：怎么协作
- 卷七：怎么收成产品

这种一句话钉子越早定，后面越省命。

### 2. 更早做“相邻篇冲突扫描”

每篇都尽早回答四个问题：

- 这篇只回答什么
- 这篇不回答什么
- 和上一篇怎么分工
- 和下一篇怎么交接

这能提前消掉很多返工。

### 3. 更早把 README 和站点入口当成正文工程的一部分

它们不是收尾工作，而应该更早进入结构一致性检查。

---

## 最短版总结

如果只保留最短版，我会写：

> 这波最大的收获是：技术写书最难的不是把单篇写深，而是把整本书的认知坡度排顺。旧文章不是不能用，但必须先退出历史顺序，重新服从卷级主问题。真正高性价比的工作也不是大重写，而是补导读、切边界、做弱去重、统一入口口径。最大的坑则是：作者自己以为已经清楚，结果结构并没有被显式写清。
