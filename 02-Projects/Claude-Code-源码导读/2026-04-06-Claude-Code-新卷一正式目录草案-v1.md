## 一、卷一总定位

新卷一的角色已经明确：

> **Claude Code 系统全景导论卷**

它的任务不是深挖某个局部模块，而是先给读者搭出整张系统地图：

- Claude Code 到底是什么系统
- 系统里有哪些核心对象
- 一次请求怎么大致跑起来
- 模型意图怎么落成执行能力
- 上下文与状态为什么重要
- 系统又是怎样继续长出更多能力

也就是说，卷一读完后，读者最该获得的不是某个源码文件的细节掌握，而是：

1. 一张稳定的系统总图
2. 一套对象边界感
3. 一条后续阅读导航路径

---

## 二、卷一的结构原则

### 1. 开头先亮总图
第一篇必须先给一张系统地图，再反过来回答：
- Claude Code 到底是什么系统
- 这本书为什么应该这样读

### 2. 中间偏静态结构，再带动态主线
卷一的中段应以静态结构为主、动态流程为辅，大致是：
- 静态结构 60
- 动态流程 40

### 3. 结尾以扩展能力地图为主，以全书导航为辅
卷一最后一篇不只是讲“还能扩展”，还要顺手把读者送向后面的卷。

大致配比：
- 扩展能力地图 60
- 全书阅读导航 40

### 4. 少篇幅、少文章、多 Mermaid 图
卷一不再拆成十几二十篇碎片，而是控制为少数大章节。

当前建议：

> **6 篇**

并且每篇至少要有 1 张核心 Mermaid 图，图承担结构解释任务，而不是装饰。

---

## 三、新卷一正式目录草案（6 篇）

---

### 01｜Claude Code 到底是什么系统

**这篇要解决的问题**
- Claude Code 到底是什么
- 它和普通聊天产品、命令行工具、脚本集合的区别是什么
- 为什么要把它理解成一个 agent runtime
- 为什么这本书不该按源码目录逛，而该按系统地图来读

**这篇的卷内作用**
- 全书入口篇
- 卷一总起篇
- 阅读姿势校准篇

**这篇最该留下的判断**
> Claude Code 不是一个聊天壳，而是一套把模型、运行时、执行能力、上下文和扩展能力编织在一起的 agent runtime。

**建议 Mermaid 图**
- Claude Code 系统总图
- 全书阅读导航图（简版）

---

### 02｜Claude Code 由哪些核心对象组成

**这篇要解决的问题**
- prompt / command / tool / agent / skill / context / session / runtime 各是什么
- 这些对象彼此是怎样协作的
- 后面每一卷大致是在继续深拆哪些对象

**这篇的卷内作用**
- 核心对象边界篇
- 名词地图篇

**这篇最该留下的判断**
> Claude Code 不是一条单线系统，而是多类对象协同工作的运行时；后面所有细节，都可以回收进这张对象地图里理解。

**建议 Mermaid 图**
- 核心对象关系图
- 对象分层图（用户层 / runtime 层 / capability 层 / extension 层）

---

### 03｜一次请求是怎么跑成一次 Agent Turn 的

**这篇要解决的问题**
- 用户输入之后，系统大概发生了什么
- 为什么 Claude Code 的核心不是“回答一句话”，而是“完成一轮 agent turn”
- tool_use / tool_result / continuation 为什么会成为主循环的一部分

**这篇的卷内作用**
- 动态主线预览篇
- 卷二导流篇

**这篇最该留下的判断**
> Claude Code 的基本运行单位不是“单条回复”，而是一轮可持续展开的 agent turn。

**建议 Mermaid 图**
- Agent turn 总流程图
- 输入 → query → tool_use → tool_result → continuation 闭环图

---

### 04｜Claude Code 怎么把模型意图落成执行能力

**这篇要解决的问题**
- tool 在全系统中的位置是什么
- 为什么模型不是直接调用函数，而是通过 runtime 把结构化意图落成真实动作
- 本地工具、搜索、写文件、命令执行这些能力为什么能被统一接住

**这篇的卷内作用**
- 执行能力层总图篇
- 工具系统预告篇
- 卷三导流篇

**这篇最该留下的判断**
> Claude Code 的执行能力层，是 runtime 把模型意图翻译成现实动作的关键接口层，而 tool 是这层里的正式执行对象。

**建议 Mermaid 图**
- `tool_use -> orchestration -> execution -> tool.call -> tool_result` 总图
- Tool 抽象层与具体工具样本关系图

---

### 05｜Claude Code 怎么维持上下文、状态与持续工作

**这篇要解决的问题**
- 为什么 Claude Code 不是“一问一答后就重启”
- message / context / system prompt / session 在系统里大概如何协同
- 为什么 collapse / compact / restore 这类机制会存在

**这篇的卷内作用**
- 持续工作能力总览篇
- 卷四导流篇

**这篇最该留下的判断**
> Claude Code 真正难的，不只是调用能力，而是怎样长期工作而不失控；上下文与状态管理是这套系统的生命线。

**建议 Mermaid 图**
- Context 生命周期图
- messages / prompt / compact / restore 关系图

---

### 06｜Claude Code 怎么长出更多能力

**这篇要解决的问题**
- skill / agent / subagent / MCP / plugin 各在什么位置
- 为什么 Claude Code 不是一个封闭系统
- 这些扩展点分别会在后续哪一卷继续展开

**这篇的卷内作用**
- 扩展地图篇
- 全书导航收束篇
- 卷五、卷六导流篇

**这篇最该留下的判断**
> Claude Code 不是一个固化产品，而是一套能够持续吸纳新能力的可扩展 runtime；后面几卷，本质上都在继续拆它怎样长能力、怎样控制能力。

**建议 Mermaid 图**
- 扩展能力地图
- skill / agent / subagent / MCP / plugin 关系图
- 全书阅读导航图（完整版）

## 四、这 6 篇之间的节奏关系

如果把卷一压成一句阅读路径，可以写成：

> 先知道这是什么系统 → 再知道它由哪些对象构成 → 再知道它怎么跑一轮 → 再知道它怎么执行 → 再知道它怎么持续 → 最后知道它怎么扩展。

这套顺序的好处在于：

1. 先立静态地图，再引出动态主线
2. 先告诉读者系统是什么，再告诉读者后面该怎么拆
3. 每篇只回答一个大问题，不会重新掉进源码散点阅读

---

## 五、卷一里不建议先讲深的东西

卷一不建议单独展开成大篇的内容包括：

- BashTool 细讲
- FileReadTool / FileEditTool / GrepTool 细讲
- AgentTool / runAgent 细讲
- skill frontmatter 字段细讲
- MCP auth / plugin loader 细讲
- compact / microcompact / snip 细讲

这些内容都应该放到后续卷里成为正菜。

卷一里只需要做到：

- 先点到它们的位置
- 先说明它们的重要性
- 先解释它们为什么会在后面被展开

---

## 六、对现有文章的态度

新卷一可以：

- 参考现有文章里的判断
- 复用已经成立的结论
- 挪用局部段落与图示思路

但不需要：

- 强绑现有卷一文章顺序
- 强行保留旧拆分粒度
- 因为“已经写过”就硬塞进新卷一

换句话说：

> 新卷一应该按新目标重写，而不是按旧文章拼装。
