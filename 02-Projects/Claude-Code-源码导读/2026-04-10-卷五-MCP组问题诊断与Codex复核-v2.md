# 卷五 MCP 组问题诊断与 Codex 复核 v2

## 结论

卷五的 MCP 组当前**不是观点错**，而是**09 / 10 / 11 三篇职责差还没完全拉开**。

更准确地说：

- **09** 应该是破误解篇
- **10** 应该是锚点主链篇
- **11** 应该是边界裁决篇

但当前状态更像：

- 三篇都在努力抬高 MCP 的对象级别
- 都在反复证明 MCP 不只是“远程工具扩容”
- 但谁负责拆误解、谁负责讲主链、谁负责收边界，还没有彻底分开

所以问题不是“写得差”，而是：

> **09 / 10 / 11 都在往对的方向写，但组内坡度还不够清，导致锚点不够稳、边界感不够利。**

---

## Leo 的判断

### 1. 09 的问题：已经开始吃 10 的主菜

#### 现在在做什么
- 先拆掉 “MCP = 多了一批远程工具” 的浅理解
- 用 resources / auth / permission / state 说明 MCP 的对象级别更高

#### 问题在哪
09 理论上应该是纯粹的**破误解篇**。

但现在它已经开始讲：
- 外部节点怎样被识别
- resources 怎样进入工作面
- 工具怎样被翻译进统一语义
- 外部能力怎样继续接受治理

这些内容本身没错，但它们已经明显进入 **10 的接入主线** 了。

#### 结果
读者会感觉：
- 09 已经在讲“接入链”的骨架
- 10 再讲接入链时，就容易显得重复

#### 应该怎么收
09 应该更像一把刀，只做一件事：

> **先把“MCP 只是远程工具扩容”这个理解砍掉。**

更具体一点：
- 重点讲为什么 resources / needs-auth / permission pipeline 足以证明它不是远程工具扩容
- 少讲完整接入链
- 少预演 10

---

### 2. 10 的问题：锚点篇还不够“主轴化”

#### 现在在做什么
10 已经有很多正确材料：
- 外部 server / resource system
- 配置归并与类型收口
- 连接与状态
- tools / prompts / resources 暴露
- 翻译成内部对象
- 进入当前能力面与执行链
- 统一治理

#### 问题在哪
10 理论上应该是 MCP 组的**锚点篇**。

也就是说，读完 10，读者脑中应该留下非常稳定的一条主线：

> 外部能力节点  
> → 配置与状态收口  
> → tools / prompts / resources 暴露  
> → 翻译成内部对象  
> → 进入能力面与执行链  
> → 接受统一治理

但现在 10 里还有一些内容在继续帮 09 证明：
- 它不只是远程工具
- 它是能力源接入层
- resources 说明它更高一级

这些判断没错，但会削弱 10 作为**主链篇**的锚点感。

#### 应该怎么收
10 应该更干净地压成一条主轴：

1. 外部 server 成为可管理节点
2. 状态与认证成立
3. tools / prompts / resources 暴露
4. 翻译为内部对象
5. 进入能力面与执行链
6. 接受统一治理

也就是：

> **10 要少证明“它不是什么”，多证明“它是怎样进入 runtime 的”。**

---

### 3. 11 的问题：层级说明对了，但还不够像边界裁决

#### 现在在做什么
11 已经在切：
- MCP vs skills
- MCP vs hooks
- MCP vs plugins

这方向是对的，而且结构也算清楚。

#### 问题在哪
它现在更像“对象关系说明文”，而不够像“边界裁决文”。

也就是说，读者读完会知道：
- skills 是方法组织层
- MCP 是外部能力源接入层
- hooks 是接缝层
- plugins 是封装层

但不一定会因此更会判断：
- 我这个需求到底为什么更像 MCP，不像 skill？
- 我现在是在接外部能力，还是在组织使用方式？
- plugin 里带 MCP，和“plugin 就是 MCP 打包壳”差在哪里？

#### 应该怎么收
11 要少一点“关系图”，多一点“判断门槛”。

比如每组都给一句更实战的切法：
- 缺方法，不缺能力源 → 更像 skill
- 缺系统外正式入口 → 更像 MCP
- 缺 runtime 可插手节点 → 更像 hooks
- 缺统一安装 / 治理 / 分发壳 → 更像 plugin

这样 11 才更像真正的**边界裁决篇**。

---

### 4. MCP 组整体问题

#### 概念密度偏高
这组三篇都有：
- 能力源
- 资源系统
- 外部节点
- 接入层
- 治理
- 封装层

这些词都对，但如果没有更多真实判断场景托住，读者会知道“概念更高了”，却不一定知道“这会改变我什么判断”。

#### 文体差异不够明显
MCP 组三篇现在都比较像：
- 技术概念文
- 架构解释文
- 对象分层文

但如果想形成真正的组内坡度，应该更明显地区分成：
- **09：破误解篇**
- **10：接入主链篇**
- **11：边界裁决篇**

#### 真实判断场景不足
skills 组后来越写越顺，是因为不断回到：
- 用户为什么会觉得这次顺、下次飘
- inline / fork 为什么差这么大
- 一个 skill 为什么会跑坏
- 什么时候该用 skill，不该用 tool / agent / MCP

而 MCP 组目前还偏对象层，还没完全压回“真实使用时怎么判断”。

## Codex 复核结论

Codex 的总体判断与 Leo 基本一致：

> **09 / 10 / 11 的意图分工其实是清楚的：09 应该是反误解，10 应该是组内锚点，11 应该是边界收口。但执行上还没有真正拉开。**

### Codex 对 09 的判断
- 09 有明显越界，提前吃掉了 10 和 11 的职责。
- 已经在讲 auth、permission、以及“否则后两篇会怎样滑坡”，会让 09 从“反误解短文”膨胀成“半个总论”。
- 建议把 auth / permission 只保留成 1 段证据，不再单列大节。
- 09 的核心判别式还不够“短硬”，建议直接收成一个更强的三层对象判断：
  - `server/node` 是接入对象
  - `tool` 是动作入口
  - `resource` 是材料入口
  - 所以 MCP 不是 remote tools list
- 09 的元叙述偏多，例如“第 10 篇会怎样、第 11 篇会怎样、这篇不展开什么”，编辑味太重，削弱文章独立性。

### Codex 对 10 的判断
- 10 现在“像锚点”，但还不是“强锚点”。
- 还在反复做 09 的“反误解”工作，锚点文应该更像“标准主链说明书”，少一点辩论，多一点定版结构。
- 建议把主链改成固定 5 步：
  1. 配置收口
  2. 连接 / 认证
  3. 能力面发现
  4. 内部对象翻译
  5. 执行链治理
- 10 缺一次性钉死的术语定义：
  - `server`
  - `node`
  - `tool`
  - `resource`
  - `internal object`

### Codex 对 11 的判断
- 11 还不是很像“边界裁决文”，更像“概念分层文”。
- 建议加一个**边界决策表**：
  - 外部系统接入 → MCP
  - 工作方法组织 → skill
  - runtime 插手点 → hook
  - 多对象封装 / 分发 → plugin
- 11 的“这篇不能吃掉后文正文”这类元叙述太多。
- plugin 的边界还不够具体，缺少最关键的裁决句：
  - 什么时候只是 MCP 配置
  - 什么时候是 plugin 打包 MCP
  - 什么时候是 skill 调用 MCP

## Codex 给出的 Top 3 组级修复

### 1. 重新切文章所有权
- **09** 只做反误解升级视角
- **10** 独占接入主链
- **11** 独占边界裁决

### 2. 给每篇一个不可替代的结构件
- **09**：保留 “remote tools vs capability-source” 对照图
- **10**：增加 “5 步接入链” 标准表
- **11**：增加 “边界决策表 / 误判示例”

### 3. 大幅删减元叙述和重复口号
例如：
- “这篇不展开什么”
- “否则后文会滑坡”
- “不要混桶”

这些编辑意识太强，对读者收益偏低。

## 归纳判断

Leo 和 Codex 的结论已经形成交叉验证：

1. **09 吃到了 10 的主线**
2. **10 锚点感还不够强**
3. **11 还不够像真正的边界裁决篇**
4. **三篇都在努力抬高 MCP 的对象级别，但职责差还没彻底拉开**

最优先的修复顺序仍然是：

### 第一优先：先把 10 立成真正的锚点篇
这是最关键的。

### 第二优先：把 09 收回成更短、更利的破误解篇
别继续吃 10。

### 第三优先：把 11 改成真正的边界裁决篇
少一点关系图，多一点判断标准。

---

## 一句话总结

> 卷五 MCP 组当前最大的问题，不是观点不对，而是三篇都在抬高 MCP 的对象级别，却还没彻底拉开“破误解 / 讲主链 / 收边界”的职责差。Leo 和 Codex 的判断基本一致，后续最值的动作是先按 orchestrator 路线把 10 这篇锚点篇打稳。
