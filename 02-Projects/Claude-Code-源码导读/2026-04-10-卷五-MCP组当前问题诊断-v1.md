# 卷五 MCP 组当前问题诊断 v1

## 结论

卷五的 MCP 组现在**不是观点错**，而是**三篇职责差还没完全拉开**。

更准确地说：

- **09** 应该是破误解篇
- **10** 应该是锚点主链篇
- **11** 应该是边界裁决篇

但当前状态更像：

- 三篇都在努力抬高 MCP 的对象级别
- 都在反复证明 MCP 不只是“远程工具扩容”
- 但谁负责拆误解、谁负责讲主链、谁负责收边界，还没有彻底分开

所以问题不是“写得差”，而是：

> **09 / 10 / 11 都在往对的方向写，但组内坡度还不够清，导致锚点不够稳、边界感不够利。**

---

## 09 的问题：已经开始吃 10 的主菜

### 现在在做什么
- 先拆掉 “MCP = 多了一批远程工具” 的浅理解
- 用 resources / auth / permission / state 说明 MCP 的对象级别更高

### 问题在哪
09 理论上应该是纯粹的**破误解篇**。

但现在它已经开始讲：
- 外部节点怎样被识别
- resources 怎样进入工作面
- 工具怎样被翻译进统一语义
- 外部能力怎样继续接受治理

这些内容本身没错，但它们已经明显进入 **10 的接入主线** 了。

### 结果
读者会感觉：
- 09 已经在讲“接入链”的骨架
- 10 再讲接入链时，就容易显得重复

### 应该怎么收
09 应该更像一把刀，只做一件事：

> **先把“MCP 只是远程工具扩容”这个理解砍掉。**

更具体一点：
- 重点讲为什么 resources / needs-auth / permission pipeline 足以证明它不是远程工具扩容
- 少讲完整接入链
- 少预演 10

---

## 10 的问题：锚点篇还不够“主轴化”

### 现在在做什么
10 已经有很多正确材料：
- 外部 server / resource system
- 配置归并与类型收口
- 连接与状态
- tools / prompts / resources 暴露
- 翻译成内部对象
- 进入当前能力面与执行链
- 统一治理

### 问题在哪
10 理论上应该是 MCP 组的**锚点篇**。

也就是说，读完 10，读者脑中应该留下非常稳定的一条主线：

> 外部能力节点  
> → 配置与状态收口  
> → tools / prompts / resources 暴露  
> → 翻译成内部对象  
> → 进入能力面与执行链  
> → 接受统一治理

但现在 10 里还有一些内容在继续帮 09 证明：
- 它不只是远程工具
- 它是能力源接入层
- resources 说明它更高一级

这些判断没错，但会削弱 10 作为**主链篇**的锚点感。

### 应该怎么收
10 应该更干净地压成一条主轴：

1. 外部 server 成为可管理节点
2. 状态与认证成立
3. tools / prompts / resources 暴露
4. 翻译为内部对象
5. 进入能力面与执行链
6. 接受统一治理

也就是：

> **10 要少证明“它不是什么”，多证明“它是怎样进入 runtime 的”。**

---

## 11 的问题：层级说明对了，但还不够像边界裁决

### 现在在做什么
11 已经在切：
- MCP vs skills
- MCP vs hooks
- MCP vs plugins

这方向是对的，而且结构也算清楚。

### 问题在哪
它现在更像“对象关系说明文”，而不够像“边界裁决文”。

也就是说，读者读完会知道：
- skills 是方法组织层
- MCP 是外部能力源接入层
- hooks 是接缝层
- plugins 是封装层

但不一定会因此更会判断：
- 我这个需求到底为什么更像 MCP，不像 skill？
- 我现在是在接外部能力，还是在组织使用方式？
- plugin 里带 MCP，和“plugin 就是 MCP 打包壳”差在哪里？

### 应该怎么收
11 要少一点“关系图”，多一点“判断门槛”。

比如每组都给一句更实战的切法：
- 缺方法，不缺能力源 → 更像 skill
- 缺系统外正式入口 → 更像 MCP
- 缺 runtime 可插手节点 → 更像 hooks
- 缺统一安装 / 治理 / 分发壳 → 更像 plugin

这样 11 才更像真正的**边界裁决篇**。

## MCP 组整体问题

### 1. 概念密度偏高
这组三篇都有：
- 能力源
- 资源系统
- 外部节点
- 接入层
- 治理
- 封装层

这些词都对，但如果没有更多真实判断场景托住，读者会知道“概念更高了”，却不一定知道“这会改变我什么判断”。

### 2. 文体差异不够明显
MCP 组三篇现在都比较像：
- 技术概念文
- 架构解释文
- 对象分层文

但如果想形成真正的组内坡度，应该更明显地区分成：
- **09：破误解篇**
- **10：接入主链篇**
- **11：边界裁决篇**

### 3. 真实判断场景不足
skills 组后来越写越顺，是因为不断回到：
- 用户为什么会觉得这次顺、下次飘
- inline / fork 为什么差这么大
- 一个 skill 为什么会跑坏
- 什么时候该用 skill，不该用 tool / agent / MCP

而 MCP 组目前还偏对象层，还没完全压回“真实使用时怎么判断”。

---

## 当前最该优先解决的问题

### 第一优先：把 10 真正立成锚点篇
这是最关键的。

如果 10 不够强，09 和 11 都会漂。

### 第二优先：把 09 砍得更像破误解篇
别让它继续吃 10 的主线。

### 第三优先：把 11 改成真正的边界裁决篇
少一点关系图，多一点判断标准。

---

## 一句话总结

> 卷五 MCP 组当前最大的问题，不是观点不对，而是三篇都在抬高 MCP 的对象级别，却还没彻底拉开“破误解 / 讲主链 / 收边界”的职责差。
