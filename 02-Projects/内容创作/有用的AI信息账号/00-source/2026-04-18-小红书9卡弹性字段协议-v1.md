# 小红书9卡弹性字段协议 v1

## 这份文档的作用

上一份《9卡内容协议与 HTML 图卡模板系统》已经把骨架定下来了。

这一份继续往前走一步：

> **把“9 张卡该怎么填”写成一套可渲染、可检查、但不僵硬的字段协议。**

这里不追求把所有内容压成同一格式，
而是要实现三件事：

1. agent 知道每张卡至少该有什么
2. 模板系统知道哪些字段该接、哪些可以空
3. 写作者不会被“死表单”绑住手脚

---

## 总原则

### 1. 固定的是任务，不是句子

每张卡固定的是它要解决的问题，
不是固定句式，不是固定 bullet 数，不是固定字数到个位数。

### 2. 字段分四层

每张卡都按四层来定义：

- **必填字段**：没有这些，这张卡不成立
- **可选字段**：按题型和信息密度决定是否出现
- **禁用字段**：避免越位和重复
- **推荐区间**：给健康范围，不是死命令

### 3. 模板要接受“有些字段为空”

HTML 模板不应该假设每张卡所有字段都同时存在。

比如：
- 有的卡只有 `summary + 2 bullets`
- 有的卡只有 `quote + interpretation`
- 有的卡只有 `thesis + closing`

所以模板系统必须支持条件渲染。

### 4. 字段协议服务两套 schema

同一套卡位会服务两类内容：

- `single-change`：单条变化卡
- `trend-line`：赛道线索卡

字段名尽量复用，差别主要体现在：

- 内容侧重点
- 某些字段是否更常出现
- 示例写法不同

# 一、通用字段字典

为了后面模板渲染统一，先把通用字段含义定一下。

## 基础元字段

- `slot`：卡位编号，1~9
- `template`：模板类型，如 `cover` / `info` / `quote` / `case` / `closing`
- `type`：内容题型，如 `single-change` / `trend-line`
- `topic`：整组图卡的主题
- `issue`：期数 / 编号（可选）

## 通用内容字段

- `title`：卡内主标题
- `subtitle`：副标题
- `eyebrow`：小眉题 / 分类标签
- `summary`：这张卡最核心的一句说明
- `bullets[]`：要点列表
- `footnote`：底部补充说明
- `tags[]`：标签

## 证据类字段

- `source_name`：来源名称
- `source_type`：来源类型，如 official / repo / post / media
- `quote`：原话 / 原始摘录
- `interpretation`：解释这段原话为什么重要
- `source_ref`：日期 / 链接短标识 / 平台标识

## 对照 / 判断类字段

- `before`：过去是什么
- `after`：现在是什么
- `thesis`：我们的核心判断
- `next_watch[]`：后续观察点
- `closing`：结尾收束句

## 案例 / 方向类字段

- `case_name`：案例名
- `trend_name`：方向名
- `points[]`：案例点或趋势点
- `footer`：底部说明

# 二、每卡字段协议

## 卡 1｜封面 / 钩子卡

**信息任务**
- 让人知道这组图在讲什么
- 让人愿意继续滑

**推荐模板**
- `cover`

**必填字段**
- `slot`
- `template`
- `title`

**可选字段**
- `section`
- `subtitle`
- `issue`
- `tags[]`
- `eyebrow`

**禁用字段**
- 长段正文
- 4 条以上要点列表
- 大段证据摘录

**推荐区间**
- `title`：12~22 字
- `subtitle`：14~28 字
- `tags[]`：0~2 个

**适用提醒**
- `single-change`：标题里尽量点出具体变化
- `trend-line`：标题里尽量点出迁移方向

**示例**
- `title`: Codex 现在能操作电脑了
- `subtitle`: AI agent 正在从写代码走向直接执行

---

## 卡 2｜发生了什么

**信息任务**
- 讲清楚事件本身
- 把“谁做了什么”交代完整

**推荐模板**
- `info`

**必填字段**
- `slot`
- `template`
- `title`
- `summary`

**可选字段**
- `eyebrow`
- `bullets[]`
- `footnote`
- `source_ref`

**禁用字段**
- 长篇趋势预测
- 抢先给很重的观点判断
- 未经交代的抽象结论

**推荐区间**
- `summary`：24~40 字
- `bullets[]`：0~4 条
- bullet 单条：16~32 字

**适用提醒**
- `single-change`：重点是发布 / 升级 / 新入口 / 新能力
- `trend-line`：重点是最近集中冒头的 2~4 个信号

**示例**
- `title`: 这次不是普通更新
- `summary`: OpenAI 正在把 Codex 从代码助手往可执行 agent 推进。
- `bullets[]`: ["新增 computer use 能力", "开始能直接操作界面", "入口和权限边界更明确"]

---

## 卡 3｜变了什么

**信息任务**
- 讲清楚这次变化和以前相比到底新在哪

**推荐模板**
- `info`

**必填字段**
- `slot`
- `template`
- `title`
- `summary`

**可选字段**
- `before`
- `after`
- `bullets[]`
- `footnote`

**禁用字段**
- 重复卡 2 的事实清单
- “更强了 / 更智能了”这类空词
- 和本卡无关的引用堆砌

**推荐区间**
- `summary`：24~40 字
- `before` / `after`：各 10~24 字
- `bullets[]`：0~3 条

**适用提醒**
- `single-change`：优先写 before / after
- `trend-line`：优先写旧范式 / 新范式

**示例**
- `before`: 以前主要是写代码
- `after`: 现在开始直接操作电脑
- `summary`: 这次关键不是模型更会说，而是 agent 开始获得执行界面动作的能力。

---

## 卡 4｜原声 / 证据卡

**信息任务**
- 提供可信度
- 让判断有出处，不像凭空分析

**推荐模板**
- `quote`

**必填字段**
- `slot`
- `template`
- `source_name`
- `quote`

**可选字段**
- `source_type`
- `interpretation`
- `source_ref`
- `title`

**禁用字段**
- 没有来源的“有人说”
- 只贴原话，不解释重要性
- 二手转述伪装成一手证据

**推荐区间**
- `quote`：40~90 字
- `interpretation`：0~40 字

**适用提醒**
- 两类 schema 都至少保留 1 张证据卡
- 没有硬证据时，不要勉强做 quote 卡，先回去补源

**示例**
- `source_name`: OpenAI / release note
- `quote`: Codex can now interact with the computer environment...
- `interpretation`: 真正该看的不是演示噱头，而是能力边界从“生成建议”进入“执行动作”。

---

## 卡 5｜为什么值得看

**信息任务**
- 讲意义
- 解释这件事为什么不只是一个普通更新

**推荐模板**
- `info`

**必填字段**
- `slot`
- `template`
- `title`
- `summary`

**可选字段**
- `bullets[]`
- `thesis`
- `footnote`

**禁用字段**
- 空喊“很重要”
- 没有指向对象的行业黑话
- 和卡 8 重复成纯判断卡

**推荐区间**
- `summary`：24~40 字
- `bullets[]`：0~3 条

**适用提醒**
- `single-change`：讲这次变化对产品 / workflow 的意义
- `trend-line`：讲为什么这条线不是一次性杂音

**示例**
- `summary`: 值得看，不是因为功能名新，而是因为 agent 的工作层开始从命令行往图形界面扩。 

---

## 卡 6｜案例 / 场景卡

**信息任务**
- 把抽象判断拉回真实使用

**推荐模板**
- `case`

**必填字段**
- `slot`
- `template`
- `title`
- `summary`

**可选字段**
- `case_name`
- `points[]`
- `footer`
- `source_ref`

**禁用字段**
- 没有场景的抽象判断
- 生造一个并不存在的案例
- 用大段背景代替场景

**推荐区间**
- `summary`：20~36 字
- `points[]`：0~3 条

**适用提醒**
- `single-change`：优先给一个具体使用链路
- `trend-line`：优先给一个代表项目 / 产品 / workflow

**示例**
- `case_name`: claude-mem
- `summary`: 它说明 memory 已经不只是存聊天记录，而是在做可回注的上下文层。

---

## 卡 7｜往哪边长

**信息任务**
- 讲未来几条最可能的延长方向
- 帮读者知道后续该盯什么

**推荐模板**
- `case`
- `info`

**必填字段**
- `slot`
- `template`
- `title`
- `summary`

**可选字段**
- `trend_name`
- `points[]`
- `bullets[]`
- `footer`

**禁用字段**
- 凭空押宝式预测
- 一口气列 5 条以上方向
- 和卡 8 混成观点宣言

**推荐区间**
- `summary`：20~36 字
- `points[]` / `bullets[]`：0~3 条

**适用提醒**
- 常见方向数控制在 2~3 个
- 如果只看得到 1 个明确方向，就只写 1 个，不要硬凑 3 个

**示例**
- `points[]`: ["产品化", "agent-friendly", "工作流化"]

---

## 卡 8｜我们的判断

**信息任务**
- 给出账号立场
- 明确短期最值得盯的一层

**推荐模板**
- `closing`
- `info`

**必填字段**
- `slot`
- `template`
- `title`
- `thesis`

**可选字段**
- `bullets[]`
- `footnote`
- `next_watch[]`

**禁用字段**
- 鸡汤化抒情
- 伪装成事实的价值判断
- 只有态度，没有依据

**推荐区间**
- `thesis`：20~40 字
- `bullets[]`：0~2 条
- `next_watch[]`：0~3 条

**适用提醒**
- 这是判断卡，不是总结卡
- 最好出现“我们真正觉得该盯的是……”这种明确立场

**示例**
- `thesis`: 这条线短期最该盯的不是谁先喊得响，而是谁先把记忆真正接进未来 session。

---

## 卡 9｜收口 / 收藏理由 / 下一步

**信息任务**
- 收束整组图卡
- 给用户一个保存理由

**推荐模板**
- `closing`

**必填字段**
- `slot`
- `template`
- `title`
- `closing`

**可选字段**
- `next_watch[]`
- `bullets[]`
- `footnote`

**禁用字段**
- “关注我看更多”式机械收尾
- 没有信息量的口号结尾
- 和卡 8 完全重复

**推荐区间**
- `closing`：18~36 字
- `next_watch[]`：0~3 条

**适用提醒**
- 这张卡更像“收藏理由 + 后续观察点”
- 下篇预告是可选，不是必须

**示例**
- `closing`: 如果你只记一件事，记住：现在真正变的不是工具数量，而是 agent 开始获得持续工作的结构。

# 三、两种题型怎么用同一套字段

## A. single-change

更常出现的字段：
- `before`
- `after`
- `source_name`
- `quote`
- `case_name`

优先顺序：
1. 事实讲清
2. 新旧对照讲清
3. 证据给够
4. 场景落地

## B. trend-line

更常出现的字段：
- `thesis`
- `trend_name`
- `points[]`
- `next_watch[]`
- `case_name`

优先顺序：
1. 线索汇总
2. 旧范式 / 新范式讲清
3. 代表案例撑住
4. 方向拆开

```json
    }

```

# 五、渲染层需要遵守的事

HTML 模板实现时，至少要支持下面这些逻辑：

1. **字段缺省时自动隐藏区块**
   - 没有 `subtitle` 就不占位
   - 没有 `bullets[]` 就不渲染列表

2. **同模板允许轻微版式变体**
   - `info` 可以有标准 bullet 版 / 对照版
   - `case` 可以有案例版 / 趋势版

3. **不要把“字段多”误认为“内容更好”**
   - 字段够用就停
   - 模板应该鼓励克制，而不是鼓励塞满

# 六、发布前快检

在产图前，快速过一遍：

- 卡 2 有没有讲清楚发生了什么
- 卡 3 有没有对照变化
- 卡 4 有没有真正来源
- 卡 6 有没有真实案例 / 场景
- 卡 8 有没有明确判断
- 卡 9 有没有保存价值
- 有没有任何一张卡只是漂亮话

如果有两张以上答不上来，就说明内容还没到能发的程度。

## 一句话收束

这份协议的目标不是把内容做成表单，
而是：

> **把“稳定生产”需要的骨架固定下来，同时给每一张卡保留足够的表达弹性。**
