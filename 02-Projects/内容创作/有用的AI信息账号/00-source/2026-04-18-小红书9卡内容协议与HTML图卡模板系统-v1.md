# 小红书9卡内容协议与HTML图卡模板系统 v1

## 这份文档要解决什么问题

现在的问题已经不是“有没有题可写”，而是：

- 卡片内容还不够稳定
- 每张卡承担什么角色还没固定
- 内容到视觉还没有一条可复用的生产链

所以这份文档要把三件事一次定下来：

1. **9 张卡各自负责什么信息任务**
2. **每张卡允许放什么字段、多少字、什么不能写**
3. **内容如何填进 HTML 模板，再稳定导出成 9 张图**

这套系统的目标不是“设计得很炫”，而是：

> **让每一组图卡都看起来像同一个产品产出的、有信息密度的内容，而不是临时拼出来的海报。**

---

## 核心原则

### 1. 卡片不是口号图，而是信息图

每一张卡都必须完成一个明确的信息动作。

允许的主动作只有这些：

- 讲事实
- 讲变化
- 给证据
- 讲意义
- 给案例
- 做判断
- 做收束

不允许一张卡同时承担太多任务。

### 2. 内容先于样式

先固定内容协议，再固定 HTML 模板。

如果内容协议不定，后面任何模板都会被每次新题反复打破。

### 3. 固定卡位，不死做死样式

对外呈现是 **9 张固定卡位**。

对内生产不是 9 个完全不同的模板文件，而是：

> **9 个固定卡位 + 5 种底层卡型组件**

这样既稳定，又不会维护成本过高。

### 4. 每组卡至少有 1 张证据卡

这是和“空喊观点号”拉开差距的关键。

没有证据卡，整组图卡就容易只剩判断；
有证据卡，整个账号会更像有来源的情报产品。

### 5. 默认服务“滑动叙事”

小红书图卡不是把长文拆 9 段，
而是让读者在滑动过程中逐步完成：

- 被钩住
- 看懂发生了什么
- 明白变化点
- 相信这不是空话
- 感到值得收藏

---

# 一、内容协议：先固定 9 个卡位

下面先定义 **通用 9 卡骨架**。

后面两种题型：

1. **单条变化卡**
2. **赛道线索卡**

都基于这个骨架，只在字段权重和表达方式上做微调。

---

## 卡 1｜封面 / 钩子卡

**任务：** 让人知道这组图在讲什么，并愿意继续往下滑。

**必须回答：**
- 这篇在讲哪件事 / 哪条线
- 它为什么值得看

**推荐字段：**
- 栏目名
- 主标题
- 副标题
- 时间 / 期数
- 标签 1~2 个

**字数建议：**
- 主标题：12~22 字
- 副标题：14~28 字
- 标签：2 个以内

**禁止：**
- 只有情绪，没有事实
- 标题大而空，例如“AI 又变天了”

**合格标准：**
读者即使只看第一页，也知道这篇不是泛泛而谈。

---

## 卡 2｜发生了什么

**任务：** 把事情本身讲清楚。

**必须回答：**
- 谁做了什么
- 新东西是什么
- 它属于发布 / 升级 / 开源 / 演示 / 趋势汇流中的哪一种

**推荐字段：**
- 小标题
- 1 句事实结论
- 2~4 条事实 bullet
- 来源标签（可选）

**字数建议：**
- 结论句：24~40 字
- bullet：每条 16~32 字
- bullet 数量：2~4 条

**禁止：**
- 直接跳到观点，没把事情讲清楚
- 把不同层的东西混成一段

---

## 卡 3｜变了什么

**任务：** 讲清楚“这次为什么不是旧闻重讲”。

**必须回答：**
- 相比以前，到底变了哪一层
- 这次的新意是什么

**推荐字段：**
- 对照标题
- 过去 vs 现在
- 变化点 2~3 条

**适合写法：**
- 以前是 X，现在是 Y
- 以前只能 A，现在开始能 B

**禁止：**
- 只重复卡 2 的事实
- 用“更强了”“更智能了”这种空描述代替变化点

---

## 卡 4｜原声 / 证据卡

**任务：** 提供可信度。

**必须回答：**
- 这个判断是基于什么原话 / 原始描述 / 官方表述

**推荐字段：**
- 来源名
- 原话摘录
- 中文解释 / 拆译
- 链接短标识 / 日期

**字数建议：**
- 原话：40~90 字
- 中文解释：20~40 字

**禁止：**
- 只写“很多人都在说”
- 引一大段却不解释为什么重要

**说明：**
每组图卡至少 1 张；默认优先固定在第 4 张。

---

## 卡 5｜为什么值得看

**任务：** 讲意义，不讲虚词。

**必须回答：**
- 这件事对产品、工作流或用户行为意味着什么

**推荐字段：**
- 1 句核心判断
- 2~3 条意义拆解
- 可加“这不是因为 X，而是因为 Y”式对照

**禁止：**
- “很重要”“值得关注”但不说为什么
- 用行业大词糊过去

---

## 卡 6｜案例 / 场景卡

**任务：** 把抽象判断落回具体场景。

**必须回答：**
- 这件事在真实使用中会怎么体现
- 哪个产品 / 项目 / 使用链路能说明它

**推荐字段：**
- 案例名
- 场景描述
- 案例说明了什么
- 1~3 个关键点

**禁止：**
- 继续讲抽象判断，不给例子
- 用假大空场景代替真实案例

---

## 卡 7｜往哪边长

**任务：** 讲趋势方向，而不是做预测表演。

**必须回答：**
- 这条变化最可能继续往哪几条方向长

**推荐字段：**
- 方向标题
- 2~3 条趋势分叉
- 每条趋势后补 1 句解释

**禁止：**
- 凭空预测
- 一口气列太多方向，导致信息散

---

## 卡 8｜我们的判断

**任务：** 给出账号立场。

**必须回答：**
- 我们真正觉得值得盯的是什么
- 这条线当前最值得看的一层是什么
- 还有什么没解决

**推荐字段：**
- 1 句判断
- 2 条判断依据
- 1 条保留意见 / 未解决点（可选）

**禁止：**
- 鸡汤化收尾
- 把“观点态度”写成“事实”

---

## 卡 9｜收口 / 收藏理由 / 下一步

**任务：** 收束整组内容，并给出保存价值。

**必须回答：**
- 如果用户只记一件事，应该记什么
- 之后这条线可以继续怎么看

**推荐字段：**
- 1 句收口
- 2~3 个后续观察点 / 收藏理由
- 下篇预告（可选）

**禁止：**
- 空洞 CTA
- “关注我看更多”式机械结尾

---

# 二、两种题型的 schema

虽然都可以用 9 张骨架，但 **单条变化卡** 和 **赛道线索卡** 的字段重点不同。

---

## A. 单条变化卡 schema

### 适用题型
- 某次 release / 新功能为什么值得看
- 单个产品变化拆解
- 平台信号卡
- “这件事为什么重要”类卡

### 内容重心
- 事实层要清楚
- 对照变化要清楚
- 证据要硬
- 案例要具体

### 9 卡映射建议
1. 封面：这件事是什么，为什么值得看
2. 发生了什么：谁发了什么
3. 变了什么：对照旧版本 / 旧能力
4. 证据：原帖 / release / 官方说明
5. 值得看：这次变化真正重要在哪
6. 案例：真实使用场景
7. 往哪边长：下一阶段能力方向
8. 我们判断：现在最该盯什么
9. 收口：值得收藏的结论

### 特别要求
- 必须明确“以前 vs 现在”
- 必须至少出现 1 个真实产品说明或原话证据
- 不允许整篇都停留在“趋势感”

---

## B. 赛道线索卡 schema

### 适用题型
- 最近最该盯的一条线
- 某赛道进入第二阶段了吗
- 一类产品为什么突然开始密集出现
- 一种工作流为什么在升温

### 内容重心
- 不是单一事件，而是多信号汇总
- 要讲清楚“这条线从哪走到哪”
- 重点是结构判断，不是事件复述

### 9 卡映射建议
1. 封面：这条线正在发生什么迁移
2. 线索概览：最近有哪些信号集中出现
3. 核心变化：从旧形态往新形态迁移
4. 证据：代表原话 / repo / 官方表述
5. 为什么值得看：这条线为什么不是杂音
6. 代表案例：1 个项目 / 产品 / workflow
7. 拆成 2~3 个方向：产品化 / agent-friendly / 工作流化
8. 我们判断：短期最该盯哪一层
9. 收口：后续观察点

### 特别要求
- 至少要有 2 个以上独立信号支撑
- 必须讲清楚“旧范式 vs 新范式”
- 不能写成一句模糊大观点撑满 9 张

---

# 三、统一字数与信息密度约束

这是为了防止模板一会儿太空，一会儿爆字。

## 每卡硬约束（建议先严格执行）

- 标题：12~22 字
- 小标题：4~10 字
- 主结论：24~40 字
- bullet：每条 16~32 字
- 每卡 bullet 数：2~4 条
- 每卡总字数：**80~140 字** 为宜
- 证据卡：可放宽到 150 字，但不建议再长

## 版式约束

- 一张卡只允许 1 个主结论
- 一张卡最多 1 层 bullet
- 不做深层嵌套
- 不用大段连续正文
- 留白优先于塞满

## 内容质量约束

每张卡至少满足下面之一：
- 给了一个新事实
- 给了一个清楚对照
- 给了一个原话证据
- 给了一个真实案例
- 给了一个非空判断

如果一张卡不满足以上任一项，说明它大概率是空卡。

---

# 四、底层 HTML 模板：先做 5 种卡型组件

外面看起来是 9 张固定卡，底层只维护 5 种模板。

---

## 模板 A｜封面卡

**适用卡位：** 卡 1

**字段：**
- `section`
- `title`
- `subtitle`
- `issue`
- `tags[]`

**视觉重点：**
- 主标题最大
- 副标题控制 2 行内
- 标签轻量
- 封面要显得“像一个栏目”，不是像海报口号

---

## 模板 B｜信息说明卡

**适用卡位：** 卡 2、3、5

**字段：**
- `eyebrow`
- `title`
- `summary`
- `bullets[]`
- `footnote`（可选）

**视觉重点：**
- 最通用
- 标题 + 主结论 + bullet 三层清楚
- 适合讲事实、变化、意义

---

## 模板 C｜证据卡 / 原声卡

**适用卡位：** 卡 4

**字段：**
- `source_name`
- `source_type`
- `quote`
- `interpretation`
- `source_ref`

**视觉重点：**
- 要明显区别于普通说明卡
- 引文区和解释区要分层
- 最好保留来源感，而不是完全抹平成普通正文

---

## 模板 D｜案例 / 趋势分叉卡

**适用卡位：** 卡 6、7

**字段：**
- `title`
- `case_name` / `trend_name`
- `summary`
- `points[]`
- `footer`（可选）

**视觉重点：**
- 适合“一例一说明”或“一方向一解释”
- 可以做左右分栏或上下一主二辅

---

## 模板 E｜判断 / 收口卡

**适用卡位：** 卡 8、9

**字段：**
- `title`
- `thesis`
- `bullets[]`
- `next_watch[]`（可选）
- `closing`

**视觉重点：**
- 这类卡要稳，不要花
- 收口比煽动更重要

---

# 五、推荐的数据结构：内容与模板分离

不要直接“写 9 张图”。

应该先写一份结构化内容，再让模板去渲染。

## 最小 JSON schema（示意）

```json
{
  "meta": {
    "type": "single-change",
    "topic": "Codex 现在能操作电脑了",
    "date": "2026-04-18",
    "issue": "首发02"
  },
  "theme": {
    "section": "有用的AI信息账号",
    "style": "clean-news"
  },
  "cards": [
    {
      "slot": 1,
      "template": "cover",
      "title": "Codex 现在能操作电脑了",
      "subtitle": "AI agent 正在从写代码走向直接执行",
      "tags": ["AI Agent", "Computer Use"]
    },
    {
      "slot": 2,
      "template": "info",
      "eyebrow": "发生了什么",
      "title": "这次不是普通更新",
      "summary": "OpenAI 正在把 Codex 从代码助手往可执行 agent 推进。",
      "bullets": [
        "新增 computer use 能力",
        "开始能直接操作界面",
        "入口和权限边界更明确"
      ]
    }
  ]
}
```

## 为什么要这样做

因为这样后面：

- 改内容，不用改模板
- 改模板，不用改内容
- 能做批量导图
- 能做版本控制
- 能做自动校验

---

# 六、渲染链路建议：HTML → 截图 → 9 张 PNG

## 推荐最小链路

1. 写内容 JSON / YAML
2. 将数据填入 HTML 模板
3. 用浏览器无头渲染
4. 输出 9 张 PNG
5. 统一检查边距、字号、断行、溢出

## 推荐技术

- HTML + CSS：负责模板
- JSON / YAML：负责数据
- Playwright / Puppeteer：负责渲染和截图

## 推荐目录结构

```text
xhs-card-system/
  templates/
    cover.html
    info.html
    quote.html
    case.html
    closing.html
  styles/
    base.css
    theme-clean-news.css
  content/
    2026-04-18-codex-computer-use.json
    2026-04-18-ai-knowledge-base-product-shift.json
  output/
    2026-04-18-codex-computer-use/
      01.png
      02.png
      ...
      09.png
```

## 导图规则

- 画布比例固定：优先 3:4 或小红书常用竖图比例
- 每张卡固定安全边距
- 默认导出 `01.png` 到 `09.png`
- 输出后自动检查溢出和空卡

---

# 七、发布前校验清单

每组图卡在导出前，至少过这 8 条：

1. **封面不是空喊话**：第一页本身有信息
2. **卡 2 讲清事实**：谁做了什么
3. **卡 3 有对照**：以前 vs 现在
4. **至少 1 张证据卡**：最好固定卡 4
5. **至少 1 张案例卡**：避免全篇抽象
6. **判断卡有明确立场**：不是模糊支持
7. **收口卡能被收藏**：有观察点或记忆点
8. **任一张卡都不只是漂亮话**

如果其中 2 条以上过不了，就说明这组卡还没到能发的程度。

# 八、从现在开始的最小可运行版本（MVP）

## 第一步：先只支持 2 套内容 schema

1. `single-change`
2. `trend-line`

## 第二步：只实现 5 个模板

- cover
- info
- quote
- case
- closing

## 第三步：先拿现有两篇 v3 做样板

优先验证：
- [[2026-04-18-首发02-Codex现在能操作电脑了-资讯卡稿-v3]]
- [[2026-04-18-首发03-AI知识库正在从笔记流派变成产品流派-资讯卡稿-v3]]

这两篇刚好对应：
- 单条变化卡
- 赛道线索卡

很适合拿来做模板回填实验。

## 第四步：再根据首轮结果微调

重点只看：
- 哪些卡总是字数爆掉
- 哪些卡容易空
- 证据卡是不是太重
- 卡 7 / 卡 8 会不会重复
- 卡 9 是否真的需要预告

# 九、当前版本的明确决策

## 已拍板

1. **小红书信息卡必须保留信息密度，不做纯喊话图**
2. **固定为 9 张卡位**
3. **先做两套 schema：单条变化卡 / 赛道线索卡**
4. **底层只维护 5 种 HTML 卡型组件**
5. **内容与模板分离，优先用结构化数据填充模板**
6. **每组卡至少 1 张证据卡，至少 1 张案例或场景卡**

## 暂不拍板

1. 最终画布具体像素
2. 最终品牌配色
3. 字体方案
4. 是否加入封面插图 / icon
5. 是否做多套视觉皮肤

这些都属于下一层，不影响先把系统跑起来。

# 十、下一步建议

## 先做内容，不先卷设计

最适合的执行顺序是：

1. 先把这份 9 卡协议定为 v1
2. 选 Codex / 知识库两篇做 schema 回填
3. 再据此做第一版 HTML 模板
4. 导出首批 9 图
5. 用实物再改协议和样式

## 当前最值得继续推进的产物

接下来不是继续空谈“风格”，而是应该直接补两份可执行文档：

1. **《小红书 9 卡字段表 v1》**
   - 每张卡的字段名
   - 必填 / 可选
