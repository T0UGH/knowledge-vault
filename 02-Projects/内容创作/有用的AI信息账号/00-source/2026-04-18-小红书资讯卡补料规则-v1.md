# 小红书资讯卡补料规则 v1

## 为什么要有这份规则

这次拿 `Codex computer use` 做实验之后，结论已经比较清楚：

> **日报很适合找题，但不总适合直接做厚单篇。**

日报的长处是：

- 发现最近发生了什么
- 给出大方向
- 帮你快速判断哪条线值得继续跟

但日报很多时候更像：

> **高质量信号入口**

而不是：

> **可以直接压成高信息密度小红书单篇卡的成品材料**

所以后面的流程不能再默认：

- 日报定题
- 直接改卡

更合理的是：

- 日报定题
- 判断题型
- 需要时补 1~2 个强源
- 再写卡

## 总原则

### 1. 日报负责“发现题”，补料负责“把题做厚”

不要一上来就重新大搜一圈。

先问：

- 日报里这个 topic 已经够不够写？
- 是不是只缺一点事实层？
- 还是本来就是轻卡，不需要厚写？

### 2. 不是每篇都要补料

补料不是默认动作，而是题型动作。

如果每篇都补，成本太高，节奏会崩。

### 3. 补料优先补“强源”，不是补“很多源”

一个好的一手源 + 一个高密度强二手源，通常比乱补十条碎信息更值钱。

### 4. 补料的目标不是做研究报告

目标是补到：

- 不再偏薄
- 有事实层支撑
- 用户读完觉得“这条消息被讲厚了”

不是补到无穷尽。

---

## 题型分级：哪些卡该补，哪些不该补

## A 类：日报可直出，最多轻补

这些卡本来就偏轻，不要求做得特别厚。

### 适用题型
- 本周最值得看的 3 / 5 个变化
- 过去 7 天最值得普通人理解的几个变化
- GitHub 趋势里最值得跟的 3 / 4 个项目
- Product Hunt 本周最像 Agent 工作流拼图的几类产品

### 原因
这些卡的价值主要在：

- 筛选
- 排序
- 归类
- 给继续跟的方向

它们不是拿来打厚单条消息的。

### 做法
- 用日报直接出
- 如果有明显缺口，只补 1 个最关键的事实
- 不要为这种卡重做深度研究

---

## B 类：建议补料再出

这些卡如果不补，最容易出现的问题是：

- 有判断
- 但事实层太薄
- 看起来像“凭感觉在说”

### 适用题型
- 单条变化拆解卡
- 平台信号卡
- 某次 release / 产品更新为什么值得看
- “这件事为什么重要”类卡

### 典型例子
- Codex 现在能操作电脑了
- Claude Design 出来后真正该看什么
- Claude Code 最近几版更新的共同主线
- Anthropic 最近连续放出的不只是 feature，而是一整套平台信号

### 做法
- 日报先定题
- 再补 1~2 个强源
- 再写卡

这是后面最应该稳定执行的动作。

---

## C 类：建议做横向补料

这类卡不是缺某一条消息，而是缺“这条线真的在升温”的证据密度。

### 适用题型
- 最近最该盯的一条线
- 赛道阶段判断
- 某条线是不是进入第二阶段了

### 典型例子
- AI 知识库正在从笔记流派变成产品流派
- 多 Agent 内容开始从协作转向治理
- browser / computer-use 基建正在成型
- 单点能力开始被包装成独立产品

### 做法
- 不一定补单一外部源
- 更常见的是横向翻 3~7 天日报
- 再从里面抽重复出现的信号
- 必要时补 1 个官方源或 1 个社区强源

---

## 补料时优先补什么源

## 第一优先：官方 / 一手源

最有价值的通常是：

- 官方发布页
- 官方 release / changelog
- 官方产品说明
- 官方原帖

### 适合补什么
- rollout 范围
- 入口
- 功能边界
- 新增组件
- 平台能力变化

### 好处
- 事实最稳
- 不容易传错
- 适合做卡里的“硬信息层”

---

## 第二优先：高密度强二手源

不是所有二手都值钱。

优先选：

- 真正在拆功能层的人
- 长帖里有结构化增量信息的人
- 比日报摘要更厚、但还没失真的强二手

### 适合补什么
- 使用场景
- 对照理解
- 隐含方向
- 社区视角下“这到底意味着什么”

### 这次 Codex 实验里最典型的就是
- 一条高密度长帖补功能层 / 场景层
- 一条短帖补入口 / 设置层

这就是很好的组合。

---

## 第三优先：横向社区反应

适合补：

- 同一 topic 在不同账号那里的不同说法
- 热度和争议点
- 社区到底在看哪一层

### 适合题型
- 热点变化
- 争议型话题
- MCP / Skills / Agent 这种讨论带强的题

### 注意
不要为了“搜更多”而搜更多。
目的是确认：

- 这条到底是热闹
- 还是已经形成稳定讨论

---

## 不同题型，补到什么程度就够了

## 单条变化拆解卡：至少补 4 层里 2~3 层

最好能补出：

1. **功能层**：到底多了什么
2. **场景层**：拿来干什么
3. **入口层**：谁能用、怎么开
4. **边界层**：先上哪、哪些地方还没开

如果能补出其中 2~3 层，这张卡通常就不会太薄。

---

## 赛道线索卡：至少补 3 组重复信号

这类不是靠一条消息成立，
而是要看：

- 这几天是不是反复出现
- 不同来源是不是都在往同一个方向指
- 是不是已经从单条消息长成主线

所以对赛道卡来说，
更重要的不是单条细节，
而是：

> **重复出现的证据。**

---

## 精选卡：补到每条都不空就够

精选卡不需要打太厚。

原则是：

- 每条都能三句讲清 what / change / why
- 读者能知道为什么它排在这张卡里

就够了。

---

## 一条卡什么时候说明“还没补够”

如果写出来后，你会明显感觉：

- 只有判断，没有料
- 换个 topic 也能套同样的话
- 读完像“方向是对的，但没有肉”
- 只能反复说“边界变了 / 值得看 / 值得继续盯”

那基本就说明：

> 该补料了。

---

## 一条卡什么时候说明“已经补够了”

如果写出来后，读者能看见：

- 这次到底新增了什么
- 这东西现在能在哪用
- 为什么它和以前不一样
- 后面该继续盯哪一层

那通常就够了。

不需要为了“更全面”继续拖。

---

## 实战建议：默认补料流程

### 流程 1：日报找题
先从日报里找到：
- topic
- 主判断
- 这题属于哪一类卡

### 流程 2：判断题型
- 轻卡？直接写
- 重卡？进入补料

### 流程 3：补 1~2 个强源
优先：
1. 官方 / 一手源
2. 高密度强二手
3. 横向社区反应（必要时）

### 流程 4：再写卡
写卡时把补出来的信息分到：
- 功能层
- 场景层
- 入口层
- 边界层
- 赛道层

不一定每层都写，但你至少知道自己补到了哪。

---

## 这套规则最适合解决什么问题

它最适合解决的，就是现在你已经明确指出的这个问题：

> **方向对了，但信息量不够足。**

用这套规则之后，后面我们就不会再凭感觉决定“要不要搜”。

而是按题型决定：

- 哪些题直接出
- 哪些题必须补料
- 补几层就够
- 补几个源最划算

这会比现在稳定很多。
