# HTML图卡模板实现说明 v1

## 这份文档解决什么问题

现在内容层已经有两份上游协议：

- [[2026-04-18-小红书9卡内容协议与HTML图卡模板系统-v1]]
- [[2026-04-18-小红书9卡弹性字段协议-v1]]

缺的不是“再讨论一下样式”，而是把实现层定下来：

- 模板放哪
- 数据怎么进来
- 哪些区块条件渲染
- 如何截图导出
- 如何避免爆字、空卡、断行崩坏

这份文档就是把这些落成一条最小可运行实现链。

---

## 一、实现目标

这套 HTML 系统先只追求 4 件事：

1. **能稳定接 9 卡 JSON 数据**
2. **能复用 5 种模板产图**
3. **字段为空时不崩布局**
4. **能批量导出 01~09.png**

第一阶段不追求：

- 多主题皮肤
- 花哨动画
- 在线编辑器
- 响应式适配所有终端
- 复杂插图系统

先把最小生产链跑通。

---

## 二、推荐目录结构

建议在后续真正实现代码时，单独建一个目录，例如：

```text
xhs-card-system/
  package.json
  README.md
  content/
    2026-04-18-codex-computer-use.json
    2026-04-18-ai-knowledge-base-shift.json
  templates/
    cover.html
    info.html
    quote.html
    case.html
    closing.html
    card-shell.html
  styles/
    base.css
    tokens.css
    theme-clean-news.css
    components.css
    utilities.css
  scripts/
    render.js
    validate.js
  output/
    2026-04-18-codex-computer-use/
      01.png
      02.png
      ...
      09.png
```

### 目录职责

#### `content/`
存结构化输入，不存样式。

#### `templates/`
存 5 种模板和 1 个通用外壳。

#### `styles/`
把基础 token、公共组件、主题样式拆开。

#### `scripts/`
- `validate.js`：校验输入数据
- `render.js`：渲染 HTML 并截图

#### `output/`
按主题或日期输出图片结果。

## 三、模板拆分方式

## 1. 不要做 9 个独立 HTML

正确拆法不是：

- card-1.html
- card-2.html
- ...
- card-9.html

而是：

- `card-shell.html`：统一外壳
- 5 个模板 partial：cover / info / quote / case / closing

也就是：

> **卡位是固定的，模板是复用的。**

## 2. `card-shell.html` 负责什么

它只负责所有卡通用的壳：

- 统一画布尺寸
- 外边距 / 安全区
- 顶部栏目位
- 底部页码 / 主题 / 日期位
- 主内容插槽 `{{body}}`

`card-shell` 不负责业务字段。

## 3. 五种模板 partial 各自负责什么

### `cover.html`
负责：
- 封面标题
- 副标题
- 标签
- 期数

### `info.html`
负责：
- 标题
- 主结论
- bullet 列表
- 补充说明

### `quote.html`
负责：
- 来源
- 引文
- 解释
- 来源短标识

### `case.html`
负责：
- 案例名 / 趋势名
- 一句概括
- 1~3 个点
- 底部说明

### `closing.html`
负责：
- 判断句
- 收口句
- 后续观察点

## 四、CSS 层级建议

不要把所有样式揉成一个大文件。

推荐拆成下面 5 层：

## 1. `tokens.css`
存设计 token：

- 颜色变量
- 字号变量
- 行高变量
- 间距变量
- 圆角
- 阴影

例如：

```css
:root {
  --bg: #f7f3ee;
  --card: #fffdf9;
  --text: #1f1f1f;
  --muted: #6b6b6b;
  --accent: #1f6fff;
  --border: #e8e2d9;

  --font-title: 44px;
  --font-h2: 28px;
  --font-body: 18px;
  --font-small: 14px;

  --space-1: 8px;
  --space-2: 12px;
  --space-3: 16px;
  --space-4: 24px;
  --space-5: 32px;
}
```

## 2. `base.css`
负责 reset 和基础排版：

- box-sizing
- body 默认字体
- 标题 / 段落默认间距
- 列表样式重置

## 3. `components.css`
负责共用组件：

- `.eyebrow`
- `.title`
- `.summary`
- `.quote-box`
- `.bullet-list`
- `.tag`
- `.footer-meta`

## 4. `theme-clean-news.css`
负责当前主题皮肤：

- 背景色
- 强调色
- 边框风格
- 不同模板在该主题下的细调

## 5. `utilities.css`
只放少量原子辅助类：

- `.muted`
- `.clamp-2`
- `.mt-sm`
- `.hidden`

不要把业务逻辑写进 utility class 里。

## 五、画布与版式规范

## 1. 画布尺寸

第一版建议先固定一个尺寸，不要多尺寸并行。

推荐：

- **1200 x 1600**（3:4）

原因：
- 足够清晰
- 常见小红书图卡比例友好
- HTML 截图容易稳定控制

## 2. 安全区

建议统一：

- 外边距：64px
- 顶部安全区：72px
- 底部安全区：56px

任何可见文字不要贴边。

## 3. 排版硬规则

- 主标题最多 2 行
- 副标题最多 2 行
- summary 最多 4 行
- bullet 最多 4 条
- 单条 bullet 尽量 2 行内
- quote 区控制在卡面高度的 40% 内

这些不是内容协议，而是实现层防炸规则。

## 六、数据装载方式

## 1. 输入文件格式

先只支持 JSON，别一开始就同时支持 YAML。

原因：
- 更易校验
- 更利于脚本处理
- 更适合后续模板程序接入

## 2. 顶层结构

```json
{
  "meta": {
    "topic": "Codex 现在能操作电脑了",
    "type": "single-change",
    "issue": "首发02",
    "theme": "clean-news",
    "date": "2026-04-18"
  },
  "cards": []
}
```

## 3. 渲染流程

1. 读 JSON
2. 先跑 `validate.js`
3. 校验通过后逐张 card 渲染
4. 根据 `template` 选择 partial
5. 把 partial 填进 `card-shell`
6. 写成临时 HTML
7. 用 Playwright 打开并截图

---

## 七、条件渲染规则

这部分很关键，不然弹性字段协议会失效。

## 1. 字段为空就不渲染节点

例如：

- 没有 `subtitle` → 不输出 subtitle DOM
- `bullets[]` 为空 → 不渲染列表容器
- 没有 `source_ref` → 不显示来源脚注
- 没有 `next_watch[]` → 不显示“继续看”模块

## 2. 不要输出空占位

错误做法：

- 渲染一个空白块保持高度
- 让卡面看起来像漏内容

正确做法：

- 内容区自动收缩
- 模板保持紧凑

## 3. 允许轻微版式分支

例如 `info` 模板可有两个 mode：

- `info-standard`
- `info-compare`

触发方式可以是：

- 有 `before + after` → 走 compare 布局
- 只有 `summary + bullets` → 走 standard 布局

`case` 模板也一样：

- 有 `case_name` → 走案例版
- 有 `trend_name` → 走趋势版

## 八、校验脚本需要检查什么

`validate.js` 第一版建议至少检查下面这些：

## 1. 基础结构校验

- 是否有 `meta`
- 是否有 `cards`
- `cards` 是否为数组
- `slot` 是否在 1~9 之间
- `template` 是否属于允许值

## 2. 字段协议校验

按模板检查必填字段：

- `cover` 必须有 `title`
- `info` 必须有 `title + summary`
- `quote` 必须有 `source_name + quote`
- `case` 必须有 `title + summary`
- `closing` 必须有 `title` 且 `thesis/closing` 至少一个存在

## 3. 内容长度校验

- 标题过长告警
- bullet 数超限告警
- `quote` 过长告警
- `summary` 过长告警

这里建议：

- **第一版先告警，不直接报错**
- 结构错误再报错退出

## 4. 语义校验（轻量）

- quote 卡没有来源 → 报错
- card 8 没有 thesis → 报错
- card 9 没有 closing → 报错
- 全组没有 quote 卡 → 告警
- 全组没有 case 卡 → 告警

## 九、截图导出流程

## 1. 为什么优先 Playwright

推荐 Playwright 而不是先上 Puppeteer，原因很简单：

- 截图能力稳定
- API 清楚
- 控制 viewport 和等待字体加载更方便

## 2. 建议流程

### Step 1
启动浏览器，固定 viewport：

- 宽：1200
- 高：1600
- deviceScaleFactor：2

### Step 2
打开临时 HTML 页面。

### Step 3
等待：

- 字体加载完成
- DOM 渲染完成
- 图片 / icon（如果有）加载完成

### Step 4
截图输出到：

```text
output/<topic-slug>/01.png
output/<topic-slug>/02.png
...
output/<topic-slug>/09.png
```

### Step 5
执行导出后快检：

- 是否缺页
- 是否有空白页
- 文件命名是否连续

## 十、渲染后快检规则

HTML 能渲出来，不代表图卡能发。

导出后建议再跑一层简单检查：

## 1. 布局检查

- 标题是否超过 2 行
- 是否有明显裁切
- 是否出现孤行 bullet
- 是否有脚注压到底边

## 2. 内容检查

- 卡 4 是否真的像证据卡，不像普通说明卡
- 卡 8 / 卡 9 是否重复
- 有没有一张卡明显太空
- 有没有一张卡信息爆炸

## 3. 输出检查

- 总数是否为 9
- 文件名是否从 `01` 到 `09`
- 输出目录是否按主题独立

## 十一、推荐的实现顺序

别一上来全做完。

按下面顺序最稳：

### Phase 1｜先把链跑通
- 建目录
- 写 `card-shell`
- 写 5 个模板
- 支持 JSON 输入
- 用假数据渲 1 组 9 卡

### Phase 2｜接入真实内容
- 用 Codex v3 做第一组样板
- 用知识库 v3 做第二组样板
- 调整爆字点和空卡点

### Phase 3｜补校验
- 加 `validate.js`
- 加长度告警
- 加缺卡告警

### Phase 4｜再谈皮肤
- 再加第二套 theme
- 再考虑 icon / 插图 / 封面变体

## 十二、当前明确决策

### 已拍板

1. 先只做 5 个模板，不做 9 个模板
2. 先只做 1 套主题皮肤：`clean-news`
3. 先只支持 JSON
4. 先用 Playwright 导图
5. 先做结构错误报错、长度问题告警
6. 输出固定为 `01.png ~ 09.png`

### 暂不拍板

1. 是否加插图系统
2. 是否支持 YAML
3. 是否加在线预览页
4. 是否做多主题换肤
5. 是否支持一键拼长图

这些都不影响第一版跑起来。
