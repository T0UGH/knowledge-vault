# AI 从工具到协作者的具体场景

## 当前判断

如果要写《我为什么开始把 AI 当协作者，而不是工具》，最有价值的不是空谈“AI 很强”，而是把真实协作中发生的结构变化写出来。

当前最值得写的，不是“AI 帮我做了很多事”，而是：

> **AI 开始进入我的工作结构。**

---

## 4 个最值得写的具体场景

### 1. AI 开始帮我收口问题，而不只是回答问题

以 `Agent Git Mail` 为例，MT 做的不是单次回答，而是在持续帮用户收口关键边界：

- 要不要 server
- monorepo 还是拆仓
- shared 要不要
- 文件名是不是主标识
- checkpoint 要不要自定义文件
- OpenClaw 集成走 plugin 还是碰 core

这说明协作已经从“给答案”变成“持续维护问题边界”。

### 2. AI 开始帮我维护状态，而不只是完成单次任务

在这轮推进中，MT 持续帮助维护：

- 哪些已经拍板
- 哪些还没定
- 哪些要写进 design
- 哪些要写进 Obsidian
- 哪些要发给 Hex
- 哪些已经 commit / push / publish

这已经不是单轮问答，而是持续上下文里的共同推进。

### 3. AI 开始进入多 agent 分工设计，而不只是单个工具调用

尤其在 MT / Hex 分工上，MT 做的不只是“派任务”，而是在持续判断：

- 哪些任务该给 Hex
- 哪些不该给它
- 模型降配后分工如何调整
- implementation plan 是否该由 MT 接管
- review 应该卡在哪个标准上

这说明 AI 开始进入协作组织层，而不是只停留在执行层。

### 4. AI 开始参与 review、验收和质量门禁

在 review Hex 的 `Agent Git Mail` 实现时，MT 明确执行了质量门禁：

- 没有 E2E，不准合入
- `send/reply/archive` 没有端到端测试，不准过
- plugin 现在只能说“可安装可加载”，不能说“功能闭环已验证”

这体现出 AI 不再只是生成器，而是开始进入结果判断链路。

---

## 这 4 个场景为什么值钱

它们共同说明了一件事：

- AI 不只是提高了局部效率
- AI 开始进入分工、状态管理、review、验收这些更高层的工作结构

也正因为如此，AI 对用户来说开始更像“协作者”，而不只是“工具”。
