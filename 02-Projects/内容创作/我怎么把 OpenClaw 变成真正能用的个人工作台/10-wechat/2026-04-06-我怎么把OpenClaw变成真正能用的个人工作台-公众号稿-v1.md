# 我怎么把 OpenClaw 变成真正能用的个人工作台

我现在越来越不相信一件事：

**装好了一个 AI agent，它就会自动变成你的工作搭档。**

现实通常不是这样。

第一次把 OpenClaw 跑起来的时候，我也会有那种熟悉的兴奋感：它能聊天，能调工具，能跑 agent，能接很多模型，看起来已经不是一个普通的 AI 对话框了。

但新鲜感过去之后，真正的问题很快就冒出来了：

**它到底该怎么接进我的日常工作？**

如果没有后面的那套工作流，OpenClaw 很容易停留在一种不上不下的状态：看起来比聊天框更高级，但日常里还是只是一个偶尔被叫出来问几句的工具。

我后来慢慢意识到，OpenClaw 真正有价值的地方，不是“我又装了一个 agent”，而是它逼着我把 AI、CLI、知识库、内容工作流和长期协作习惯，重新编成一个系统。

所以这篇文章不讲安装步骤，也不讲一堆参数怎么配。

我更想讲的是：

> **我是怎么把 OpenClaw 从一个装好的 agent，慢慢变成一个真正能进入我日常工作流的个人工作台。**

## 一、真正的问题从来不是“装没装好”，而是“有没有默认工作方式”

我后来发现，很多 AI 工具之所以最后没进入工作流，不是因为能力不够，而是因为没有形成稳定的默认路由。

说白了，就是当一个任务过来时，你有没有一条足够省脑子的默认反应。

比如：

- 看到一篇普通网页，是直接丢给聊天框，还是先抽成干净 Markdown？
- 要查一个平台上的信息，是开浏览器手动点，还是让 agent 去做？
- 一份高价值判断，是留在聊天记录里，还是沉淀进知识库？
- 一个已经反复验证有效的流程，是继续靠记忆，还是固化成 skill？

如果这些问题每次都临场重想一次，系统很快就会变成负担。

所以我后来做的，不是“再学会几个新工具”，而是先把默认路由固定下来。

比如我现在大概会这么处理：

- 普通网页、博客、教程、文档页，先抽成干净 Markdown 再读
- 要读平台内容、做站内搜索或简单操作，优先交给对应的 CLI
- 飞书文档、云盘、权限这类结构化操作，优先走飞书 CLI，不回浏览器点来点去
- GitHub 上的 repo、issue、PR、CI，优先走 `gh`
- 长期有价值的判断，及时落进 Obsidian
- 一种流程只要开始反复出现，就想办法把它收成 skill

这些习惯看起来都不大，但它们连起来以后，OpenClaw 才第一次不再像一个临时调用的工具，而开始进入我日常工作的分工结构里。

## 二、我真正缺的不是另一个 agent，而是一个长期知识层

如果一个 AI 系统想长期帮到你，它就不能只有会话，没有知识层。

这件事我后来感受越来越强烈。

因为很多真正值钱的东西，都不是一轮对话里的答案，而是这些更慢的东西：

- 某个项目到底为什么这么设计
- 哪些默认路由已经验证过，别再反复试错
- 什么工具适合什么场景
- 哪些规则值得固化成 skill
- 哪些内容以后还会继续被拿出来复用

如果这些东西全都留在聊天记录里，它们本质上还是一次性的。

所以我后来给 OpenClaw 补上的第一层，不是更多 agent，而是一个长期知识库。

我现在把 Obsidian 当成这套系统的长期知识层。它不是记事本，也不是简单的文章仓库，而更像是外部大脑：

- 项目笔记放在 `Projects`
- 架构理解、调研和共读放在 `Research`
- 稳定事实和文档镜像放在 `Reference`
- 内容产出和主题工作区也各有自己的位置

更重要的是，我后来开始意识到，知识库一旦变大，真正重要的已经不是“放进去了”，而是：

> **它有没有一个稳定的结构，能让未来的你和 agent 继续用起来。**

所以后面我又补了一层治理动作：

高密度目录要有 README / index，重复正文要收掉，项目目录和研究目录要分清，孤儿文件不能长期躺在根目录里。

这时候 OpenClaw 和 Obsidian 的关系，也就不再是“一个负责聊天，一个负责记笔记”，而是变成了：

- OpenClaw 负责推进、调用工具、组织工作流
- Obsidian 负责把长期有价值的结构沉淀下来

这两层一接上，整个系统才第一次开始有“长期工作台”的味道。

## 三、CLI 不是附属品，而是这套工作台的骨架

我现在越来越不相信“全靠一个超级 agent 就能把所有事都做好”。

真正顺手的状态，反而更像是：

> **OpenClaw 作为主控，下面接一组各司其职的 CLI 和 skill。**

这也是我后来越来越喜欢 CLI 协作方式的原因。

因为很多事情，本来就不该全靠一个聊天框来吞。

拿几个最典型的场景来说。

### 1. 信息获取
普通网页、博客、文档页，最怕的是噪音太多。先把网页抽成干净 Markdown，再交给 agent 读，成本更低，也更稳定。

而像 Twitter、YouTube、B 站、小红书、微博这些平台内容，本来就不是普通网页。这个时候与其硬上浏览器自动化，不如直接走更贴近平台的 CLI。

### 2. 飞书这类工作平台
飞书文档、知识库、云盘、权限，这些都是典型的结构化操作。用浏览器当然能做，但一旦你开始频繁处理这些东西，就会越来越希望它们可以通过 CLI 或 skill 稳定完成。

### 3. GitHub 和工程协作
如果你本来就深度用 GitHub，那 `gh` 这种工具几乎应该是标配。repo、issue、PR、CI、release，本来就适合成为工作流的一部分，而不是每次回网页点几下。

### 4. 写作与分发
后来我甚至把写作也拆成了一条工具链：格式化、翻译、转 HTML、发公众号、发 X、生成封面、做配图……

这时候你会发现，OpenClaw 最强的地方不是“它自己会做所有事”，而是：

> **它能把这些分散的能力组织起来，让你形成一套稳定的默认工作流。**

所以我现在越来越把 CLI 看成这套工作台的骨架，而不是附属工具。

## 四、我后来又补了一层：把好用的工作流做成 skill

如果一个工作流反复出现，但你每次都重新解释一次，那它其实还不算进入系统。

这件事在知识库治理上我感受特别深。

比如我们后来反复在做这些事：

- 扫描 Obsidian 哪里乱了
- 判断哪些是低风险结构修复，哪些需要人拍板
- 合并重复目录
- 清理重复正文
- 给高密度目录补 README / index
- 最后 pull / commit / push

一开始这些都只是聊天里的临时判断。

但当一个流程反复出现以后，我就不想再让它只存在于聊天里了。

更好的做法是把它收成一个 skill。

后来我给自己建了一个独立的自建 skill 仓库，把这些已经验证过的好流程当成长期资产来管理。仓库是真身，运行目录只是安装态。后面再加新 skill，也都走同一套路径。

这件事看起来像是“多做了一步”，但长期看非常值钱。

因为从那一刻起，经验就不再只是“我记得以前这么做过”，而变成：

> **这已经是系统的一部分。**

## 五、OpenClaw 真正改变我的，不是效率，而是工作结构

如果只把 OpenClaw 当成一个更强一点的聊天框，那它当然也有用。

但我后来越来越清楚地意识到，它真正改变我的地方，不只是“更快做完一件事”，而是工作结构本身开始变化了。

我现在更少问“这个工具还能不能多帮我一点”，而更常问：

- 这个任务应该走哪条默认路由？
- 哪些东西应该沉淀进知识库？
- 哪些流程应该固化成 skill？
- 哪些工作适合由长期 agent 承担，哪些更适合交给工程执行器？

换句话说，OpenClaw 对我真正有价值的地方，不是替我多做几个动作，而是逼着我把自己的工作台重新组织了一遍。

而一旦这套组织开始成形，AI 对我来说也就不再只是一个回答器，而开始变成一个真正能进入工作结构的协作者系统。

## 结尾

所以如果你刚装好 OpenClaw，我其实不太建议你一上来就追求多模型、多 agent、自动化拉满，什么都想接。

我更建议你先把四件事跑顺：

1. 一套默认工具路由
2. 一个长期知识库
3. 一套顺手的 CLI 工具箱
4. 一个自建 skill 的真身仓库

这四样一旦连起来，OpenClaw 才会从一个“已经装好的 agent”，慢慢变成一个真正能进入你日常工作流的个人工作台。

对我来说，这才是它最有意思的地方。
