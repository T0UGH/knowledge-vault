# Obsidian Reminder System - Spec v0.1

## 1. 目标
构建一套**低频但可靠**的提醒体系：
- 你在 Obsidian 中记录提醒项
- 系统定时扫描提醒项
- 到触发时间后，通过飞书私聊提醒你
- 必要时把提醒结果写回 Obsidian，避免重复提醒

这不是一个完整 GTD/Todo 系统，而是一个**提醒系统**。

## 2. 设计原则
1. **Obsidian 是记录源**：提醒定义写在 Obsidian 中，而不是写死在脚本里。
2. **飞书是主提醒通道**：你不依赖日历，因此提醒要主动推送。
3. **低频优先稳**：场景不会特别多，优先简单、透明、可维护，不追求复杂自动化。
4. **双时间模型**：既支持固定重复时间，也支持一次性提醒。
5. **避免重复轰炸**：同一提醒命中后需要记录状态，防止重复发送。

## 3. 项目位置
项目目录放在 Obsidian Vault：

`02-Projects/obsidian-reminder-system/`

第一版文件：
- `README.md`：项目说明
- `spec.md`：设计规格
- 后续实现阶段再补：提醒数据文件、状态文件、脚本说明

## 4. 提醒模型
第一版支持两种触发方式：

### A. 固定周期提醒
使用 cron 表达式：

```yaml
id: renew-gpt-subscription-monthly
text: 检查 GPT 订阅是否需要更新
channel: feishu
cron: 0 10 1 * *
```

含义：每月 1 日 10:00 提醒一次。

### B. 一次性提醒
使用绝对时间：

```yaml
id: renew-gpt-subscription-once
text: 更新 GPT 订阅
channel: feishu
at: 2026-04-24 09:00
```

含义：在指定时间触发一次。

## 5. 为什么不是只用 cron
cron 表达式适合“每月 1 号”“每周一 9 点”这种**固定重复规则**，但不适合自然表达“25 天后提醒我”这种**相对一次性提醒**。

因此第一版采用：
- `cron:` 处理重复提醒
- `at:` 处理一次性提醒

当你说“25 天后提醒我更新 GPT 订阅”时，系统/代理应把它转成一个具体的 `at:` 时间，而不是强迫你手写奇怪的 cron。

## 6. 存储格式建议
第一版建议将提醒项写入一个独立文件，便于维护与扫描。

建议文件：

`02-Projects/obsidian-reminder-system/reminders.yaml`

每条提醒字段：
- `id`：稳定唯一 id
- `text`：提醒内容
- `channel`：默认 `feishu`
- `cron` 或 `at`：二选一
- `enabled`：是否启用
- `last_notified_at`：最近一次提醒时间（可写回）
- `done`：是否完成（一次性提醒用）

示例：

```yaml
- id: renew-gpt-subscription-monthly
  text: 检查 GPT 订阅是否需要更新
  channel: feishu
  cron: 0 10 1 * *
  enabled: true

- id: renew-gpt-subscription-once
  text: 更新 GPT 订阅
  channel: feishu
  at: 2026-04-24 09:00
  enabled: true
  done: false
```

## 7. 运行方式
第一版推荐使用**定时扫描器**，而不是每条提醒都直接注册为系统任务。

### 方案对比
#### 方案 A：每条提醒都注册成系统 cron/launchd 任务
- 优点：触发直接
- 缺点：管理复杂、增删改麻烦、与 Obsidian 脱节

#### 方案 B：固定频率扫描提醒文件（推荐）
- 例如每小时扫描一次，或者每 15 分钟扫描一次
- 读取 `reminders.yaml`
- 解析 `cron` / `at`
- 判断是否命中触发窗口
- 命中后发送飞书提醒
- 写回状态，避免重复提醒

推荐 **方案 B**，因为它更简单、更稳，也更容易维护。

## 8. 提醒输出
主输出通道：**飞书私聊**。

提醒文案应简洁，例如：

```text
提醒：更新 GPT 订阅
来源：Obsidian Reminder System
规则：2026-04-24 09:00
```

后续可选增强：
- 附上提醒 id
- 附上 Obsidian 源文件路径
- 附上“已完成/稍后提醒”的快捷操作

## 9. 去重与状态
必须避免重复提醒。

### 一次性提醒
- `at` 命中后发送一次
- 成功发送后标记：`done: true`
- 不再重复发

### 周期提醒
- 需要记录 `last_notified_at`
- 同一个 cron 时间窗口内只发一次

## 10. 第一版边界
第一版**不做**：
- 自然语言时间解析
- 复杂优先级/标签体系
- 多通道路由
- 完整 GTD / 项目管理
- 复杂交互式确认流

第一版**只做**：
- Obsidian 文件定义提醒
- 支持 `cron` + `at`
- 定时扫描
- 飞书提醒
- 基础去重/状态写回

## 11. 成功标准
如果满足以下几点，就算第一版成功：
1. 你能在 Obsidian 中新增一条提醒项
2. 系统能稳定识别 `cron` 和 `at`
3. 到点后会通过飞书私聊提醒你
4. 不会在短时间内重复轰炸
5. 一次性提醒触发后能正确结束

## 12. 推荐实现顺序
1. 确定提醒文件格式（`reminders.yaml`）
2. 写一个本地扫描器，先只打印命中结果
3. 接飞书发送
4. 加入状态写回
5. 接入定时运行
6. 最后补 README / 使用说明 / 示例提醒

## 13. 当前推荐结论
推荐采用：
- **Obsidian 作为唯一记录源**
- **`cron + at` 双轨时间模型**
- **固定频率扫描器**，而不是每条提醒注册系统任务
- **飞书私聊作为主提醒出口**

这是目前最贴合贵平使用习惯、复杂度最低、可靠性最高的方案。
