# AGM 1.0 contract design

日期：2026-04-05
状态：draft

## 1. 目标

AGM 1.0 的目标不是把系统扩成完整的 agent coordination platform，而是把当前已经成立的 MVP 收成一个**边界清楚、对小团队可复用、工程语义自洽**的 Git-backed async mailbox transport。

一句话：

> AGM 1.0 是一个面向长期在线助理型 Agent 的 Git-backed async mailbox transport。

它不是：

- IM 系统
- 低延迟消息总线
- task queue
- workflow orchestrator
- 完整 agent runtime

本轮设计的重点不是扩功能，而是把以下 4 类 contract 收死：

1. transport contract
2. failure and recovery contract
3. wakeup and activation contract
4. observability and diagnosis contract

---

## 2. transport contract

### 2.1 message identity

AGM 1.0 中：

- `filename` 是 **canonical message id**
- AGM 1.0 **不引入独立 `message_id` 字段**
- `filename` 同时承担：
  - 物理文件名
  - transport 层唯一消息标识
  - `reply_to`、dedupe、checkpoint correlation 的引用目标

这不是“暂时拿文件名代替 id”，而是协议层的正式选择。

### 2.2 filename uniqueness

AGM 1.0 对 filename 的要求是：

- 必须人可读
- 必须包含独立的 uniqueness token
- token 的具体形式属于实现细节，不在协议层写死为 UUID / ULID / 短随机串中的某一种
- 协议层只要求：**filename 必须承担唯一标识职责**

也就是说：

> 人可读是产品形态要求；唯一性是协议要求；token 形式是实现选择。

### 2.3 reply semantics

`reply_to` 是可选字段。

当 `reply_to` 存在时：

- 它必须引用一个 message filename
- AGM 在执行 `reply` 动作时，必须基于**本地可验证存在的原消息**生成 reply
- AGM 1.0 不承诺全局 reply graph 完整性
- AGM 1.0 不把 thread / obligation / conversation completion 等上层语义塞进 transport contract

这意味着 AGM 对 reply 的承诺是：

> 当前 reply 动作有本地可验证依据；至于更高层的 thread 完整性，由上层协议负责。

### 2.4 duplicate semantics

AGM 1.0 中，duplicate 只按 filename 判定：

- 同 filename = 同一消息
- 不同 filename = 不同消息
- AGM 1.0 **不定义内容级 duplicate 语义**

即使两条消息内容完全相同，只要 filename 不同，AGM transport 仍视其为不同消息。

这条规则的目的，是防止 transport 层自作聪明地引入内容级 dedupe，并把本应属于上层协议的判断带入底层。

### 2.5 archive semantics

archive 是目录迁移动作，而不是 identity 变更动作。

AGM 1.0 中：

- `inbox/<filename>` → `archive/<filename>` 不改变 message id
- message identity 与消息当前位于 `inbox / outbox / archive` 无关
- archive 只改变生命周期位置，不改变消息身份

### 2.6 send / reply success semantics

由于 AGM 当前采用 dual-write 模型：

- sender side 写入 outbox copy
- recipient side 写入 inbox copy

所以 AGM 1.0 必须承认 `send` / `reply` 至少有两个层次的结果：

1. `local_success`
2. `delivery_success`

定义如下：

- `local_success`：sender / replier 的本地副本创建成功
- `delivery_success`：recipient inbox 副本创建并完成交付成功

因此：

- AGM 1.0 不再假装 `send` / `reply` 只有 success / fail 两态
- `partial success` / `partial failure` 是一等事实

这条是 1.0 收口的核心之一，因为它决定后续 failure、doctor、log、diagnosis 的全部语义基础。

## 3. failure and recovery contract

### 3.1 partial success handling

AGM 1.0 对 partial success 的处理策略是：

> **fail-fast + explicit partial failure**

也就是：

- 如果 local copy 已创建，但 recipient delivery 失败
- 命令应返回失败
- 但失败必须明确说明：
  - 哪一步成功了
  - 哪一步失败了

AGM 1.0 不把这种情况压扁成一个模糊的“发送失败”。

### 3.2 no dedicated recovery state store

AGM 1.0 不引入独立的 partial-failure recovery state store。

也就是说：

- 不维护 pending delivery state machine
- 不维护 recovery queue
- 不维护单独的 compensating-action ledger

partial success / partial failure 的暴露方式只包括：

1. 命令返回结果
2. structured event log

原因很直接：

- AGM 1.0 需要诚实暴露失败事实
- 但不应该因此偷偷长出第二套运行时真相

### 3.3 common repo failure strategy

AGM 的真实高频风险，并不主要是复杂 rebase conflict，而是：

- remote advanced / non-fast-forward
- stale local clone
- 短暂网络抖动
- pull/push 过程中的常见并发失败

因此 AGM 1.0 的策略不是“自动 rebase 一切”，而是：

> **防住常见、无歧义、可重试的问题；暴露真实语义冲突。**

### 3.4 retry policy for repo operations

AGM 1.0 中：

- 对以下问题可做**有限自动重试**：
  - transient infrastructure failure
  - remote advanced / non-fast-forward 类常见并发失败
  - stale local clone 导致的可恢复失败
- 对以下问题一律 **fail-fast**：
  - 真实语义冲突
  - 需要猜测用户意图的恢复
  - 隐式 merge / reconcile

也就是说：

- AGM 1.0 可以吞掉高频机械问题
- AGM 1.0 不自作聪明地修语义问题

这条边界必须写死，否则系统会滑向半个 Git orchestrator。

## 4. wakeup and activation contract

### 4.1 activation retry strategy

AGM 1.0 对 activation failure 的策略是：

> **单次 poll 内有限同步重试；失败后等待后续 poll 再试。**

具体来说：

- 对明确 transient 的 activation failure，可在同一次 poll 中重试少量次数
- 若仍失败：
  - 记录 structured event
  - 本轮结束，不进入独立 activation retry queue
- 后续依赖下一次 poll 再尝试

AGM 1.0 不引入独立 activation retry subsystem。

### 4.2 checkpoint semantics

checkpoint 记录的不是“消息已处理”，而是：

> **AGM 已成功完成 wakeup dispatch**

这意味着 checkpoint 只表示：

- activator / host adapter 已成功返回成功
- AGM 认为唤醒动作已成功发出

checkpoint 不表示：

- 宿主已 ingest
- agent 已读
- agent 已处理
- obligation 已关闭
- 任务已完成

### 4.3 checkpoint write timing

checkpoint 的写入时机必须非常明确：

- **只有 activation success 后才允许写 checkpoint**
- activation failed 时不能提前写 checkpoint

否则系统会把失败消息错误地视为“已处理过，不要再唤醒”，这会直接损害唤醒可靠性。

### 4.4 first-run semantics

AGM 1.0 中，daemon 首次启动的语义是：

- 建立 waterline
- 不对历史邮件补发 wakeup

也就是说：

> AGM 1.0 daemon 默认只处理“启动后观察到的新变化”。

这条约束的目的，是避免 first-run 变成历史邮件重放器，并造成不可预测的历史邮件风暴。

如果将来有历史扫描需求，应通过显式命令或独立机制处理，而不混入 daemon 默认语义。

## 5. observability and diagnosis contract

### 5.1 structured event truth

AGM 1.0 的关键运行时事实，必须通过 structured event log 留下证据。

至少包括：

- activation success
- activation failure
- activation skipped by checkpoint
- pull / push / timeout / remote-advance 类关键失败
- partial success / partial failure
- checkpoint write / checkpoint-based skip

原则是：

> AGM 不一定自动恢复，但必须留下足够证据，让人和工具能判断发生了什么。

### 5.2 doctor expectations

AGM 1.0 的 `doctor` 不负责执行恢复，但要负责把 failure class 说清楚。

至少应能识别：

- recent activation failures
- recent repo sync / remote-advance 类问题
- waterline / checkpoint 的基本一致性
- first-run 正常状态 vs 异常状态
- partial success evidence

也就是说，doctor 应该能回答这类问题：

- 最近 daemon 有在正常轮询吗？
- 最近 activation 是成功还是失败？
- 是第一次运行没建 waterline，还是运行异常？
- 有没有发生 local success 但 delivery 失败的情况？

### 5.3 diagnosis philosophy

AGM 1.0 不追求一上来就自动修所有问题。

它更强调：

- 把关键失败分型清楚
- 把证据暴露出来
- 把高频机械问题自动吞掉
- 把真正危险的语义问题升级为明确诊断

这比“表面上很智能，实际上偷偷吞错”更符合 AGM 的 transport 定位。

## 6. 这版 1.0 设计的核心判断

这轮 design 的核心不是加功能，而是收边界。

最终 AGM 1.0 的姿态应当是：

1. **承认 transport 的真实复杂度**
   - dual-write
   - partial success
   - wakeup dispatch
   - repo concurrently advances

2. **拒绝把复杂度伪装成不存在**
   - 不再只说 success / fail
   - 不再把 activation failure 和 checkpoint 混掉

3. **同时避免长成更重的系统**
   - 不引入 recovery queue
   - 不引入 pending state machine
   - 不引入独立 activation retry subsystem

一句话总结：

> AGM 1.0 要做一个诚实、克制、工程边界清楚的 transport，而不是一个偷偷膨胀的半成品 orchestrator。

---

## 7. 下一步建议

在这版 contract design 基础上，下一步应进入 implementation planning，重点围绕：

1. 哪些 contract 已由现有代码满足
2. 哪些 contract 需要补测试
3. 哪些 contract 需要补 structured events / doctor checks
4. 哪些命令返回结构需要更清晰地区分 local success / delivery success
5. 哪些 repo failure class 需要明确定义为 retryable vs fail-fast
