# AGM 1.0 contract closure note

日期：2026-04-05
状态：closed

## 结论

AGM 1.0 的 contract closure 可以认为已经完成。

这里的“完成”不是指 AGM 已经变成一个更重、更全的 agent orchestration system，而是指：

> AGM 作为一个 **Git-backed async mailbox transport**，其 1.0 该收口的核心 contract、关键实现语义、主要验证路径，已经基本对齐。

这轮收口之后，AGM 的系统姿态更清楚了：

- 它不是 IM
- 不是 MQ
- 不是 task orchestrator
- 不是完整 agent runtime
- 它是一个诚实、克制、边界清楚的 transport + wakeup layer

---

## 本轮完成了什么

### 1. transport contract 收口

以下 1.0 contract 已被明确并落实到实现/测试语义中：

- `filename` 作为 canonical message id
- 不引入独立 `message_id`
- `reply_to` 采用本地可验证原消息存在的语义
- duplicate 仅按 filename 判定
- archive 不改变 message identity
- `send` / `reply` 明确区分：
  - `local_success`
  - `delivery_success`
- partial success / partial failure 被正式承认为一等事实

这一步最关键的价值在于：

> AGM 不再假装 send/reply 只有 success / fail 两态，而是正面承认 dual-write transport 的真实复杂度。

### 2. CLI 语义收口

CLI 侧已与 transport contract 对齐：

- `send` / `reply` 的返回结果不再只剩 filename
- partial failure 会以显式方式暴露
- exit code 已回到 final plan 所定义的语义
- `--json` 输出可承载机器可读结果

这意味着 AGM 对“人类调用方”和“自动化调用方”都比之前更诚实。

### 3. wakeup / activation contract 收口

本轮一个关键修正是：

- daemon first-run 语义已改成 **只建立 waterline，不 replay 历史 wakeup**

同时：

- activation failure 已支持单次 poll 内有限同步重试
- activation retries exhausted 有明确事件证据
- checkpoint 继续保持“仅在 activation success 后写入”

这让 AGM 的 daemon 语义从“实现里大概这样”变成了“可以被明确陈述和验证的 contract”。

### 4. doctor / event 诊断能力增强

本轮收口后，AGM 的运行时证据能力比之前更强：

- activation retrying / exhausted 等事件已可见
- push / pull / remote-advance 类失败有了更明确的 event / doctor 支撑
- doctor 对 recent activation / retry / repo failure class 的诊断能力增强

这一步的价值不是“自动修复”，而是：

> AGM 现在更像一个有证据链的系统，而不是一个只能靠猜的工具。

### 5. 验证闭环成立

本轮最终验证结果：

- **76 / 76 测试通过**
- **类型检查通过**
- **构建通过**

这意味着：

- contract closure 不是停留在文档层
- 关键实现与验证已经形成闭环

## 这轮为什么算“收口完成”

因为 AGM 1.0 最关键的事情不是“再多长几个功能”，而是以下三件事：

### 1. 承认真实复杂度

AGM 现在正式承认了：

- dual-write 会带来 partial success
- wakeup dispatch 不是 task completion
- daemon first-run 需要明确边界
- repo concurrently advances 是 transport 现实，不是意外

### 2. 不偷长系统

本轮刻意没有引入：

- recovery queue
- pending delivery state store
- compensation worker
- host-ingest acknowledgment protocol
- 独立 `message_id`

这很重要。

因为 AGM 的价值不在于变成“半个 orchestrator”，而在于：

> 在 transport 这一层把事情说清楚、做扎实、证据留够。

### 3. 用验证而不是口头完成

这轮不是“感觉差不多了”，而是：

- 有 design
- 有 implementation plan
- 有双 plan review
- 有两轮 review round
- 有测试 / build / typecheck 结果

从工程纪律角度，这已经满足“可以宣布收口”的标准。

---

## 仍然保留的 defer 项

这轮收口不等于 AGM 没有后续空间。

当前最明确的 defer 项包括：

### 1. `error-classification.ts`

即 git 层面更精细的结构化错误分类，例如：

- `NON_FAST_FORWARD`
- `TRANSIENT_NETWORK`
- `SEMANTIC_CONFLICT`
- `REBASE_CONFLICT`

这部分当前没有完全做成独立、系统化的错误分类层。

判断：
- 有价值
- 但不阻塞 1.0 contract closure
- 更适合作为 **1.1 failure semantics 精细化** 继续推进

### 2. event taxonomy 的进一步精细化

当前 observability 已经明显增强，但仍可能存在“事件已注册、语义还有进一步细化空间”的尾项。

判断：
- 这是 1.1 级 polish
- 不影响 1.0 的核心 contract 已成立

## 当前推荐状态表述

如果需要对内/对外描述 AGM 当前状态，推荐直接用这句话：

> AGM 1.0 contract closure 已完成：核心 contract、CLI 语义、daemon first-run、activation retry、doctor 诊断、测试/类型检查/构建都已对齐并验证通过。剩余少量 1.1 级尾项主要在 git failure classification 与更细的 event taxonomy，不阻塞 1.0 收口。

---

## 下一步建议

AGM 现在不应该立刻重新发散，而应该进入更节制的下一阶段：

### 建议 1：把 1.0 当作已收口基线使用

后续若继续改 AGM，应默认以当前 contract closure 结果为稳定基线，而不是重新回到“README 气质项目”阶段。

### 建议 2：1.1 只做 failure semantics 精细化

下一轮若继续推进，建议把重点收窄在：

- git error classification
- event taxonomy polish
- 必要时再补少量 adverse-case tests

### 建议 3：不要轻易让 AGM 长成更重系统

后续所有新增设计都应持续接受一个问题的审查：

> 这是不是在偷偷把 AGM 从 transport 推向 orchestrator？

如果答案是“有点像”，那就应该非常谨慎。

---

## 一句话总结

AGM 1.0 这轮最重要的成果，不是“功能又多了几个”，而是：

> **它终于像一个边界清楚的系统，而不是一个很酷但容易被误读的概念件。**
