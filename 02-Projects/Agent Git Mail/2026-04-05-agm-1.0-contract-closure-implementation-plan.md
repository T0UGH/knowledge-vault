# AGM 1.0 contract closure implementation plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Align AGM code, tests, events, and doctor behavior with the approved 1.0 contract design, without expanding AGM into a heavier orchestration or recovery system.

**Architecture:** Keep AGM as a transport + wakeup layer. Close the contract in four passes: transport semantics, failure semantics, wakeup semantics, and observability/doctor semantics. Prefer small, contract-driven code changes backed by tests, and avoid adding new persistent recovery state or background compensation subsystems.

**Tech Stack:** TypeScript, Node.js, Vitest, Git CLI, AGM profile-based config, structured event logs

---

## Scope and phase boundaries

### Must for 1.0

1. Make `send` / `reply` results and failures reflect `local_success` vs `delivery_success`
2. Strengthen filename uniqueness behavior without introducing a separate message ID
3. Add explicit event coverage for partial failure and retry exhaustion / retryable repo failures
4. Tighten daemon / activation semantics around retry, checkpoint timing, and first-run behavior
5. Extend tests to cover the new contract truth, especially adverse cases
6. Extend doctor to diagnose the newly formalized failure classes

### Should for 1.0 polish

1. Remove stray debug output from tests
2. Improve log messages / help output to match contract wording
3. Add one concise contract-oriented developer doc inside repo docs

### Can defer

1. Any dedicated recovery state store
2. Any background compensation queue
3. Any host-level ingest acknowledgment protocol
4. Any content-level duplicate detection
5. Any protocol v2 redesign with explicit `message_id`
