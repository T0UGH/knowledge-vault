# AGM 1.0 收口设计稿（平衡版）

日期：2026-04-05

状态：draft

## 1. 背景

当前 `agent-git-mail`（AGM）项目已经不是概念验证阶段，而是 **MVP 闭环已基本成立** 的阶段。

从当前仓库与实现状态看，已经具备：

- CLI 主体命令
- daemon
- external activator
- OpenClaw 类长期在线 agent 的唤醒路径
- profile 模型
- Docker / E2E / smoke 测试痕迹
- `doctor` / `log` 等运行期辅助命令

当前真正的问题，不再是“缺少核心功能”，而是：

> **主路径、主模型、主文案还没有完全统一成一个真相。**

这使得 AGM 虽然已经具备 MVP 能力，但距离一个工程收口型 1.0 还差一轮系统性的 polish。

---

## 2. 本轮目标

本轮目标定义为：

> **把 AGM 从“已经能跑的 MVP”收成“边界清楚、入口统一、文案一致、工程认知成本可控的 1.0”。**

这不是一次功能扩展迭代，而是一次 **收口迭代**。

### 本轮目标关键词

- 统一主入口
- 统一 profile 模型心智
- 消除 pre-profile 残留
- 重写 README
- 移除过时安装脚本
- 保证 MVP 闭环在收口后仍然成立

---

## 3. 本轮范围

### 3.1 包含

1. **README 重写**
2. **CLI bootstrap 主入口收口**
3. **删除 shell wrapper 主路径**
4. **help / usage / 输出文案收口**
5. **daemon / discovery / activation 相关旧术语清理**
6. **必要的小代码修正**
7. **收口后验证主闭环仍然可用**

### 3.2 不包含

本轮明确不做以下内容：

- 新功能扩展
- 新 host integration 能力
- activation 模型扩展
- 新 plugin / runtime 抽象
- 协议层重写
- 大规模 identity 模型重构
- 新 installer / wizard / UI

一句话：

> **本轮只做收口，不长功能。**

---

## 4. 当前问题归纳

### 4.1 README 与真实状态不一致

当前 README 仍然保留较强的早期项目气质：

- 还没准确强调 AGM 的 MVP 已成立
- 入口叙事仍受旧安装脚本影响
- profile-first 模型没有成为唯一 onboarding 心智
- “谁适合 / 谁不适合”表达仍可更锐利

### 4.2 bootstrap 是最明显的收口断点

当前 CLI 已经进入 profile 模型：

- `--profile` 是核心入口
- `getSelfRepoPath(profile)` 已成为标准路径派生方式
- config 也已经改成 `profiles` 结构

但 `bootstrap` 仍保留较强的旧世界痕迹：

- `--self-id`
- `--self-remote-repo-url`
- `--self-local-repo-path`

其中：

- `--self-local-repo-path` 与 profile 模型的“路径应由 profile 派生”理念冲突最明显
- shell wrapper 仍在围绕旧参数体系组织主入口

### 4.3 shell wrapper 成为第二真相

当前：

- `scripts/bootstrap.sh`
- `scripts/install-openclaw.sh`

本质上在维持一套与 CLI bootstrap 并行的安装心智。

这会导致：

- README 与 CLI 真实入口分裂
- profile 改造后脚本持续漂移
- 用户无法判断哪条路径才是正式入口

### 4.4 其他命令虽然整体已 profile-first，但仍有残留毛边

当前 review 结果表明：

- `send`
- `reply`
- `read`
- `list`
- `archive`
- `daemon`
- `doctor`
- `log`

整体已经是 profile-first，没有第二个像 bootstrap 那样严重的结构断裂。

但仍有两类问题：

#### A. 文案 / 日志残留 pre-profile 术语
例如：

- `self.local_repo_path`
- `repo_path configured`

这类表述已经不符合当前 schema 与路径派生模型。

#### B. `profile / self.id / from` 身份语义尚未完全收干净
当前实现里仍有一些地方会让人感觉：

- `profile` 是运行主体
- 但 `from` 又参与 repo path 推导或执行心智

这不一定立即导致 bug，但会提高认知成本。

## 5. AGM 1.0 的推荐主心智

本轮收口后，AGM 1.0 的推荐主心智应统一为：

> **Agent Git Mail 是一个 git-native 的 agent 异步邮箱系统。它让 OpenClaw 类长期在线 agent 通过 git repo 收发异步邮件，并在新信到达时被唤醒。**

关键点：

- 仍然保留 AGM 当前 README 的主叙事
- 不重写成某个特定宿主（例如 HappyClaw）配套功能
- 强调长期在线助理型 agent 场景
- 不把它定位为任务总线或完整 orchestration 系统

---

## 6. 主入口设计

### 6.1 正式主入口

AGM 1.0 的唯一正式 onboarding 主入口应为：

```bash
npm install -g @t0u9h/agent-git-mail

agm --profile mt bootstrap \
  --self-id mt \
  --self-remote-repo-url <repo>
```

### 6.2 参数设计原则

#### 必须保留在主路径中的参数
- `--profile`
- `--self-id`
- `--self-remote-repo-url`

#### 仅作为高级参数保留，不进入 README 主路径
- `--self-local-repo-path`
- `--config-path`
- `--dry-run`
- `--json`
- `--activation-open-id`

### 6.3 设计理由

#### 为什么 `--profile` 是主入口
因为当前运行时已经是 profile-first：

- config 以 profile 为单位组织
- self repo path 以 profile 派生
- contact cache 以 profile 派生
- state / activation / daemon log 都以 profile 派生

#### 为什么 `--self-id` 仍保留在主路径
本轮采用平衡版，而不是最激进的 profile-only 方案。

保留 `--self-id` 的原因：

- 对用户更显式
- onboarding 更透明
- 排障更直接
- 与当前 bootstrap 实现兼容成本更低

#### 为什么 `--self-local-repo-path` 不再作为主路径参数
因为它与 profile 模型的核心价值冲突。

profile 模型已经建立：

- `getSelfRepoPath(profile)`
- `getContactCachePath(profile, contact)`

因此本地路径应该成为：

> **派生值 / override，而不是用户第一层 onboarding 参数。**

## 7. shell 脚本处理方案

### 7.1 结论

本轮建议：

- 删除 `scripts/bootstrap.sh`
- 删除或下线 `scripts/install-openclaw.sh`

### 7.2 原因

因为 shell wrapper 当前已经不再是轻量便利层，而是：

- 维持第二套参数真相
- 继续固化 pre-profile 心智
- 与 CLI bootstrap 主入口产生竞争关系

这与 AGM 1.0 的收口目标冲突。

### 7.3 原则

本轮不是反对 shell 脚本本身，而是反对：

> **在 1.0 入口阶段继续维持 shell 脚本与 CLI bootstrap 并行的 onboarding 真相。**

---

## 8. README 重写方案

### 8.1 README 目标

README 的目标不再是“项目还在探索什么”，而是：

1. 让用户快速理解 AGM 是什么
2. 让用户快速知道它已经做到什么
3. 让用户快速进入唯一正式路径
4. 让用户理解 AGM 的边界和适用场景

### 8.2 推荐结构

建议 README 调整为以下结构：

1. **一句话定位**
2. **它解决什么问题**
3. **MVP 已经做到什么**
4. **谁适合 / 谁不适合**
5. **Quickstart（CLI bootstrap）**
6. **启动 daemon**
7. **如何验证 AGM 正常工作**
8. **核心模型**
9. **Monorepo 结构**
10. **当前状态 / roadmap**

### 8.3 README 中应删除的内容

- shell 脚本作为主入口的安装路径
- 任何仍暗示 plugin 是主入口的叙事
- 与当前 profile 模型不一致的配置说明

### 8.4 README 中应新增的内容

- 明确标注 MVP 已成立
- Quickstart 只保留 CLI bootstrap 路径
- doctor / log 的正式验证路径
- AGM 服务的是长期在线 agent，不是任务总线

---

## 9. 其他命令与运行时收口

### 9.1 文案 / 日志收口

需要统一替换或改写以下类型的表达：

- `self.local_repo_path`
- `repo_path configured`
- 其它 pre-profile schema 词汇

改写方向：

- `profile`
- `derived self repo path`
- `contact cache path`
- `activation enabled / disabled`
- `host integration`

### 9.2 identity 语义轻收口

本轮不做大规模 identity 重构，但建议明确以下原则：

- `profile` 是运行主体
- `self.id` 是该 profile 的 agent identity
- `from` 是消息语义字段

如有低成本位置，建议补一层 guard 或说明，避免 `profile / self.id / from` 继续漂移。

## 11. 执行顺序

推荐顺序如下：

1. **先收 bootstrap 命令与 help**
2. **删除脚本入口**
3. **重写 README**
4. **清理 daemon / discovery / activation 文案**
5. **必要时补最小 identity guard**
6. **跑一轮 doctor / daemon / send / reply 验证闭环**

这个顺序的原因是：

- 先收真入口
- 再移除旧入口
- 再让 README 对齐新的唯一真相
- 最后补内部文案与轻量 guard

---

## 12. 成功标准

本轮完成后，应满足以下条件：

### 用户入口层
- README 只呈现一个正式 onboarding 路径
- shell wrapper 不再作为主入口存在
- bootstrap 主路径与 profile 模型一致

### 工程一致性层
- CLI help 与 README 一致
- daemon / discovery / activation 不再暴露旧 schema 术语
- 主要命令的使用心智不再在 profile 与旧 self/local path 模型之间摇摆

### 验证层
- 按 README Quickstart 能成功初始化 profile
- daemon 能启动
- send / reply / read / list / archive 主闭环仍可用
- doctor / log 能作为正式诊断入口工作

---

## 13. 最终判断

AGM 距离 1.0 现在差的不是新功能，而是：

> **一次把主路径、主模型、主文案统一成一个真相的收口。**

本设计稿推荐采用 **平衡版** 收口方案：

- 不重写整个系统
- 不扩展功能范围
- 只把用户入口和工程真相收成一致

如果这一轮完成，AGM 就能从“已经能跑的 MVP”提升为：

> **工程边界清楚、入口统一、可以对外讲清楚的 1.0。**
