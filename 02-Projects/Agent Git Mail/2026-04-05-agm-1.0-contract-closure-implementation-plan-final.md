# AGM 1.0 contract closure implementation plan (final)

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Align AGM code, CLI behavior, tests, events, and doctor diagnostics with the approved 1.0 contract design, while keeping AGM a transport + wakeup layer rather than letting it grow into a heavier orchestrator or recovery system.

**Architecture:** Execute contract closure in five passes: transport semantics, CLI/result semantics, failure classification and bounded retry, wakeup/checkpoint alignment, and observability/doctor diagnostics. Use contract-driven tests to force the implementation shape. Prefer explicit return semantics and structured events over any new persistent recovery state.

**Tech Stack:** TypeScript, Node.js, Vitest, Git CLI, AGM profile-based config, structured event logs

---

## Working assumptions and review corrections

Before implementation, lock these judgments in:

1. **Do not treat current code behavior as equivalent to contract closure.**
   Several 1.0 semantics are only implicit today and still need explicit return structures, event semantics, tests, and doctor alignment.

2. **Current daemon first-run behavior is not yet aligned with the approved contract.**
   The approved 1.0 contract says: first run establishes waterline and does **not** replay historical wakeups. Current implementation still processes historical inbox files on first run. This is a real must-fix item.

3. **Do not add a recovery queue, pending-delivery state store, compensation worker, or host-ingest acknowledgment layer.**
   That would violate the design goal of a restrained transport.

4. **Doctor should diagnose failure classes, not become a reconciliation engine.**
   Deep mailbox consistency / cache freshness intelligence is not a 1.0 must unless required by a contract-backed failure class.

---

## Scope and phase boundaries

### Must for 1.0

1. Make `send` / `reply` semantics explicitly distinguish `local_success` vs `delivery_success`
2. Align CLI output / exit code / `--json` behavior with that two-stage truth
3. Strengthen filename uniqueness behavior without introducing a separate message ID
4. Add bounded retry for common retryable repo failures only
5. Add explicit event coverage for partial failure, retry exhaustion, push/pull failure classes, and wakeup retry attempts
6. Fix daemon first-run behavior to match the approved contract
7. Tighten activation retry and checkpoint timing semantics
8. Extend tests to cover the newly formalized contract, especially adverse cases
9. Extend doctor to diagnose the new failure classes using structured evidence

### Should for 1.0 polish

1. Remove stray debug output from tests
2. Improve log and help wording to match the new contract vocabulary
3. Add one concise repo-local contract closure note for maintainers/operators
4. Optionally improve machine-readable CLI output consistency across `send` / `reply`

### Can defer

1. Dedicated recovery state store
2. Background compensation queue
3. Host-level ingest acknowledgment protocol
4. Content-level duplicate detection
5. Protocol v2 with explicit `message_id`
6. Deep mailbox divergence reconciliation
7. Broad cache freshness heuristics beyond contract-backed checks
8. Full concurrent-writer test harness if lightweight approximations already cover the main risk

## Failure classification baseline

These classes are guidance for implementation. Keep them small and contract-driven.

### Retryable common failures

| Class | Typical trigger | Expected handling |
|---|---|---|
| `NON_FAST_FORWARD` | remote advanced before push | bounded retry after refresh/rebase path |
| `TRANSIENT_NETWORK` | timeout / temporary transport issue | short bounded retry |
| `STALE_CLONE` | contact cache or local repo behind remote in a recoverable way | refresh then retry |
| `REMOTE_UNREACHABLE` | temporary remote unavailable | one short retry, then fail |

### Fail-fast failures

| Class | Typical trigger | Expected handling |
|---|---|---|
| `REBASE_CONFLICT` | retry path produces real conflict | fail-fast |
| `SEMANTIC_CONFLICT` | operation meaning is ambiguous / unsafe to auto-fix | fail-fast |
| `CONCURRENT_WRITE` | same logical path or operation conflict that needs judgment | fail-fast |
| `UNKNOWN_RECIPIENT` | contact not configured | fail-fast |
| `IDENTITY_MISSING` | git identity / profile identity invalid | fail-fast |

Important rule:

> AGM 1.0 may absorb high-frequency mechanical failures, but it must not silently repair semantic conflicts.
