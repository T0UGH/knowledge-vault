# Hermes 语音终端方案调研：Home Assistant / 小爱音箱 / 小主机落地路径

## 结论

要做成“唤醒词唤醒，然后直接语音和 Hermes 聊”的体验，**技术上完全可做**。

但主线应该是：

> **做一个可控的语音前端接 Hermes，而不是硬改小爱音箱本体。**

对当前条件的最佳判断：
- 如果只是验证方向：旧手机 / 电脑先跑通语音链路
- 如果要独立设备：`ESP32-S3-BOX` 是很好的第一台语音终端
- 如果要长期可用、能持续演进：**小主机 + 麦克风/麦阵 + 音箱 + Hermes** 才是正路

当前最推荐路线：

> **小主机做中枢，Home Assistant 或轻量桥接层负责唤醒/STT/TTS，Hermes 做大脑。**

---

## 先说判断

### 1. Hermes 现成已经有语音能力，但不是完整“音箱态”
Hermes 已有：
- CLI voice mode
- Telegram / Discord 语音消息
- Discord 语音频道实时对话

这说明：
- Hermes **已经能处理语音输入输出**
- 但它当前并不是一个现成的“常驻待机 + 唤醒词 + 多轮会话”的消费级 AI 音箱方案

所以差的不是“大脑”，而是**语音前端 / 设备层**。

### 2. 小爱音箱本体改造不是主线
小爱这类智能音箱的问题通常不是“有没有麦克风/喇叭”，而是：
- 系统封闭
- 语音链路上云，不可控
- 很难把它变成真正低延迟、持续可编排的本地语音终端

因此更合理的定位是：
- 可以尝试做 Home Assistant 控制入口
- 可以尝试做 TTS 播报终端
- **不建议把它当成 Hermes 主语音终端的主战场**

### 3. Home Assistant 是合适的语音基础设施层
Home Assistant 当前已有较成熟的语音链：
- `openWakeWord`：唤醒词
- `Whisper`：语音转文字（STT）
- `Piper`：文字转语音（TTS）
- `Assist` / `Wyoming` 体系：语音设备接入与管道编排

关键判断：

> Home Assistant 适合做“语音基础设施”；Hermes 适合做“智能处理层”。

不要把复杂 agent 逻辑埋进 HA 自动化里。

---

## 想要的目标体验

目标体验是：

1. 设备常驻待机
2. 说出唤醒词
3. 系统开始监听问题
4. 语音转文字
5. Hermes 收到文本并处理
6. 返回文本结果
7. TTS 播放回答
8. 短时间内可继续追问，超时后退出会话态

这已经是一个完整的 AI 音箱闭环。

---

## 架构建议

### 推荐架构

分 5 层足够：

1. **唤醒词层**
   - `openWakeWord`
   - 负责检测“嘿 XXX”

2. **语音采集层**
   - 麦克风 / 麦阵
   - 将音频送入本地服务

3. **STT 层**
   - `Whisper` 或 HA Assist pipeline
   - 负责把语音转成文本

4. **Hermes 层**
   - 接收文本
   - 维持 session
   - 调工具
   - 生成回答

5. **TTS 播放层**
   - `Piper` 或其他 TTS
   - 把回复播出来

### 一句话理解

> **Home Assistant / 桥接层负责听和说，Hermes 负责想。**

---

## 设备选择判断

### 不是“随便一个音箱都能跑”
关键不是“音箱”，而是：
- 有没有麦克风
- 能不能持续传音频
- 能不能播报回音
- 有没有可控的软件入口

所以设备分三类：

#### 1. 普通音箱
- 只有扬声器
- **不能直接做语音终端**
- 只能当 TTS 播放器

#### 2. 封闭智能音箱（小爱/小度/天猫精灵）
- 有麦克风和喇叭
- 但系统封闭
- 能否接入不取决于硬件，而取决于能否刷机/桥接/开放接口

#### 3. 可编排语音终端
- 小主机
- 树莓派
- ESP32-S3-BOX
- 旧手机 / 安卓设备

**真正适合接 Hermes 的，是第 3 类。**

---

## 硬件路线对照

### 方案 1：旧手机 / 电脑验证版
适合：先验证方向，不买新硬件。

优点：
- 成本最低
- 自带麦克风、扬声器、网络
- 最快跑通体验

缺点：
- 不像独立音箱
- 常驻/唤醒词体验受系统限制

判断：
- **适合先证明方向**

### 方案 2：ESP32-S3-BOX 独立终端版
适合：想要第一台独立语音终端。

优点：
- 路线成熟
- 成本较低
- 比封闭智能音箱更可控

缺点：
- 算力有限
- 更适合做 satellite，不适合承载复杂逻辑

判断：
- **适合作为第一台专用语音终端**

### 方案 3：树莓派 / 小主机 DIY 版
适合：长期使用、深度可控、后续要持续演进。

优点：
- 可控性最强
- 容易接 Hermes
- 容易加 session / 日志 / 状态机 / 多设备扩展

缺点：
- 安装与调优复杂度更高

判断：
- **这是长期正路**

### 方案 4：小爱音箱改造版
适合：只想复用现有设备做轻量实验。

优点：
- 家里可能已有
- 可尝试做控制入口或播报器

缺点：
- 封闭
- 边界受限
- 不适合作为主语音终端方案

判断：
- **不推荐作为主线**

---

## 当前最优路线：小主机方案

已知条件：**已经有小主机**。

因此当前最优路线很明确：

> **小主机做中枢，先跑通单点闭环，再逐步升级到常驻 AI 音箱。**

### 第一版最合理硬件
- 小主机（已有）
- USB 麦克风（先不追求麦阵）
- 本地直连音箱 / 有源音箱
- Hermes
- Whisper
- Piper
- 一个很薄的桥接层（或 HA Assist pipeline）

### 先别一上来就追求
- 多房间
- 多设备漫游
- 声纹识别
- 远场阵列麦复杂优化
- 全双工打断式对话
- 小爱音箱接管

这些都能做，但不该是第一阶段。

---

## 能干什么

第一版如果做成，至少可以支持：
- 唤醒后问问题，Hermes 直接回答
- 通过 Hermes 调本地工具/联网能力完成任务
- 用自然语音触发家居动作（如果后续接 Home Assistant）
- 做个人桌面语音助手
- 做家里的“语音入口”，后面再逐步叠加自动化、播报、提醒、查询

如果接入 Home Assistant，后面还可以继续扩展：
- 控灯、空调、电视、窗帘
- 主动播报通知
- 场景联动
- 多房间部署
- 结合 Hermes 的工具链做“能听会做”的 agent

核心价值不是“像智能音箱”，而是：

> **它可以从“语音问答器”升级成“能调工具、能执行业务流程的语音 agent”。**

## 怎么装 / 怎么做

这里分成两条：**MVP 路线** 和 **Home Assistant 正式路线**。

### 路线 A：MVP 最短路径（推荐先走）

目标：先跑通完整闭环，不先追求最优声学体验。

#### Phase 1：先不做唤醒词
先把这条链跑通：

1. 录音
2. STT（Whisper）
3. 把文本送给 Hermes
4. Hermes 返回文本
5. TTS（Piper）播出来

这一步的验收标准：
- 能说一句话
- Hermes 能正确回复
- 本地能播报回复

#### Phase 2：再加唤醒词
- 常驻监听
- 命中唤醒词后开始录音
- 问题结束后提交给 Hermes

#### Phase 3：再加短时会话态
- 唤醒后进入 10~20 秒短会话窗口
- 期间可连续追问
- 超时自动退出待机态

只要做到这里，就已经很像真正的 AI 音箱。

### 路线 B：Home Assistant 正式路线

如果目标是未来多房间/家居联动/长期基础设施，建议上 HA。

典型结构：

```text
麦克风/麦阵
  -> 唤醒词（openWakeWord）
  -> STT（Whisper）
  -> 桥接层 / HA Assist Pipeline
  -> Hermes
  -> TTS（Piper）
  -> 音箱播放
```

HA 在这里负责：
- 设备管理
- 语音基础设施
- 自动化和场景联动

Hermes 负责：
- 对话智能
- 工具调用
- session / 任务执行

### 安装判断

#### 如果走 MVP 路线
重点不是某个大一统安装包，而是把这几部分装起来：
- `Hermes`
- `Whisper`（本地 STT）
- `Piper`（本地 TTS）
- 唤醒词组件（如 `openWakeWord`）
- 一个轻量桥接服务

#### 如果走 Home Assistant 路线
重点安装：
- `Home Assistant`
- `openWakeWord`
- `Whisper`
- `Piper`
- `Assist` / `Wyoming` 相关组件
- 再通过桥接把文本请求送到 Hermes

关键原则：

> **先打通单点闭环，再谈房间级体验。**

## 工程建议

### 1. 不要让 Home Assistant 反客为主
建议边界：
- HA = 语音基础设施 + 家居自动化
- Hermes = 智能处理 / 工具调用 / 多轮任务

### 2. 做一个薄桥接层
桥接层只负责：
- `session_id`
- 唤醒态 / 超时
- 调 Hermes API / CLI
- 把返回值交给 TTS
- 基本日志与重试

不要把逻辑散在多个脚本里。

### 3. 第一版先用普通 USB 麦克风
远场拾音、回声消除、阵列麦、打断式语音这些都是真问题，但不应该第一天就全上。

先把：
- 输入
- 理解
- 输出

这条主链做通。

---

## 推荐执行顺序

### 当前拍板版本
1. 先用小主机做单点闭环
2. 先用普通 USB 麦克风 + 本地音箱
3. 先跑通 STT -> Hermes -> TTS
4. 再加唤醒词
5. 再加短时连续对话
6. 再考虑是否引入 Home Assistant 做长期设施

### 不推荐的顺序
- 一上来先折腾小爱音箱改造
- 一上来就做多房间
- 一上来就追求全双工自然插话

这样只会把问题变复杂。

---

## 最终建议

一句话拍板：

> **你真正要做的不是“改小爱”，而是“做一个能接 Hermes 的本地语音终端”。**

对当前条件来说，**小主机路线最值**。

### 第一版最推荐配置
- 小主机
- USB 麦克风
- 本地音箱
- Whisper
- Piper
- Hermes
- 薄桥接层

### 第二版再升级
- 阵列麦
- 唤醒词
- 多轮短会话
- Home Assistant 纳管
- 多设备/多房间

> **能做，而且值得做；但应该沿着可控语音终端这条线做，不该把小爱音箱本体改造当主线。**
