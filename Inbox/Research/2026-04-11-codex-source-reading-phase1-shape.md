# 2026-04-11 Codex 源码拆解 Phase 1 产物形态与执行骨架

## 结论

Codex 源码拆解项目建议明确分成两个阶段：

- **Phase 1：源码拆解 / 原材料生产**
- **Phase 2：导读手册 / 面向工程读者的解释层重组**

当前阶段只做 **Phase 1**。

也就是说，当前不追求“好读”，而是优先建立一套高密度、可扩展、可继续累积的 **源码证据库**。

## Phase 1 的目标

这一阶段的目标不是“做公开导读成品”，而是先持续提取有用物料。

允许的特征：

- 很细
- 很技术
- 大量路径、函数、模块名
- 不一定对普通读者友好
- 更像研究笔记 / 拆解卡片，而不是最终文章

一句话说：

> 先把源码里的有用理解挖出来，存成结构化原材料。

## Phase 2 的目标

等 Phase 1 物料足够多之后，再做第二层工作：

- 重新组织
- 去重
- 抽主线
- 做导读层
- 做阅读路径
- 做术语解释
- 让没读过源码但有工程经验的人，也能快速理解 Codex 的设计

也就是说，Phase 2 更接近此前 Claude Code 源码导读项目的最终产物形态。

## 当前最适合的 Phase 1 产物形态

### 1. 模块级拆解笔记

这是核心产物。

每篇笔记聚焦一个问题，例如：

- Codex CLI 总入口怎么分流？
- `codex-cli/` 为什么只是分发壳？
- `codex-rs/core` 在系统里处在什么位置？
- `tui` 和 `core` 的边界是什么？
- `state` 怎么存、怎么恢复？
- `sandboxing` 是怎么分平台实现的？
- `MCP` 怎么接进 runtime？
- `sdk` 为什么是 spawn CLI + JSONL event？

每篇建议固定结构：

1. 这篇看什么
2. 先给结论
3. 关键代码路径
4. 主调用链 / 关键对象
5. 设计判断
6. 还没确认的点
7. 延伸阅读

---

### 2. 代码路径索引卡

这是非常高杠杆的辅助材料。

可以做成按问题分组的索引，例如：

- 入口层索引
- 配置层索引
- 状态层索引
- 沙箱层索引
- MCP 层索引
- TUI 层索引

每张卡只做一件事：

> 把“看这个问题该去哪几个文件”列清楚。

这类材料对后续继续拆解、以及喂给 agent 都特别有价值。

---

### 3. 概念 / 术语草表

不是最终公开 glossary，而是给 Phase 1 自己降噪。

例如先收口这些词：

- Codex CLI
- TUI
- exec
- core
- state
- config
- app-server
- MCP
- sandbox
- skills
- hooks
- thread / turn / session
- responses-api-proxy
- review
- cloud tasks

目的不是写漂亮，而是：

> 后续笔记不必每次重新解释同一个词。

---

### 4. 主链调用图草稿

不需要一开始就画漂亮图，但至少应该有文本版主链。

例如：

- `codex` 启动主链
- interactive TUI 主链
- `exec` 非交互主链
- config 加载主链
- session / state 恢复主链

哪怕先是伪代码、层级缩进、调用步骤，也很有价值。

---

### 5. 边界判断笔记

每拆一块，都最好顺手写一句：

> 这个模块在系统里负责什么，不负责什么。

因为源码阅读里最容易糊掉的，不是代码细节，而是模块边界。

例如：

- `codex-cli/` 是分发壳，不是主体
- `codex-rs/cli` 是入口层，不是全部核心
- `core` 是大脑，但不是 UI
- `sdk/typescript` 是 embedding layer，不是 runtime 主体

这些判断，后面会直接变成手册层的重要骨架。

## 第一批最值得拆的模块

建议不要平均用力，而是先抓系统骨架层。

### 第一组：系统骨架

1. 仓库总貌与分层判断
2. `codex-cli/` 分发壳
3. `codex-rs/cli` 总入口
4. `codex-rs/core` 在系统中的定位
5. `codex-rs/tui` 在系统中的定位

目标：

> 先回答“Codex 到底是什么形态的系统”。

### 第二组：运行主线

7. `exec` 非交互主线

### 第三组：能力边界

10. sandboxing
11. MCP
12. skills / hooks
13. SDK / app-server

目标：

## Phase 1 的执行原则

### 1. 先追求正确和可扩展，不追求好读

先保证：

- 结论正确
- 代码路径清楚
- 模块边界清楚
- 后续可继续扩写

### 2. 每篇笔记必须带证据

至少带：

- 目录路径
- crate 名
- 文件路径
- 关键类型 / 函数名
- 自己的结构判断

### 3. 允许重复，但要控制重复类型

Phase 1 的重复可以接受，但应该尽量是：

- 不同角度复用同一份证据

而不是：

- 重复抄一遍相同描述

## 当前拍板

Codex 这个项目的 **Phase 1 不是“先写书”**，而是：

> **先建一套源码证据库。**

最重要的三件事是先打稳：

1. **分层结构**
2. **主链调用**
3. **模块边界**

只要这三件打稳，Phase 2 才有稳定材料可重组。

一句话收口：

> 现在先做“源码拆解原材料生产线”，不要过早切到“面向大众的导读成品”。
