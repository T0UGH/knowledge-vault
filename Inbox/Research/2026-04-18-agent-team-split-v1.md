# Agent 团队分工 v1（MT / Rook / Leo / Nova / Kite / Mira）

## 结论

团队先按三组闭环拆分：

1. **MT + Nova**：主控 / 拍板 + 项目推进副驾
2. **Leo + Mira**：源码研究 / 技术写作 + 知识运营 / Obsidian 治理
3. **Rook + Kite**：signals / AI 情报 + 内容创作 / 小红书抖音发布

这版不是平均分工，而是按当前真实瓶颈来拆：
- MT 过载，需要拆推进
- Rook / Leo 定位已清晰，不应大改
- 新增角色要顺着机器共居关系形成最小摩擦闭环

## 为什么这么拆

### 1. MT 身上的问题不是能力不够，而是 owner 过载
MT 之前同时挂了太多线：
- agentgitmail 开发
- agentpixel 看板开发
- Obsidian 知识库维护
- videoclaw 项目开发
- 小红书 / 微信公众号创作

判断：
- MT 应继续做总控、拍板、最终收口
- 不应继续重度承担多个项目的日常推进与体力型维护

### 2. Rook 和 Leo 的主线已经成立

#### Rook
当前主线：
- signals 系统
- 多信息源 AI 情报采集
- 提炼成日报

判断：
- Rook 是情报官 / 观察哨 / 日报生产线 owner
- 当前问题是质量还不够稳，不是方向错了
- 这条线非常重要，应提质，不应改道

#### Leo
当前主线：
- 多个源码解读项目
- Claude Code / Codex 源码解读书
- Hermes 源码解读中

判断：
- Leo 是源码研究员 / 技术作者
- 负责长期研究型内容资产
- 不应被拉去做高频杂务或日常调度

### 3. 新角色必须补真正的组织缺口
当前最缺的不是“再多几个通用 agent”，而是三个长期 owner：
- 替 MT 扛推进的人
- 替团队扛内容节奏的人
- 把成果变成知识资产的人

## 最终分工

## MT
**一句话定位**：总控 / 架构判断 / 最终收口

### own
- 团队方向、边界、优先级
- 多项目资源分配与最终拍板
- 知识体系主权与标准
- 关键项目的验收与收口

### 不该继续重度 own
- 每个项目的日常推进
- 高频内容生产体力活
- 大量知识库维护细活

### 上下游关系
- 下游：Nova（推进）、Mira（知识运营）、Kite（内容执行）
- 上游输入：Rook（情报）、Leo（研究）

## Nova
**一句话定位**：MT 副驾 / 项目推进官

### own
- 从 MT 身上拆出来的项目推进
- 实验验证 / 候选方案探索
- 状态维护 / handoff
- 把散乱事项推进到“可收口”阶段

### 不该碰
- 替 MT 做最终拍板
- 长期吞掉知识库主权

### 上下游关系
- 上游：MT 的方向、边界、验收标准
- 下游：项目状态、验证结果、handoff 包

## Leo
**一句话定位**：源码研究员 / 技术作者

### own
- 源码解读
- 深度研究
- 技术长文 / 解读书
- 长期技术学习资产

### 不该碰
- 高频信息流运营
- 项目推进杂务 / 救火型调度

### 上下游关系
- 向 MT 提供深度研究产物
- 向 Mira 提供待沉淀的研究材料

## Mira
**一句话定位**：知识运营官

### own
- Obsidian 知识库日常运营
- 研究 / 项目资料归档
- 索引页、状态页、知识结构治理
- 把 Leo / MT / Rook / Nova 的产物收成知识资产

### 不该碰
- 抢 MT 的知识体系主权
- 退化成单纯“记笔记秘书”

### 上下游关系
- 上游：Leo / Rook / Nova / MT 的产物
- 下游：结构化知识库、索引、可复用资料

## Rook
**一句话定位**：情报官 / signals owner

### own
- AI 信息流采集
- 日报原料生产
- 信息筛选 / 聚类 / 初步判断
- 趋势观察

### 不该碰
- 大量深度源码研究
- 重内容发布节奏 owner 位

### 上下游关系
- 上游：多个信息源
- 下游：Kite（内容转译）、MT（情报输入）

## Kite
**一句话定位**：内容创作与发布官

### own
- 小红书 / 抖音内容创作
- 选题转译
- 发布编排
- 内容节奏与复盘

### 不该碰
- 纯情报采集 owner
- 复杂多项目推进总控

### 上下游关系
- 上游：Rook 的 signals 原料、MT 的方向要求
- 下游：平台内容产出与发布

## 三条机器闭环

### 1. MT + Nova
**闭环名**：决策与推进闭环

作用：
- MT 定方向
- Nova 往前拱项目
- 减轻 MT owner 过载

### 2. Leo + Mira
**闭环名**：研究与沉淀闭环

作用：
- Leo 产出深度研究
- Mira 把研究收成知识资产
- 避免“研究做完就散”

### 3. Rook + Kite
**闭环名**：情报与传播闭环

作用：
- Rook 产出信息原材料
- Kite 转译成适合平台发布的内容
- 避免内容线没有稳定素材源

## 关键边界

### 知识库边界
- **MT** 负责知识体系主权与标准
- **Mira** 负责日常运营与治理
- 结论：知识库不能完全从 MT 手里剥离，但也不能继续让 MT 亲自做大量维护动作

### 项目推进边界
- **Nova** 负责把 MT 手里的若干项目拆解、推进、跟踪
- **MT** 只保留方向、拍板、验收

### 内容边界
- **Rook** 负责情报与原料
- **Kite** 负责外发内容与节奏
- 不让 Rook 同时承担完整内容运营链条

### 研究边界
- **Leo** 负责深度研究
- **Mira** 负责整理成知识资产
- 不让 Leo 被高频杂务切碎

## 当前版本的组织判断

### 好处
- 顺着机器共居关系设计，执行摩擦低
- MT 从“什么都做”退回到“定方向 + 做拍板”
- 每组都有相对自然的上下游闭环
- Rook / Leo 的既有定位不被破坏

### 风险
1. **Nova 可能继续吸收过多杂务**
   - 要防止它变成新的垃圾桶 owner
2. **Mira 可能被误用成书记员**
   - 必须强调她是知识运营，不是纯记录
3. **Kite 可能缺稳定选题标准**
   - 需要让 Rook 持续供给原料，并由 MT 定内容边界
4. **MT 可能忍不住重新接回推进细活**
   - 必须主动把项目执行权放给 Nova

## 最小执行顺序

### Step 1：确认 owner 口径
验收标准：
- 六个人都有一句话定位
- 六个人都有“不该碰的边界”

### Step 2：把 MT 身上的项目列清单
验收标准：
- 列出 MT 当前所有在推项目
- 明确哪些转给 Nova 跟进

### Step 3：建立三组配对协作规则
验收标准：
- MT↔Nova、Leo↔Mira、Rook↔Kite 各有输入/输出边界

### Step 4：让 Mira 接手知识运营动作
验收标准：
- 至少有一轮真实归档 / 索引 / 状态页维护完成
- MT 不再亲自处理大部分知识库体力活

### Step 5：让 Kite 接手内容线
验收标准：
- 至少完成一轮从 Rook 原料到平台内容发布的完整闭环

### Step 6：观察两周，再做 v2 调整
验收标准：
- 识别谁 overload，谁 role 太轻，谁边界模糊

## 最终拍板

推荐方向：
- **MT + Nova / Leo + Mira / Rook + Kite** 三组闭环
- MT 保留主控与知识体系主权
- Nova 拆推进
- Mira 做知识运营
- Kite 做内容转译与发布

不推荐方向：
- 让 MT 继续兼任主控、推进、内容、知识库维护
- 让 Mira 只做轻量笔记整理
- 让 Kite 在没有稳定素材源的情况下独立空转

如果只先做 20%，最该先做的是：
1. 把 MT 手上的项目推进职责拆给 Nova
2. 把知识库日常运营交给 Mira
3. 让 Rook -> Kite 跑通一条真实内容闭环
