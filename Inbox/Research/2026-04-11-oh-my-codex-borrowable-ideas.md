# 2026-04-11 oh-my-codex 可借鉴思路

## 结论

`oh-my-codex` 最值得借鉴的，不是它那套完整 runtime，而是它背后的 **AI 工程工作流设计**。

对贵平来说，更合适的方向不是直接全盘采用 OMX，而是抽出其中真正高杠杆的部分，做成一套更轻、更贴合现有 Codex 使用方式的协作骨架。

## 最值得借的 5 个点

### 1. 把 Codex 使用流程显式分层

它最核心的价值，不是多加了几个命令，而是把 AI 协作流程拆成了几层：

- 先澄清需求
- 再收计划
- 再执行
- 最后验收

这点非常对。
很多 AI 协作效果差，不是模型差，而是 **澄清、计划、执行、验收全糊在一起**。

对贵平的启发：
后续自己的 Codex 工作流，也应该显式分成：

1. clarify / boundary
2. plan / tradeoff
3. execute
4. verify / accept

而不是让 Codex 一上来就直接写。

---

### 2. 长任务状态必须落盘

OMX 用 `.omx/` 存 plan、logs、memory、mode state。

这背后的原则很值得借：

> 长任务如果不落盘，恢复成本一定高。

真正有价值的不是 `.omx/` 这个目录名，而是：

- 当前目标是什么
- 最近做到哪了
- 哪些任务在执行中
- 哪些已完成
- 哪些有 blocker
- 验收标准是什么

对贵平的启发：
可以做一个比 `.omx/` 更轻的版本，但 **状态落盘** 这件事一定要保留。

---

### 3. 多 agent 的关键不是并发，而是协调

OMX 的 team mode 真正有意思的地方，不是“多开几个 pane”，而是：

- leader / worker 分工明确
- worker 有 inbox / task state
- leader 有统一可见状态
- 整个过程可恢复、可继续、可收尾

这点非常重要。

多 agent 的核心问题从来不是“能不能并发”，而是：

- 怎么分工
- 怎么交接
- 怎么避免互相覆盖
- 怎么让 leader 判断整体是否完成

对贵平的启发：
后续做多 agent 工作流时，重点不该放在“再多开一个 agent”，而应该放在 **任务边界、状态协议、验收归口**。

---

### 4. 把高频流程做成稳定入口

OMX 把一些高频流程固化成固定触发入口，例如：

- 澄清类入口
- 计划类入口
- 并行执行入口
- 持续推进入口

这件事的价值在于：

> 减少每次临场重新组织流程的认知开销。

对贵平的启发：
后续自己使用 Codex，也可以收敛出少量高频入口，而不是每次靠自然语言重新描述工作方式。

不是要学它的命令名，而是要学它的 **工作流入口固定化**。

---

### 5. 把验证当成系统级约束，而不是口头提醒

OMX 明显把 verification 看得很重。
这和贵平当前主线非常一致：

> no verification, no done

真正值得借的不是它某个 skill，而是这个设计倾向：

- 执行不是完成
- 验证需要单独占有一层
- 多 agent 体系里，验收不应混在执行线程里糊过去

对贵平的启发：
如果后续要收自己的 Codex 工作流，**执行层** 和 **验收层** 应该明确分开。

## 不建议直接照搬的部分

### 1. 不建议直接照搬整套运行时

OMX 太重，侵入性太强。
它会接管：

- `.codex/config.toml`
- `.codex/hooks.json`
- `AGENTS.md`
- `.omx/`
- tmux / hooks / MCP / HUD / state runtime

这对已经有自己协作规则和长期记忆体系的人来说，成本很高。

### 2. 不建议先学它的术语和命令

`$deep-interview` / `$ralplan` / `$ralph` / `$team` 这些命名只是外壳。
真正应该学的是背后的流程设计，而不是优先记住术语。

### 3. 不建议为了“多 agent”而多 agent

OMX 的 team mode 有价值，但如果没有明确分工协议，单纯多开 worker 只会放大噪音。

## 适合贵平的借鉴方向

更合适的不是“引入 oh-my-codex”，而是沉淀一套 **轻量 Codex 工作流骨架**。

可以优先只保留这几件事：

1. **任务分层**：clarify → plan → execute → verify
2. **轻量状态交接文件**：记录目标、当前状态、下一步、验收标准
3. **多 agent 时的 leader / worker 协议**
4. **固定的高频工作流入口**
5. **显式验收层**

## 当前拍板

- 值得借鉴：**工作流分层、状态落盘、多 agent 协议、固定入口、验证优先**
- 不值得直接照搬：**整套 runtime、整套命令体系、整套侵入式接管**

一句话收口：

> `oh-my-codex` 值得学它的“工程思想”，不一定值得直接学它的“产品形态”。
