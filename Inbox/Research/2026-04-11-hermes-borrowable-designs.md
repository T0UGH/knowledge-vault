# 2026-04-11 Hermes 最值得借鉴的设计

## 结论

Hermes 最值得借鉴的，不是它整套“全家桶”产品形态，而是它背后的 **AI 协作系统工程化思路**。

对贵平来说，真正高杠杆的不是照搬 Hermes，而是抽出其中最关键的设计原则，放进自己当前已经逐步成形的工作流里。

## 最值得借的 5 个设计取向

### 1. 把“知道什么”和“怎么做”分开

Hermes 在设计上把两类长期信息拆开了：

- **memory**：记稳定事实、偏好、约定、环境信息
- **skills**：记方法、流程、SOP、可复用工作流

这点非常对。

很多 agent 系统失败，不是没有记忆，而是把：

- 用户偏好
- 工具约定
- 环境事实
- 任务 SOP
- 错误经验

全都糊进一个 memory，最后越积越脏。

对贵平的启发：
后续应继续坚持这个分层：

- 偏好 / 稳定事实 / 环境约定 → memory
- 可复用方法 / 流程 / playbook → skill / 模板 / SOP

这会直接降低长期协作中的混乱度。

---

### 2. 工具是主干，模型是调度器

Hermes 的设计重心明显不是 prompt 花活，而是：

- 工具边界清楚
- 工具可组合
- 模型负责路由、调度、推进

这背后的思路很值得借：

> 高质量 agent，不该主要靠“提示词更会说”，而该靠结构化工具和稳定工件。

对贵平的启发：
后续构建自己的工作流时，优先级应该是：

1. 先把工具边界、输入输出、状态更新定义清楚
2. 再决定 prompt 怎么写

也就是说：**先结构，后语言。**

---

### 3. 长期协作必须有持久化层

Hermes 很重视这些持久化对象：

- sessions
- memory
- skills
- cron
- logs
- config

背后的原则非常扎实：

> 只靠当前聊天上下文，长期协作一定不稳。

这和贵平当前强调的方向完全一致。

对贵平的启发：
需要继续强化：

- 会话记录可检索
- 关键规则可落盘
- 长任务状态可恢复
- 经验和踩坑点可沉淀

不是为了“看起来更系统”，而是为了 **降低恢复成本和重复沟通成本**。

---

### 4. 重要规则要逐步从“语言提醒”变成“结构化约束”

Hermes 的一个很强的工程取向是：

它尽量不让重要规则只停留在 prompt 或口头提醒层，而是下沉到：

- config
- toolsets
- slash commands
- skills
- session store
- cron jobs

这点特别值得借。

因为真正重要、重复发生的规则，如果永远停留在“请记住”，最终一定会被忘。

对贵平的启发：
后续凡是高频重复的重要规则，都应该持续往结构层推进：

- 先写进规则文件
- 再收成模板 / skill / 固定入口
- 再尽量做成自动检查或固定流程

这条线，本质上和当前贵平主推的“规则机械化”完全一致。

---

### 5. 核心逻辑和平台入口分离

Hermes 不是只做 CLI，而是一个核心挂多个入口：

- CLI
- Feishu
- Telegram
- Discord
- Webhook
- Cron

这说明它不是把“平台”当成本体，而是把平台当成输入输出通道。

这点很值钱。

因为它逼着系统设计去区分：

- 核心协作逻辑
- 平台接入层
- 消息格式层
- 状态持久化层

对贵平的启发：
后续在做多入口 / 多 agent / 多平台协作时，应继续坚持：

> 核心逻辑不要和具体聊天平台强耦合。

## 当前最值得贵平直接吸收的部分

如果只按优先级排，最值得借的是：

1. **memory / skill 分层**
2. **规则结构化，不只停留在 prompt**
3. **长任务持久化与恢复**
4. **工具优先，模型负责调度**
5. **核心逻辑与平台入口解耦**

## 当前不建议直接照搬的部分

虽然 Hermes 很完整，但这些部分不适合直接照搬：

- 全量 gateway 体系
- 复杂 profile 系统
- 太重的 runtime
- 过多的内建命令面

原因不是它们不好，而是当前阶段贵平更需要的是 **高杠杆抽象**，不是更重的系统负担。

## 当前拍板

- 值得借：**工程化方向**
- 不值得照搬：**整套产品形态**

一句话收口：

> Hermes 最值得学的，是“把 AI 协作系统工程化”的方向，而不是把 Hermes 本身完整搬过来。
