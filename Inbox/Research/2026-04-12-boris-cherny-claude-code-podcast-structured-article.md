# Boris Cherny 对谈整理：Claude Code、一年增长与“编程被解决之后”

- 原视频：<https://www.youtube.com/watch?v=We7BZVKbCVw>
- 整理方式：基于现有中文字幕转录，按话题重组为文章；尽量保持原意，不额外加入评论。

## 开场：Boris 现在怎么工作

Boris 在开场先描述了自己现在的工作状态：

- 100% 的代码都由 Claude Code 完成
- 从去年 11 月开始，一行代码都没有亲手改过
- 每天仍然会提交很多 PR
- 录节目时，还有多个 agent 同时在跑

当主持人问他会不会怀念亲手写代码时，Boris 说不会。相反，他现在比以前更享受编程，因为不用再处理那些琐碎细节。在他的感受里，工程师的生产力已经显著提升。

## 他们提出的问题：如果编程被解决了，下一步是什么

主持人很快把问题推到更大的层面：

- 未来还要不要学编程
- 如果每个人都能编程，软件构建会变成什么样
- 软件行业接下来的重大变化会是什么

Boris 的回答是，Claude Code 已经开始不只是“写代码”。它会去看用户反馈、bug 报告和遥测数据，寻找应该修的 bug，也会提出可以发布的东西。它正在变得更像一个同事。

主持人接着把话题推向岗位变化。他说，听节目的人里有很多产品经理，他们可能已经开始紧张了；到今年年底，可能人人都会写代码，人人都会是产品经理，而“软件工程师”这个头衔会逐渐被“builder”取代。

## Boris 为什么离开 Anthropic，又为什么很快回来

主持人问到一个很多人都知道但没听过完整故事的话题：Boris 曾短暂离开 Anthropic 去了 Cursor，但两周后又回到 Anthropic。

Boris 说，他加入 Cursor，是因为一直很喜欢这个产品，也很佩服那个团队，觉得他们很早就看到了 AI 编程会走向哪里。

但去了之后，他很快意识到，自己真正怀念的是 Anthropic 的使命感。对他来说，Anthropic 最大的吸引力一直是“安全”这件事。那里的人谈起为什么加入公司，答案通常都和安全有关。他说，这种使命驱动感对他个人很重要，也是他获得幸福感的一部分。再酷的产品工作，也替代不了这一点，所以他很快就意识到，自己缺少的就是这个，于是回到了 Anthropic。

## Claude Code 这一年：增长速度比数字本身更惊人

主持人接着把话题转回 Claude Code 一周年。他提到一些外部信号：

- 有报告称，GitHub 上已经有相当比例的提交来自 Claude Code
- 越来越多资深工程师公开表示，自己已经不再亲手写代码
- 包括 Boris 在内，一些顶尖工程师都在用 AI 完成大部分编码工作

Boris 说，对他来说最疯狂的甚至不是某个静态数字，而是增长速度本身：Claude Code 不只是在增长，而是在加速增长。按他的说法，如果去看私有仓库里的使用情况，这个比例可能还会更高。

## Claude Code 最初是怎么做出来的

Boris 说，Claude Code 一开始只是一个小 hack。

Anthropic 内部一直知道自己想做某种编码产品，但更大的路线其实是：

1. 先让模型非常擅长编码
2. 再让模型非常擅长使用工具
3. 再让模型非常擅长使用电脑

他提到，后来做出 Claude Code、MCP 和桌面应用的那支团队，本质上都在沿着这条路线工作。

对 Anthropic 来说，这条路线和安全有关。因为 AI 越来越强、能力越来越大，它不再只是一个会聊天、会写代码的模型，而是开始会使用工具、会在现实世界中行动。Boris 认为，这种能力也会逐渐扩展到非技术用户。

他回忆说，自己刚加入 Anthropic 时，先花了一个月到处 hack 原型，试各种奇怪的东西，目的是摸清模型能力边界；后来又花了一个月做后训练，去理解研究层面的问题。他说，如果想把工作做好，就必须理解自己所构建系统下面的那一层；做 AI 产品尤其如此。

## 第一个原型：ClaudeCLI 与“它自己会用工具”的震撼

后来，Boris 开始为 Claude Code 的前身做原型。当时它还叫 ClaudeCLI。

他做过一个 demo，其中最让他震惊的一次，是给了模型一个 batch 工具后，它居然自己写代码，回答出了“我现在在听什么音乐”这个问题。

Boris 说，关键不在这个问题本身，而在于他并没有告诉模型该怎么做，也没有规定必须用哪个工具、按什么步骤做。模型只是拿到了工具，就自己想明白了如何用它来解决问题。

这件事让他继续往前做原型。但他第一次在内部发这个东西时，只收到了两个赞。按他的说法，当时 Anthropic 内部很多人一想到编码工具，想到的都是 IDE，很少有人会认为终端会成为合适的形态。

## 为什么最早选择终端

Boris 说，Claude Code 一开始做成终端形态，并不是一个先验的产品哲学，更现实的原因是：

- 最初几个月几乎只有他一个人在做
- 终端是最容易实现的方式
- 模型进步太快，其他形态未必跟得上

他后来回头看，认为这也是一个重要的产品经验：早期应该稍微少配一点资源，先用最直接的方式把东西做出来。

后来团队也想过别的产品形态，但最后还是先坚持终端，原因就是它最能跟上模型的进展。

## 发布后的变化：内部先爆，再慢慢被外部理解

Boris 说，Claude Code 在 Anthropic 内部很快爆了，日活用户几乎是垂直起飞。内部有人甚至很早就提醒他把 DAU 图画出来，而那条图线几乎立刻冲了上去。

但对外发布时，情况并不是一夜之间全面爆发。它确实很快吸引到一批早期用户，但花了几个月，大家才慢慢理解这到底是什么。

按 Boris 的说法，Claude Code 很不一样，而它之所以能成功，部分原因在于一种“潜在需求”。

## Boris 提到的“潜在需求”

Boris 解释，“潜在需求”指的是：如果一个产品能以某种方式被用户“hack”一下，或者被以一种原本并未设计好的方式使用，但却能帮助用户完成他们真正想做的事，那么这种使用方式就会暴露出产品下一步应该去哪里。

他举的例子是 Facebook Marketplace。它最初的观察是：某个时期 Facebook 群组里有很大一部分帖子都在买卖东西。也就是说，人们在“借用”群组这个产品做交易。产品本来不是为这个目的设计的，但既然用户已经自己摸索出这种用法，就说明这里存在强需求，应该做一个更好的正式产品来承接。

Boris 说，Claude Code 早期也有点类似：它进入的是工程师已经在的地方，让现有工作流变得更容易一点；同时因为它在终端里，又带着一点意外和异质感，所以用户需要保持开放心态，去学会如何使用它。

## 现在的 Claude Code 已经不只在终端里

虽然起点在终端，但 Boris 也提到，Claude Code 现在已经进入很多不同入口：

- iOS 和 Android 应用
- 桌面应用
- 网页
- IDE 插件
- Slack
- GitHub

他反复强调的一点是，应该把产品带到用户已经在的地方，而不是强迫他们切换到完全不同的工作流。

对不是工程师的用户来说，桌面应用和其他界面可能更自然。虽然外面看起来是不同入口，但背后仍然是同一个 Claude Code agent。

## Boris 对当前阶段的判断：写代码已经差不多被解决了

Boris 在节目中多次回到一个判断：对他做的那类编程来说，写代码已经基本是一个被解决的问题了，因为 Claude Code 已经能做到。

所以在他看来，接下来的问题不是继续停留在“写代码”本身，而是往它相邻的任务扩展。

他举了一些自己日常使用 co-work 的例子：

- 交停车罚单
- 做团队项目管理
- 在不同电子表格之间同步内容
- 发 Slack 消息
- 发邮件

按他的说法，这些事已经和写代码无关，而更像是让 agent 去完成一般性的电脑工作。他认为，前沿正在朝这个方向移动，而不再只是停留在写代码。

## 他们如何看待“不会某门语言，也能把服务做出来”

主持人提到，自己职业早期花了很多时间在库、依赖和各种细节上，现在则更像是把问题交给 agent，让它一步一步把事情搞定。

Boris 接着讲了一个例子：有位工程师写了一个 Go 服务，做了差不多一个月，最后服务运行得不错。但当他问对方写这个过程感觉如何时，对方说，自己其实到现在还是不太懂 Go。

Boris 说，他觉得这种情况会越来越常见。只要你知道系统运行正确、效率也可以，就未必需要掌握所有细节。

## 谁会是下一个被明显改变的角色

当主持人问接下来最会被 AI 影响的是哪些角色时，Boris 的回答是：很多与工程相邻的岗位。

他点到的包括：

- 产品经理
- 设计师
- 数据科学

他的判断是，只要一项工作主要在电脑上完成，模型都会越来越擅长它。

## co-work 是怎么做出来的

节目后面聊到 co-work 的产品实现时，Boris 提到，团队最后找到的一个有效方案，是直接把 Claude Code 放进桌面应用里。

按他的说法，这个方向一旦确定下来，团队在大约 10 天里就用 Claude Code 把它做了出来。

他提到 co-work 内置了一个比较复杂的安全系统，里面有很多护栏机制，目的是保证模型大体会做正确的事，不容易跑偏。产品里还带了完整虚拟机。Claude Code 写出了这套系统的大量代码，而团队主要在思考的是，怎样让它对非工程师更安全、更容易引导。

## 多 agent、焦虑与新的工作状态

主持人还提到一个现象：很多和 agent 一起工作的人，会在 agent 卡住时感到焦虑，觉得好像有很多生产力正在白白流失，必须马上起来把它重新推起来。

节目在这一段继续展开了多 agent 协作、不同工作方式和实际使用习惯，但整体方向仍然是：随着 agent 真正开始工作，人和工具之间的关系也在变化，新的工作节奏和新的管理方式正在形成。

## Boris 给用户的几个使用建议

节目靠后，Boris 也给了一些更偏实践的建议：

- 使用规划模式，先和模型来回讨论计划，等计划看起来没问题再执行
- 在合适的时候开启自动接受编辑
- 多尝试不同界面，不一定非得用终端

他特别强调的一点是：

- 不同工程师不一样
- 不同产品人也不一样
- 关键是找到最适合自己的那个入口和工作方式
- 不管你用哪个入口，背后运行的都是同一个 Claude agent

## 收尾问题：书、科幻和常识

在节目最后的快问快答里，Boris 提到自己很爱读书，也推荐了《Functional Programming in Scala》这本技术书。

他还提到自己喜欢科幻，因为科幻能让人提前为未来可能的发展方式做好思想准备。

最后，当被问到一个经常回到的生活或工作信条时，他给出的答案很简单：**运用常识。**

他说，自己看到的很多失败，尤其在工作环境里，都是因为人们没有使用常识：他们可能照着流程走，却不去思考；也可能是在惯性中推进一个并不好的产品或主意，而没有认真停下来想一想。

## 一页版整理

把这期对谈压缩成一页，大致就是这些点：

- Boris 现在几乎不再亲手写代码，Claude Code 已经成为他的主要工作方式
- 他认为写代码这件事对自己这类编程来说，已经基本被解决
- Claude Code 的意义不只是生成代码，而是开始使用工具、读取信号、推动下一步工作
- Anthropic 内部做 Claude Code 的路线，是编码 → 工具使用 → 电脑使用
- 终端是最早的载体，不是因为它最漂亮，而是因为它最能跟上模型迭代
- 产品成功的一条关键经验，是识别并顺着“潜在需求”前进
- AI 的影响会继续从工程扩展到 PM、设计、数据科学等更广泛的电脑工作
- 对 Boris 来说，使命感、安全以及常识，仍然是他做选择和做判断的重要基点
