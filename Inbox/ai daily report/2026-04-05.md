### OpenClaw
- **OpenClaw 这一轮不只是连续发版，CHANGELOG 还补进了一批真正有结构性意义的能力更新，重点集中在记忆系统、上下文可见性、审批流和 provider 面。**
- 全部变更点：
  - 4 月 1 日先后发布 `2026.4.1-beta.1` 与正式版 `2026.4.1`，4 月 2 日继续发布 `2026.4.2`，说明版本推进节奏很快。
  - `CHANGELOG.md` 新增了 **Breaking** 说明：移除多组 legacy public config alias，包括 `talk.voiceId` / `talk.apiKey`、`agents.*.sandbox.perSession`、`browser.ssrfPolicy.allowPrivateNetwork`、`hooks.internal.handlers` 以及 channel/group/room 的 `allow` 开关，统一迁移到 canonical public paths 与 `enabled` 语义；现有配置仍保留 load-time 兼容，并提供 `openclaw doctor --fix` 迁移支持。
  - 实验性 **memory/dreaming** 能力增强：加入 weighted short-term recall promotion、`off|core|rem|deep` 四档 dreaming mode、`/dreaming` 命令、Dreams UI、多语言概念标签，以及 doctor/status repair 支持，目标是把 durable memory promotion 做成更少人工介入的后台流程。
  - **渠道上下文可见性** 增加 `contextVisibility` 配置，可按 channel 控制为 `all`、`allowlist`、`allowlist_quote`，使 quote/thread/fetched history 这类补充上下文可以按发送者 allowlist 过滤，而不是原样透传。
  - **Matrix exec approvals** 增加 Matrix 原生审批提示，支持 account-scoped approvers、channel/DM 投递和 room-thread 感知的处理方式。
  - 新增 **StepFun provider plugin**，同时覆盖 standard 与 Step Plan endpoints；`step-3.5-flash` 在两个 catalog 都可用，`step-3.5-flash-2603` 当前出现在 Step Plan。
  - provider/config 面继续收敛：`models.providers.*.request` 增加针对 model-provider path 的 headers / auth 覆盖，media provider HTTP path 也得到完整 request transport override。
  - 新增 **MiniMax TTS provider**，基于 T2A v2 API，可直接接入 MiniMax 原生语音与鉴权。
  - Skills 面板直接接入 **ClawHub** 的 search / detail / install 流程。
  - README 更新本身不是产品功能变化，主要是 sponsor 区块布局调整，并新增 GitHub sponsor 位，因此信息价值低于 CHANGELOG。

## GitHub 趋势项目

- **`claude-howto`（#1，本周 +14996 stars）**：这是一个围绕 Claude Code 的可视化、示例驱动型教程仓库，不是单点脚本合集，而是从基础概念一路覆盖到 agents、hooks、skills、MCP servers 的学习路径。README 里明确把自己定位成“不是 feature reference，而是结构化上手路线”，并给出 10 个 tutorial modules、Mermaid 图示、copy-paste 模板和 15 分钟起步路径。对想系统化使用 Claude Code 而不是只会“打一条 prompt”的用户，这类仓库会继续吃到需求。

- **`oh-my-codex`（#2，本周 +10578 stars）**：定位是 OpenAI Codex CLI 的 workflow layer。它把 Codex 作为执行引擎，在外层补上默认启动、工作流编排、project guidance 和持久状态管理，核心命令包括 `$deep-interview`、`$ralplan`、`$team`、`$ralph`。README 还单独放了 OpenClaw integration 文档，说明这条“给 coding agent 再套一层流程框架”的方向已经开始跨代理生态流动，而不是只局限在单一宿主里。

- **`hermes-agent`（#7，本周 +9193 stars）**：这类项目值得继续看，不只是因为又一个“全栈 agent”，而是因为 README 直接把卖点压在几个现在最热的结构上：自我改进技能、周期性记忆 nudges、跨会话搜索、cron 调度、多平台 gateway、subagent 并行、以及跑在 VPS / serverless 上的持久环境。它把“agent 不再只活在本机终端里”这件事讲得很完整，和 OpenClaw 当前在 cron、memory、multi-channel 上的主线高度同频。

- **`onyx`（#9，本周 +4006 stars）**：Onyx 不是单纯的聊天壳，而是把 RAG、deep research、web search、custom agents、artifacts、actions & MCP、code execution、voice mode、image generation 全塞进一个可自托管的平台层。值得跟踪的点在于它把“应用层 LLM 平台”做成了开源、可部署、可接多种模型/连接器的组合，这会继续吸走一部分企业内知识助手与 agent 平台需求。

- **`superpowers`（#12，本周 +16229 stars）**：这不是新仓库，但继续冲上趋势榜说明“技能框架 + 方法论”依然很受欢迎。README 的核心不是模型能力本身，而是强约束的软件开发流程：先澄清需求、再分块 spec、再做 TDD / YAGNI / DRY 风格的实现计划，最后进入 subagent-driven-development。它在 Claude、Codex、OpenCode、Gemini 等多端都给了安装路径，说明“跨宿主迁移同一套 agent 方法论”已经成为独立需求。

## Product Hunt 新品观察

- **ZooClaw**：在 Product Hunt 上票数 372，定位是“一个入口接入一组 AI specialists”。文案里直接写了 built on OpenClaw，强调自然语言分发到不同 specialist、结构化 domain knowledge、自然语音风格，以及可回退到顶级开源模型。它更像是把“多代理协作”包装成面向普通用户的托管产品，而不是让用户自己去搭 agent 框架。

- **Cursor 3**：票数 356，产品描述是“Unified workspace for parallel local/cloud agents and MCPs”。虽然 signal 本身给出的细节不多，但这个定位足够说明它在卖“本地 agent + 云 agent + MCP”统一工作区，而不是单一 IDE 补全；和当前 coding agent 市场从“IDE 内嵌一个模型”继续转向“编排多个执行体”的方向一致。

- **Qwen3.6-Plus**：票数 140，卖点是 1M context、面向 real-world coding agents 的 agentic coding 提升、更强多模态推理，以及对 OpenClaw、Claude Code、Qwen Code 等开发工作流的贴合。和一般模型发布不同，这条 signal 明确把“coding agent 兼容性”当成卖点写进标题与简介里，说明模型厂商已经开始反向针对 agent 使用场景打产品语言。

- **OpenRouter Model Fusion**：票数 111，OpenRouter Labs 的公开实验：把一个 prompt 同时丢给多模型，再用可配置的 judge model 做融合。这个方向对需要高质量最终回答、又不想自己手工写多模型投票逻辑的开发者有吸引力，尤其适合 agent 平台把“多模型并跑 + 结果裁决”做成上层能力。

- **OpenGyver**：票数 75，定位是把文件/单位/天气/电影/音频转换等杂活能力变成 CLI 与 agent 可调用工具，还特别强调可以用 OpenGyver skill 让 Claude Code 直接调用。它不是大型 agent 平台，但抓住了一个非常实用的空位：给 coding agent 补齐大量日常工具链能力，减少用户为各种碎片网站来回跳转。

- **订阅制本身的脆弱性也在被公开讨论**：`@thdxr` 说得更直接——大额 LLM 订阅计划在数学上很脆，用户行为稍有变化就会把问题放大；从长期市场角度看，这类大包月如果逐步退场，反而可能让市场价格与使用方式更稳定。这条可以看作上面 Anthropic 政策变化的补充视角。

- **代码知识图谱正在被包装成 agent 的结构化外脑**：`@heygurisingh` 介绍 GitNexus，核心能力包括 Tree-sitter AST 解析、函数调用/导入/继承关系图、cohesion-based 功能聚类、入口到调用链的执行流追踪、blast radius analysis、跨文件符号重命名，以及自动生成 codebase wiki。重点不是“又一个文档工具”，而是把这些结果通过 MCP 喂给 Claude Code、Cursor、Windsurf，让 agent 在修改代码前先拿到完整依赖结构。

- **自我优化 agent 仍在持续出圈**：`@servasyy_ai` 提到 AutoAgent 把 meta-agent 改 harness、自动生成验证循环与格式校验器、以及 24 小时轨迹积累放进同一个闭环里，并给出 SpreadsheetBench 96.5%、TerminalBench 55.1% 的成绩。这里最值得记的不是分数本身，而是“评估闭环 + 架构约束 + 记忆治理”开始被当成可落地工程结构，而不是抽象研究口号。

- **Karpathy 的长帖虽然不是 coding agent 发布，但仍然击中了这条主线的上游基础设施**：他讨论的是用 AI 反向提升政府信息的可见性、可理解性和问责能力，关键瓶颈不再是数据是否公开，而是能不能处理海量原始材料并抽出洞察。这和 agent / knowledge workflow 的主线是连着的：只要能把复杂、公开、但不可读的大体量资料变成可操作结构，很多以前只有专业人士能做的工作都会被重新分配。
