## GitHub 长期跟进

### Claude Code
- **这次不只是常规补丁滚动。2.1.91-2.1.92 同时把企业管控、Bedrock 接入、成本可见性、远程控制，以及一批终端和工具链细节往前推了一截。**
- 全部变更点：
  - 新增 `forceRemoteSettingsRefresh` 策略：开启后，CLI 启动时必须先成功拉到最新远程托管设置，否则直接 fail-closed 退出。这是一次比较硬的企业治理能力升级。
  - 登录页新增第三方平台的 **Bedrock 交互式配置向导**，覆盖 AWS 认证、region、凭证校验、模型 pinning，说明 Bedrock 路径正在从“能配”往“普通用户也更容易跑起来”推进。
  - `/cost` 对订阅用户新增 **按模型和 cache-hit 拆分**的成本视图，成本归因粒度比之前更细。
  - `/release-notes` 现在变成交互式版本选择器，不再只是单次展示当前版本说明。
  - Remote Control 会话名默认改用主机名作为前缀，例如 `myhost-graceful-unicorn`，并支持 `--remote-control-session-name-prefix` 覆盖，远程多机会话的辨识度更高。
  - Pro 用户在 prompt cache 失效后回到旧会话时，会看到下一轮大概要发送多少 **未命中缓存 token** 的提示，真实成本和上下文体积的关系更透明了。
  - 修掉一个会让 subagent 在长会话中“永久起不来”的问题：tmux 窗口被杀掉或重编号后，之前可能报 `Could not determine pane count`。
  - 修复 Stop hooks 在小模型返回 `ok:false` 时被错误判失败的问题，并恢复非 Stop prompt-type hooks 的 `preventContinuation:true` 语义。
  - 修复流式输出里数组/对象字段以 JSON 字符串形式出现时，工具输入校验失败的问题。
  - 修复 extended thinking 过程中“空白文本块 + 真内容”混在一起时可能触发的 API 400。
  - 修复全屏处理时有文本选区仍出现误导性的 `esc to interrupt` 提示；修复 `ctrl+e` 多行输入时跳到下一行结尾的问题；修复 iTerm2 / Ghostty 等终端中同一消息滚动时重复显示的问题。
  - Homebrew 更新提示改为使用正确 cask channel：`claude-code` 对应 stable，`claude-code@latest` 对应 latest。
  - 大文件 Write tool diff 计算提速，官方写法是对包含 tab / `&` / `$` 的文件可快约 60%。
  - 移除 `/tag` 命令。
  - 移除 `/vim` 命令，vim mode 改由 `/config` → Editor mode 切换。
  - Linux sandbox 在 npm/native 构建里重新带上 `apply-seccomp` helper，恢复对 sandboxed 命令的 unix-socket 阻断。
  - 2.1.91 同批还加了几个更偏平台/插件层的点：
    - `_meta["anthropic/maxResultSizeChars"]` 支持把 MCP 工具结果持久化上限提到 500K，数据库 schema 一类的大结果更容易完整透传。
    - 新增 `disableSkillShellExecution`，可以禁掉 skills、自定义 slash commands、插件命令中的内联 shell 执行。
    - `claude-cli://open?q=` deep links 支持多行 prompt。
    - 插件现在可以把可执行文件放在 `bin/` 下并从 Bash tool 直接调用。
    - `/feedback` 不再“消失”，而是明确解释为什么当前不可用。

- **OpenClaw 这一轮仍然是高密度平台推进：一边继续扩模型、传输层和技能安装能力，一边把 Matrix / Telegram / Discord / Slack / 移动端 / 配对 / 安全 fail-close 这些底层边角补得更完整。**

  - 新增 channel 级 `contextVisibility`，可配置 `all`、`allowlist`、`allowlist_quote`，补上引用、线程和抓取到的历史上下文不再只能“原样全传”，而是可以按发送者 allowlist 过滤。

  - 新增 bundled **Ollama Web Search provider**，允许在本地 Ollama host + `ollama signin` 环境下做 key-free 的 `web_search`。
  - OpenAI 原生路径修了几类容易踩坑的问题：

## X 推荐流

- **Karpathy 在讲一种“LLM 驱动的个人知识库编译流”**：原始资料（文章、论文、repo、dataset、图片）先进 `raw/`，再让 LLM 持续把这些东西“编译”成一套 markdown wiki，自动生成 summaries、backlinks、分类文章和链接结构；前端用 Obsidian 看 raw data、compiled wiki、派生可视化；查询层不走复杂 RAG，而是让 agent 直接在大约 100 篇文章、40 万字量级的 wiki 里研究问题、写 markdown / Marp 幻灯片 / matplotlib 图，再把输出回写知识库。这里最值得记的，不是“又一个 Obsidian 工作流”，而是他明确说现在大量 token 不再用来改代码，而是用来“操作知识”，并且已经在用 health checks 查不一致、补缺失数据、找新文章候选、开发 CLI 小工具给 LLM 调用。这条帖子本质上是在给“知识库即工程对象”做方法论背书。
- **Mintlify 的 ChromaFs 方案提供了 AI 文档助手的新检索思路**：他们没有继续堆向量 RAG，而是给 AI 伪造了一套文件系统，让模型以为自己在用 `grep` / `cat` / `ls` / `find` 查文档，底层其实把命令翻译成数据库查询。好处很具体：启动时间从 46 秒降到 100 毫秒，边际计算成本接近零；每个页面当文件、每个章节当目录，精确字符串匹配和结构遍历都比向量召回更稳；grep 通过“先元数据粗筛、再批量预取、最后内存中精确匹配”避免网络 IO 扫描过慢；权限控制则是在初始化时直接裁剪文件树，让 AI 从路径层面就看不到没权限内容。对做内部文档助手或结构化知识库的人来说，这比泛泛讨论 RAG 更有可操作性。
- **opencli 发布了 `operate` 能力，开始把“浏览器操作”直接拉进 CLI 链路**：这次更新的关键信息，不是又多一个网页自动化脚本，而是“装上 skill 之后，可以用 prompt 直接操作浏览器，然后无痛沉淀为 CLI”。帖子给的例子是 Claude Code 直接执行“opencli 帮我打开某页面然后填写申请”。如果这个链路稳定，意味着网页交互能力正在从浏览器自动化工具，变成 agent 侧可复用的 CLI 抽象。
- **Claude Code + Obsidian memory 正在变成一种可识别的社区模式**：推荐流前两条高曝光帖分别是 `tom_doerr` 的“Obsidian vault for Claude Code memory”和 `sourfraser` 的“full Claude + Obsidian system”。虽然这两条本身信息量不如 Karpathy 那条长帖，但它们和后者一起说明，社区已经不只是零散存笔记，而是在把 Obsidian 当 agent memory / compiled knowledge base / 工作前端来用。
