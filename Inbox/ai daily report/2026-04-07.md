# AI 日报（2026-04-07）

## GitHub 趋势项目

- **Yeachan-Heo/oh-my-codex：给 OpenAI Codex CLI 叠一层完整工作流与团队运行时，已冲到本周趋势第 1。**
  这个项目不只是提示词合集，而是把 Codex 当执行引擎，在外面补上更强的默认启动方式、工作流命令和持久化状态。README 给出的主路径很清楚：先 `npm install -g @openai/codex oh-my-codex`，再 `omx setup`，之后用 `omx --madmax --high` 启动；真正的工作流则围绕 `$deep-interview`、`$ralplan`、`$ralph`、`$team` 这几个入口展开，分别对应澄清、规划、持续推进和并行执行。项目还把 plans、logs、memory、mode tracking 等状态统一写进 `.omx/`，并给了 `omx team`、`omx hud --watch`、`omx explore`、`omx sparkshell` 等运行时能力。对编码 agent 生态来说，它更像是在把通用 CLI agent 包装成一套可复制的团队协作和规范执行层。

- **luongnv89/claude-howto：Claude Code 教程型仓库爆发，本周星增甚至高于大多数工具仓库。**
  这不是新 CLI，而是一套面向 Claude Code 的系统化学习与模板仓库。当前 README 标明版本为 **2.2.0**，兼容 **Claude Code 2.1+**，内容覆盖 10 个模块：Slash Commands、Memory、Checkpoints、CLI Basics、Skills、Hooks、MCP、Subagents、Advanced Features、Plugins。它的价值不在于只解释概念，而是把 `.claude/commands/`、`CLAUDE.md`、hooks、MCP 配置、subagent 定义、插件安装等都做成可直接拷贝的模板，还内置 `/self-assessment` 与 `/lesson-quiz [topic]` 这样的学习路径。放到今天的 agent 生态里看，这类“工作流教材仓库”正在从周边资料变成高传播入口。

- **Yeachan-Heo/oh-my-claudecode：Claude Code 侧的多代理编排层继续高热，核心卖点已经从 swarm 迁移到 team。**
  README 明确写到从 **v4.1.7** 起，**Team** 是规范化的编排入口，旧的 `swarm` 已移除；标准流程是 `team-plan → team-prd → team-exec → team-verify → team-fix (loop)`。项目提供两类运行面：一类是 Claude Code 内部的 `/team`、`/ccg`、`autopilot`、`ralph` 等；另一类是 CLI 侧的 `omc team N:codex|gemini|claude "..."`，直接拉起 tmux worker pane，让 Codex / Gemini / Claude 做混合编排。README 里还特别强调 **v4.4.0 移除了 Codex/Gemini MCP servers**，改走 CLI-first team runtime；同时集成 HUD、ask/provider advisor、wait 自动续跑、通知回调、OpenClaw gateway 等配套设施。它值得继续跟，因为不只是热度高，而是已经把“Claude Code + 外部模型 + tmux runtime + 通知回调”拼成了一个完整产品面。

- **NousResearch/hermes-agent：把“可学习、可跨平台、可自托管”的 agent 打包成一体化产品。**
  README 里最突出的差异化点，是它把 agent 的“长期使用”需求一次性补得很全：内建 skill 学习闭环，会从经验里生成和改进技能，也支持跨会话搜索和用户建模；同时可以跑在 CLI、Telegram、Discord、Slack、WhatsApp、Signal 等多个入口上，并通过单一 gateway 共享连续会话。运行环境也不绑死本机，支持 local、Docker、SSH、Daytona、Singularity、Modal 六类 terminal backend，定位很明确：让 agent 住在云端或 VPS，而不是必须守在用户笔记本上。命令面也直接给到 `hermes gateway`、`hermes setup`、`hermes claw migrate`、`hermes update`、`hermes doctor`，说明它已经不是纯研究 demo，而是面向迁移、部署、运维的一条成熟路线。

- **onyx-dot-app/onyx：开源 AI 平台路线继续升温，定位是“可自托管的 LLM 应用层”。**
  Onyx 把自己定义为 application layer for LLMs，核心能力已经比较完整：RAG、web search、code execution、file creation、deep research、custom agents、actions & MCP、voice mode、image generation，并强调对 Anthropic、OpenAI、Gemini、Ollama、LiteLLM、vLLM 等主流和自托管模型的兼容。README 还把部署拆成 **Lite** 和 **Standard**：Lite 面向低资源聊天与 agent 场景；Standard 才包含向量+关键词索引、worker/job queue、推理服务、Redis/MinIO 这类完整堆栈。对企业场景，它又补上 SSO、RBAC、SCIM、审计、PII 处理、白标等能力。这里的趋势信号是：AI chat / agent 平台已经不只是 demo UI，而是在往“企业内网 AI 应用底座”走。

- **google-ai-edge/gallery：端侧大模型试用和评测正在被做成标准化 app。**
  Google AI Edge Gallery 的方向很清楚：把“在手机本地跑模型”从开发者实验做成可下载 app。当前 README 主打 **Gemma 4** 官方支持，并把功能拆成 Agent Skills、Thinking Mode、Ask Image、Audio Scribe、Prompt Lab、Mobile Actions、Tiny Garden、Model Management & Benchmark 等多个模块。重点不只是在聊天，而是围绕本地模型运行、基准测试、技能扩展和隐私离线体验做了一整套沙盒。它同时上了 Google Play 和 App Store，还提供 APK release，说明端侧 GenAI 现在不再只是 SDK 或样例工程，而是已经进入“应用分发 + 模型管理 + benchmark 可视化”的阶段。

  @OpenAI 的帖子信息虽短，但关键词很清楚：这是一个新的 **Safety Fellowship**，支持 **independent research on AI safety and alignment**，同时面向 **the next generation of talent**。也就是说，这不是一次性的论文资助，而是把“独立研究 + 人才培养”结合起来，继续扩张它的安全研究外圈网络。

- **开发者对 Claude Code prompt cache 的经验开始细化到具体失效条件和规避手法。**
  @MinLiBuilds 这条很实用，信息颗粒度也足够细。帖中先强调：一旦匹配到缓存，**input token 可下降 90%**，并指出 Claude / LLM 的缓存是按 block 链式匹配。作者列出几类会导致缓存失效的具体操作：过程中改 `CLAUDE.MD`、切换模型、API 调用里输入精确时间戳、MCP 工具描述含随机路径。进一步的经验补充则包括：不要在过程中新增 MCP；尽量用多轮对话；用 `/btw` 共享缓存；一轮对话里连续做总结、改段落、写导读等多个动作；任务完成后 clear 或开新 session；以及尽量少用 compact，因为 compact 会打断缓存命中。对 Claude Code 重度用户来说，这些已经是会直接影响周额度和响应速度的操作层知识。

- **关于 Claude Max / 高额度账号风控，社区给出了一条更偏“使用行为模式”的解释。**
  @LawrenceW_Zen 的判断是，这次封号**大概率与 IP、环境、支付无关**，更像是短时间内额度消耗过快触发；作者自述的使用模式包括 **多项目、开多 subagent 并行跑、两天耗掉一半以上周额度**，并认为这种模式很像中转站的高负载使用特征。虽然这不是官方结论，但它比泛泛的“风控变严了”更具体，也给出了可操作的风险线索：并行度、短时耗额速度、项目切换密度，可能比常见的代理或支付猜测更接近触发条件。
