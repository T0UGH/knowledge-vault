### Claude Code
- **2.1.101 不只是一次热修，既补上了远程规划/远程会话链路，也集中修了一批恢复、权限、插件和长会话稳定性问题。**
- 全部变更点：
  - 新增 `/team-onboarding` 命令：可以基于你本地 Claude Code 的使用情况，自动生成给新同事的上手说明。
  - 默认信任操作系统 CA 证书库，企业内网里常见的 TLS 代理现在不需要额外配置就能工作；如果只想用内置证书，可切回 `CLAUDE_CODE_CERT_STORE=bundled`。
  - `/ultraplan` 和其他 remote-session 功能现在会自动创建默认云环境，不再要求用户先去网页端手动完成环境准备。
  - brief mode 增强：如果 Claude 返回了纯文本而不是结构化消息，会自动重试一次。
  - focus mode 增强：模型知道用户只会看到最终回复后，会生成更完整、可独立阅读的总结。
  - tool-not-available 报错更可用：不仅提示工具不可用，还会解释原因和后续该怎么做。
  - 限流提示更具体：会直接说明撞到的是哪一类 limit、何时重置，而不是只给模糊的秒数倒计时。
  - refusal 错误信息现在会带上 API 返回的解释文本。
  - `claude -p --resume <name>` 现在可以接受 `/rename` 或 `--name` 设定的 session 标题。
  - `settings.json` 中如果出现未识别 hook event，以前会导致整份配置失效；现在只忽略坏项，不会把整份设置吃掉。

### OpenClaw
- **2026.4.9 是一版偏平台层的实质更新：一边继续推进 dreaming / 控制台 / provider auth / QA 能力，一边集中补安全、路由、网关聊天面和移动端配对稳定性。**
- 全部变更点：
  - Memory / dreaming 新增 grounded REM backfill lane，可通过历史 `rem-harness --path` 回灌；还加入 diary commit/reset 流程、更干净的 durable fact 抽取，以及 live short-term promotion 集成，让旧 daily notes 能回放进 Dreams 和 durable memory，而不是再造第二套记忆栈。
  - Control UI / dreaming 新增结构化 diary 视图，支持 timeline 导航、backfill/reset 控件、可追溯的 dreaming summary，以及 grounded Scene lane、promotion hints 和安全的 clear-grounded 操作。
  - QA / lab 新增 character-vibes evaluation reports，支持模型选择和并行跑，方便 live QA 对比候选行为。
  - Plugins / provider-auth 新增 `providerAuthAliases`，让不同 provider 变体共享 env vars、auth profiles、config-backed auth 和 API-key onboarding，而不用在 core 里逐个接线。
  - iOS 发布流程改成显式 CalVer，版本号固定在 `apps/ios/version.json`；TestFlight 同一短版本可持续迭代，直到维护者主动晋升到下一 gateway 版本，并新增 `pnpm ios:version:pin -- --from-gateway` 工作流。
  - 浏览器安全补强：点击、evaluate、hook-triggered click、批量 action 等触发的主 frame 跳转后，会重新执行 blocked-destination 检查，避免用交互绕过 SSRF quarantine。
  - `.env` 安全补强：阻止不受信任 workspace `.env` 注入 runtime-control、browser-control override、skip-server 等危险环境变量；同时拒绝不安全的 URL 风格 browser control override。
  - remote node exec 事件现在会被标成 untrusted system events，并对节点返回的 command / output / reason 做净化，避免远程节点借输出把伪造的 `System:` 内容注入后续对话。
  - 插件 onboarding auth 选择逻辑更严：不受信任 workspace 插件不能在非交互 onboarding 时与内置 provider auth-choice id 冲突，减少 secrets 被不可信 handler 接走的风险。
  - dependency audit：强制 `basic-ftp` 升到 `5.2.1` 解决 CRLF 命令注入，并提升 Hono 与 `@hono/node-server`。
  - Android pairing 修复一整组扫码恢复问题：新 QR 扫描会清理旧 setup-code auth、operator 与 node session 改为从新配对引导、bootstrap 后优先使用存储的 device token、App 后台时暂停 pairing auto-retry。
  - Matrix / gateway 稳定性增强：等待 sync ready 后才把启动标记为成功；后台 handler 失败被局部吸收；fatal Matrix sync stop 改走 channel 级重启，而不是直接把整个 gateway 打挂。
  - Slack 媒体下载恢复：保留同源 `files.slack.com` 重定向中的 bearer auth，同时跨源 CDN 跳转仍剥离凭证，`url_private_download` 图片附件又能加载。
  - Reply / doctor 路径修复：queued reply run 使用活跃 runtime snapshot；reply-run SecretRefs 在 preflight helper 前解析；gateway OAuth reauth 失败可见；`openclaw doctor` 会直接指出重认证命令。
  - Control UI 修复快速切 session 时旧 history reload 覆盖当前 session 的问题。
  - Gateway/chat 不再向用户面泄漏 `ANNOUNCE_SKIP` / `REPLY_SKIP` 控制 token。
  - Auto-reply / `NO_REPLY` 现在会剥离胶着在最前面的 silent sentinel，避免用户看到不该暴露的 `NO_REPLY` 字样。
  - Sessions/routing 保留既有 external routes，`sessions_send` 跟进消息不再抢走 Telegram、Discord 等渠道的投递。
  - `/reset` 和 `/new` 现在会清掉 auto-fallback pin 的 model override，但继续保留用户显式选定的模型。
  - Slack ACP block reply 被视为已交付输出，避免 OpenClaw 在 Slack 已渲染后又补发 fallback 文本。
  - Slack partial streaming 通过 dispatch kind 做 turn-local dedupe，并在 preview finalization 失败时保留 final fallback 路径，避免旧 preview 压掉真正最终答案。
  - `openclaw doctor --fix` 会把 legacy Matrix DM policy `trusted` 迁回兼容配置，并保留 `allowFrom` 边界。
  - npm 打包流程会镜像 bundle channel runtime deps、stage Nostr deps，并在无 repo `node_modules` 场景测试 tarball，尽早暴露缺失依赖。
  - live QA 遇到 classified auth 或 runtime failure reply 会 fail fast，不再表现成模糊 timeout。
  - OpenAI provider 默认把缺失的 reasoning effort 设成 `high`；Ollama 在原生 `api: "ollama"` 路径下也可在 `/think` 开启时显示 thinking output。
  - Codex CLI 集成补上 `model_instructions_file` 覆盖，把 OpenClaw system prompt 传进 Codex CLI 新会话。
  - auth profiles 显式 upsert 直接持久化，跳过外部 CLI 同步，减少凭证状态陈旧。
  - agent timeout 行为调整：LLM idle timeout 可继承 `agents.defaults.timeoutSeconds`；cron run 若未配置则禁用 idle watchdog；报错指向 `agents.defaults.llm.idleTimeoutSeconds`。
  - failover 分类更准：Z.ai vendor `1311` 归 billing、`1113` 归 auth，含换行包裹的 1311 文本也能识别。
  - Plugin SDK / contracts / control UI models 等外围能力也有补强，包括公开 channel plugin base、阻止 test-only helper 混入生产 contract barrel，以及保留 OpenRouter 带斜杠模型 id 的 provider-qualified refs。

## Product Hunt 新品观察

- **Offsite**：把“人类成员 + agent 成员”放进同一个团队编排层里，卖点不在单个 agent，而在 live org chart、实时协作可视化、动作审批和全流程可见性。它明确写到可直接接 Claude Code、OpenClaw 和任意 MCP-compatible agent，定位已经从“agent 容器”走向“团队操作系统”。对做多 agent 工作流的人来说，值得看的是它把 agent 从 tab/terminal 里抬出来，变成可组织、可监督、可审批的协作对象。

- **Grass**：核心是给 coding agent 一个 24/7 常驻的专属 VM，不吃本机环境，也不占用户笔记本。它支持把 Claude Code 或 OpenCode 直接指过去运行，用户可在手机上看进度、半途干预、推送代码。和传统远程开发机不同，它卖的是“给 agent 用的云工位”，重点在持续在线、轻接入和移动端监工。

- **AgentMail**：给 AI agent 分配独立邮箱收发能力，而不是复用人的邮箱，或给每个 inbox 单独走一遍 OAuth。它把 email thread、reply、attachment extraction、semantic search、webhooks / websockets 全打包成 agent 基础设施，更像“邮件渠道层 API”，适合需要让 agent 独立处理外部往来的产品。

- **Claude Managed Agents**：Anthropic 自己把 agent 基建打包成可组合 API，对外提供 sandboxing、authentication、tool execution、long-running sessions、multi-agent coordination。重点不在新模型，而是把原本团队要自己搭的 agent harness 和托管运行时收成官方托管层，主张可把上线速度提到 10x。今天 Product Hunt 上虽然票数不算最高，但对 agent infra 赛道是个明确信号。

- **Integrations in Spine**：Spine 不只是网页研究代理，而是把“跨多个应用取数 + 联网调研 + 定时运行 + 产出落到 Notion / Google Docs / Sheets”等交付链路连起来。它强调可以描述一个项目，让 agent 去工具里拉上下文，再生成 report、doc、spreadsheet、presentation，并按日/周调度，明显更偏“研究型工作流自动化”。

## X 推荐流

- 围绕 **Claude Code 的“记忆层”外挂** 的讨论还在升温。@axiaisacat 在一条高互动帖里把卖点直接打成“永久记忆”“免费”“token 消耗减少 95%”“不再撞上下文限制”“自动记录决策并在下次续上”，说明市场正在把 Claude Code 的会话连续性问题打包成独立产品或工作流能力来卖。另一条来自 @leopardracer 的帖子则把路径讲得更具体：借鉴 Andrej Karpathy 的 personal wiki 思路，把 Claude Code 接到 Obsidian，试图把 goals / context / history 变成“真正认识你”的 second brain。两条合起来看，不只是“记忆很重要”，而是记忆外挂正在从概念讨论转成具体接线方案和高传播教程。

- **Claude Code 的后台监控 / 长任务观察能力** 开始进入更广泛的认知。@berryxia 提到官方推出 Monitor 工具，重点不只是“能监控”，而是 agent 可自己创建后台监控脚本：持续盯日志、发现错误就唤醒 agent；自动轮询 PR，出现新评论或修改就通知；不再需要用户自己反复轮询，直接减少 token 消耗。这类能力和今天 GitHub changelog 里的远程规划、长会话修复放在一起看，说明 Claude Code 正在把“长任务异步推进”补成一条更完整的链路。

- **对 API 中转站篡改 tool call 的警惕在上升。** @grapeot 转述 UCSB 研究者的实测结果：买了 428 个 LLM API 中转站后，发现 9 个会偷偷改 tool call 返回值，1 个甚至直接转走私钥里的 ETH。这个信号值得纳入日报，不是因为互动量高，而是它直接打到 agent 工作流里最脆弱的一层——工具调用回包。如果 tool result 能被中间人改写，那上层 agent 的权限、自动化和钱包/密钥类操作都会被放大成现实风险。

## X 关注流

- **Claude Code 的 `/ultraplan` 正式进入预览。** @trq212 给出的官方说法是：在终端里发起规划任务后，Claude 会到网页端生成实现方案，用户可在浏览器里阅读、编辑，再决定是在云端执行还是拉回终端执行。@dotey 的中文拆解把这个功能的真实使用形态说得更清楚：它解决的是“复杂实施计划在 terminal 里难审阅”的问题，网页端支持按段批注、跳大纲、加反馈，再让 Claude 反复改方案；同时云端分析代码库时终端不会被占住。补充信息包括：对开启 Claude Code Web 的用户开放预览，需要 v2.1.91+，token 消耗与本地 plan 模式相近。这个功能把“规划”和“实现”进一步拆层，说明 Claude Code 正在把 terminal 内的 agent 体验外溢到 web 协作面板。

- **Claude for Word 进入 beta。** @claudeai 公布的能力点比较完整：在 Word 侧边栏里直接起草、编辑、改写文档；保留原有格式；修改会以 tracked changes 形式出现；目前面向 Team 和 Enterprise 套餐。它不是简单把聊天窗塞进 Office，而是把“文档编辑 + 变更追踪 + 企业订阅”绑成一条工作流，对正式文档场景比普通聊天式 copilot 更接近可落地形态。

- **Claude Managed Agents 已开始出现首批应用落地信号。** @danshipper 提到 Every 旗下第一个使用 Claude Managed Agents 的应用是 @TrySpiral。虽然这条信息本身不长，但和 Product Hunt 上的 Claude Managed Agents 产品页结合起来看，至少说明官方托管 agent 基建不只停留在 API 发布阶段，已经开始有具体应用接入。

- **语音模式的“可说话 ≠ 最强模型”仍是实际使用中的认知坑。** @simonw 提醒：OpenAI voice mode 跑在更老、更弱的模型上，“能对话的 AI 看起来理应最聪明，但实际并不是”。这条对 agent / workflow 主线的意义在于，很多产品会把语音入口误当成高能力入口，但在任务设计上，语音层和高能力推理层可能仍需分离。
