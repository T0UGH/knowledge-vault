# AI 日报（2026-04-06）

## GitHub 长期跟进

### anthropics/claude-code
本轮仍以 2.1.9x 小版本连续迭代为主，但 v2.1.92 里有几项是实打实影响日常使用和企业部署的更新，不只是例行发版噪声。

全部变更点：
- 新增 `forceRemoteSettingsRefresh` 策略：CLI 启动时可强制先刷新远程托管配置，拉取失败直接退出，明显偏向企业管控与 fail-closed 安全策略。
- 登录页新增交互式 Bedrock 配置向导：把 AWS 鉴权、区域、凭证校验、模型 pinning 放进引导流程，第三方平台接入门槛继续下降。
- `/cost` 对订阅用户新增按模型与 cache hit 的成本拆分，成本可视化更细。
- `/release-notes` 改成可交互版本选择器，查版本差异更方便。
- Remote Control 会话默认名称改为 `hostname-*` 前缀，便于多机环境识别。
- Prompt cache 过期后，返回会话时会提示下一轮大致会发送多少未命中缓存的 token，方便预估成本。
- 修复了多个长期运行/复杂会话下的稳定性问题：包括 tmux 窗口变化导致 subagent 永久失败、hook 语义异常、流式工具参数校验失败、extended thinking 触发 API 400、全屏滚动重复消息等。
- 移除 `/tag` 与 `/vim` 命令，后者改为通过 `/config` 切换 editor mode。

### openclaw/openclaw
OpenClaw 的重点不是单点功能，而是基础设施能力继续外扩：任务流、视频生成、缓存稳定性、Claude CLI 迁移、以及跨平台交付与安全修补都在推进。4.2 正式版之外，CHANGELOG 还显示主线在快速累积下一批能力。

全部变更点：
- 4.2 正式版恢复并强化 Task Flow 基座：增加 managed vs mirrored 同步模式、持久化 flow state/revision、`openclaw flows` 检查与恢复能力，以及受控 child task spawning/cancel intent，说明后台编排正在走向可运营化。
- 插件侧新增 `api.runtime.taskFlow` 接缝，允许受信任插件/作者层直接创建与驱动 managed Task Flow。
- 新增 `before_agent_reply` hook，可在 LLM 输出前由插件短路为合成回复；对自动化动作、内联工作流很关键。
- Android 侧增加 assistant-role 入口与 Google Assistant App Actions 元数据，移动端入口继续扩展。
- Feishu comments 新增独立 Drive comment-event flow，可解析评论线程上下文并做 thread 内回复，明显增强文档协作场景。
- Exec 默认策略改动明显：gateway/node host exec 倾向 `security=full` + `ask=off` 的 YOLO 默认；同时配套补了一系列 approval、policy、doctor、node-host 绑定与回退修复。
- 4.2 正式版引入内建 `video_generate` 工具与多家视频生成 provider；而主线 CHANGELOG 进一步显示已支持 xAI/Alibaba 等视频模型，并把视频生成纳入 task ledger 跟踪。
- 主线 CHANGELOG 继续推进 Memory/dreaming：从原先模式切换改为 light/deep/REM 三阶段协作，加入 aging 控制、REM preview、`DREAMS.md` 持久化等，说明长期记忆体系仍在重构。
- Providers 侧继续大扩张：新增/整合 Qwen、Fireworks、StepFun、Mantle/Bedrock、MiniMax TTS、Ollama/MiniMax Search 等，并补 request override、cache、compat 路由与安全边界。
- Prompt caching 被持续强化：去掉 system prompt 中重复工具清单、加入 cache diagnostics、稳定指纹与边界控制，目标是提高 follow-up 命中率。
- 安全修补密度很高：插件 allowlist、hook fail-closed、SSRF、设备配对、HTTP scope、exec env 覆写、marketplace 路径逃逸、工具结果泄漏清洗等均有新增防护。
- 对 Claude CLI 的态度发生了更明确的产品化迁移：一方面移除新 onboarding 中的旧 Claude CLI backend/setup-token 路径，另一方面又加强 OpenClaw 驱动 Claude CLI 时的桥接、安全隔离、session 绑定与 doctor 检查。

## GitHub 趋势项目

### luongnv89/claude-howto
本周最值得注意的不是“又一个 Claude Code 教程”，而是它把 Claude Code 从零散 feature reference，包装成一条可复制的学习路径：10 个模块、可直接复制的命令/技能/hook/MCP/subagent 模板、配套测验与 roadmap。对新进入 AI coding 工作流的人，这类“结构化上手包”比单篇技巧帖更容易放大 Claude Code 的渗透速度。周增星 1.47 万，说明需求非常强。

### Yeachan-Heo/oh-my-claudecode
这是今天最值得继续跟踪的 Claude Code 周边项目之一。它抓住的不是“写 prompt”，而是“团队式编排 + 多 CLI worker + tmux runtime”。项目已经把 Team 作为主编排表面，把 `omc team N:codex/gemini/claude`、`/ccg`、HUD、wait daemon、技能抽取、OpenClaw 网关联动等拼成一个完整使用层。周增星 9232，说明市场仍在追逐“给 Claude Code 装上更强 orchestrator”的方案。

### NousResearch/hermes-agent
Hermes Agent 值得看，不只是因为它也在做 agent OS，而是因为它把“自学习闭环”放到了产品中心：技能从经验中生成、使用时再改进、跨会话检索历史、持续形成用户模型，并且兼容多模型与多消息渠道。它同时提供 CLI、gateway、多平台聊天入口与迁移自 OpenClaw 的路径，明显瞄准长期陪伴型 agent。周增星 9566，表明“能学习的 agent”正成为新卖点。

### microsoft/VibeVoice
虽然不属于 AI coding 主线，但它在本周高位上榜，说明多角色/高表现力语音生成仍是热门基础能力方向。若后续和 agent、播客、长内容工作流结合，可能会进入更广的自动内容生产链路。

## Product Hunt 新品观察

### Mercury Edit 2
一个很对当前 AI coding 主线的产品：它不做通用聊天，而是专注“next-edit prediction for coding”。核心卖点是利用最近编辑记录和代码库上下文，给出下一步修改建议，并宣称比常规 code-edit 模型更快、接受率更高。它对应的是开发过程中的连续编辑流，而非单次问答流，值得继续观察是否能嵌进 Cursor / Claude Code / Codex 一类 IDE/CLI 工作流。

### Handle Extension
定位非常明确：直接在浏览器里点选并修 UI，然后把修改反馈给 coding agent，而不是反复重写 prompt。它抓住的是 agent 时代最烦人的一段链路——视觉问题描述成本高、改 UI 靠口述效率低。若产品做得顺，这类“浏览器前端操控层 + agent 回写”会很有机会成为 AI 前端开发标配。

### Panorama
Panorama 不是单点 agent，而是从团队工作数据里反推出“隐藏结构”和可自动化工作流。它更偏向 workflow mining + AI workflow recommendation，适合团队级效率产品赛道。对企业内部自动化来说，这比单个 AI 助手更接近预算购买逻辑。

### OpenGyver
OpenGyver 把大量琐碎 CLI 小能力——格式转换、JSON/HTML 处理、编码解码、天气/媒体等——包装成可管道化调用的 terminal 工具，并直接强调可作为 Claude Code skill 使用。虽然产品本身票数不算高，但它符合一个越来越清晰的方向：为 agent 补齐长尾“杂活工具箱”。

## X 推荐流

### Karpathy 工作流外溢仍在持续扩散
推荐流里最强的一条主线仍是 Karpathy 式知识工作流的外溢：从 YouMind 的 IPO（Input-Process-Output）产品化实现，到把 LLM + Obsidian 方法整理成“可照着做”的说明手册，说明“个人知识库 + agent 工作台”已经从少数技术用户扩散到更广人群。这个方向的看点不只是工具，而是非程序员也开始能直接消费这类工作流。

### 自我优化 Agent 从概念进入“拿榜单说话”阶段
`@servasyy_ai` 提到的 AutoAgent，把评估闭环、架构约束、记忆治理三件事一起做进系统，并且强调由 agent 自己迭代 harness、自动生成验证循环、累积轨迹，已经不是“理论上可行”，而是开始用 SpreadsheetBench、TerminalBench 成绩争夺话语权。对 AI coding 赛道，这意味着下一阶段竞争点会从“会不会调工具”转向“是否能持续自改进”。

### 围绕 Claude Code 的教程化、复刻化内容仍在放量
推荐流中大量内容都在复刻、讲解、包装 Claude Code/知识库/技能流：一类是把爆款工作流做成教程，一类是把 source leak、内部工作流、技能包再加工成普通用户能用的版本。这说明市场仍处于“高质量原始实践极少，但高传播二次包装极多”的阶段。

## X 关注流

### OpenClaw：视频生成能力已开始公开预热
`@steipete` 直接预告下个版本 OpenClaw 将带来原生视频生成，并点名 Alibaba、BytePlus、fal、Google、MiniMax、OpenAI、Qwen、Together、xAI 等 provider。这和 GitHub CHANGELOG 中 `video_generate`、provider 扩展的方向相互印证，说明视频生成正从底层支持走向对外可见功能。

### OpenCLI autoresearch：把“自动研究 + 自动改进”做成可跑一夜的闭环
`@jakevin7` 公开的 autoresearch 进展非常值得继续跟。其框架受 Karpathy autoresearch 启发，用“约束 + 机械指标 + 无限循环”做闭环：自动测试、调用 Claude Code 做单次原子修改、验证提升或回滚。当前在 Operate 命令和 Save-as-CLI 流水线上的通过率已经很高，意味着这条路线开始从概念演示走向工程化收益。

### 关于 system prompt 计费的讨论，反映平台层新摩擦点
`@simonw` 对“按 system prompt 内容差异计费”公开表达不满，这类讨论本质上触到 agent 平台的下一层问题：当工作流越来越依赖长 system prompt、技能、上下文注入时，平台方如何计费，会直接影响产品设计与用户策略。它不是功能新闻，但对整个 agent 生态是个需要盯住的规则变量。
