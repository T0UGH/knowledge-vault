# AI 日报（2026-04-03）

## GitHub 长期跟进

### anthropics/claude-code

- 发布 `v2.1.90`。这版不只是补丁，而是把新手教学入口、离线插件缓存、权限与编辑稳定性、长会话性能几条线一起往前推。
- 新增 `/powerup`：用交互式 lesson + 动画 demo 带用户上手 Claude Code 功能，说明官方开始把“能力可发现性”和“新用户教育”直接做进产品入口，而不再只靠文档。
- 新增环境变量 `CLAUDE_CODE_PLUGIN_KEEP_MARKETPLACE_ON_FAILURE`：当 `git pull` 失败时保留现有 marketplace 缓存，明显是在照顾离线或弱网场景下的插件可用性。
- `acceptEdits` 保护目录新增 `.husky`，对带 hooks 的项目更谨慎，减少 agent 误改工程自动化配置的风险。
- 修复 `--resume` 首次请求出现完整 prompt cache miss 的回归问题；受影响场景包括 deferred tools、MCP servers、自定义 agents，自 `v2.1.69` 起存在。
- 修复连续 `Edit` / `Write` 在格式化 hook 改写文件后触发 “File content has changed” 的失败场景，这对启用保存即格式化的团队项目很关键。
- 修复 `PreToolUse` hook 向 stdout 输出 JSON 且以 code 2 退出时，未能正确阻止工具调用的问题。
- 修复 auto mode 不尊重用户明确边界（如“不要 push”“先等 X 再做 Y”）的行为，这属于 agent 边界遵守层面的重要修正。
- 修复 rate-limit 选项对话框反复自动弹出并最终拖垮会话的无限循环问题。
- PowerShell 权限检查继续加固：修掉尾随 `&` 后台任务绕过、`-ErrorAction Break` 调试挂死、解压 TOCTOU、解析失败时 deny 规则退化等问题，Windows 侧安全面明显在收紧。
- 性能方面有三处值得关注：
  - MCP tool schema 的 cache-key 查找不再每轮都 `JSON.stringify`
  - SSE transport 处理大流式 frame 从平方级退化修到线性复杂度
  - 长会话 SDK transcript 写入不再随长度平方级变慢
- `/resume` 的 all-projects 视图改为并行加载项目 session，项目多的重度用户恢复速度会更明显改善。
- `--resume` picker 不再展示由 `claude -p` 或 SDK invocation 创建的会话，产品边界更清楚了。
- 从 auto-allow 中移除 `Get-DnsClientCache` 和 `ipconfig /displaydns`，对 DNS 缓存隐私做了更保守的处理。

### openclaw/openclaw

- 发布 `v2026.4.2`。这版信息量很大，主线集中在三块：Task Flow 基础设施回归、provider/network 安全边界继续收紧、exec/approval 与多渠道运行时打磨。
- Breaking 变更有两项，都是配置迁移型：
  - `x_search` 从旧 core 路径迁到 `plugins.entries.xai.config.xSearch.*`，认证统一到 `plugins.entries.xai.config.webSearch.apiKey` / `XAI_API_KEY`
  - Firecrawl `web_fetch` 配置迁到 `plugins.entries.firecrawl.config.webFetch.*`
  - 两者都建议用 `openclaw doctor --fix` 做迁移
- Task Flow 主体能力恢复：
  - 核心 Task Flow substrate 回归，支持 managed / mirrored sync mode、持久化 flow state / revision tracking
  - 新增 `openclaw flows` 检查与恢复能力，把后台编排从“临时运行状态”推进到“可运维对象”
  - 支持 managed child task spawning 与 sticky cancel intent，意味着上层 orchestrator 可以停派新任务，同时让已有子任务跑完后父 flow 再稳定落到 `cancelled`
  - 插件侧新增 `api.runtime.taskFlow`，插件和受信任编排层可直接驱动 Task Flow，不必每次传 owner 标识
- exec 默认行为更激进：gateway / node host exec 默认请求 `security=full` + `ask=off`，相当于把 host 执行默认推向 YOLO 模式；对应 doctor、审批文件 fallback、文档说明都做了对齐。
- plugins/hooks 新增 `before_agent_reply`，插件可在 inline actions 之后直接短路 LLM，返回 synthetic reply。这是个很实用的拦截缝隙，适合把“动作后模板回复”从 prompt 层搬到 runtime 层。
- Feishu 相关继续增强：
  - 新增 Drive comment-event flow，可做评论线程上下文解析、线程内回复、`feishu_drive` comment actions
  - 同时修补 whole-document comments 回退 `add_comment`、延迟 reply lookup 重试、避免用户可见回复里泄漏 reasoning/planning 残留
- session routing 改为 provider/plugin-owned session-key surfaces，Telegram topic routing 与 Feishu scoped inheritance 相关路径一起收束到插件边界。
- provider / transport 安全与一致性是本版最大修补面之一：
  - 统一 HTTP / stream / websocket 的 auth、proxy、TLS、header shaping
  - 阻止不安全的 TLS / runtime transport override
  - Anthropic、OpenAI-compatible、Copilot 的 native-vs-proxy endpoint classification 都收进共享路由逻辑，避免被伪装 host 继承官方默认行为
  - Gemini media/image、Deepgram audio、Moonshot video 也统一到共享 provider HTTP 传输路径
- exec / approvals 修复线同样很密：
  - 修复 loopback pairing 后本地 exec / node client 的 `pairing required`
  - `sessions_spawn` 改为 admin-only gateway call 走 `operator.admin`，避免 loopback scope-upgrade 配对下子 agent 挂掉
  - 清洗 `~/.openclaw/exec-approvals.json` 中非法 `security` / `ask` / `askFallback` 值
  - `tools.exec.host=auto` 明确只做 routing，不允许通过 per-call override 绕过既定 sandbox/host 目标
  - Node host 对 `pnpm dlx` 的审批绑定更严格，避免“脚本实际执行体”未被审批计划覆盖
  - 增加更多环境变量覆盖阻断，避免 package root、runtime、credential/config 路径被请求级 env 重定向
- browser / SSRF / 路径安全继续加固：
  - `page.goto()` 之前就安装 Playwright route handler，拦截跳转过程中打向私网 IP 的 SSRF redirect bypass
  - 浏览器 doctor / inspection 不再提前激活 browser runtime
  - `localhost.` 这类 trailing-dot 主机名在 CDP loopback 检测中做规范化
  - 图片工具相对路径改为相对 `workspaceDir` 解析，而不是 `process.cwd()`
- 渠道与消息层细修很多，和日常使用直接相关的有：
  - Slack mrkdwn 指引内置到 inbound context，减少 generic Markdown 渲染失真
  - Matrix block streaming、onboarding、mentions 全线修补
  - Telegram approvals、retry、topic followup 继续清理边角问题
  - WhatsApp connect 时发送 `unavailable`，避免自聊模式抢走手机推送

- 本周 GitHub Trending 第 4，周增星 `8991`。
- 项目定位非常明确：面向 Claude Code 的 teams-first 多 agent orchestration，核心卖点不是“多一堆 prompt 模板”，而是把 Team / tmux worker / 多模型协作包装成一套学习成本更低的工作流层。

## Product Hunt 新品观察

### Claude Code Voice Mode

- 主打“对 Claude Code 说话”：支持语音输入 prompt、即时听回复、在 voice / text 之间切换，还有 hands-free / push-to-talk 模式。
- 如果产品真能无缝接入现有 CLI 工作流，它补的是 coding agent 当前最弱的一环：在走路、切屏、看文档、临时 brainstorm 时的低摩擦输入。
- 但目前公开描述仍偏功能宣传，缺少更细的工作流细节，比如是否支持上下文控制、命令模式切换、终端反馈可视化等。

### Mngr

- 定位非常准：用 CLI 批量拉起上百个 Claude agents 并进行程序化编排。
- 核心卖点包括：
  - 同一套命令可跑 1 个或 100+ agents
  - 可组合 workflow，如批量修测试、批量开 PR、批量验证 use case
  - 中途可 attach 到某个 agent 会话里追问或 debug
  - agent 启动 <2 秒，空闲自动退出
  - 支持本地、Modal、Docker，且宣称与任何 agent harness 兼容
- 这类产品很契合当前“agent fleet management”需求，尤其适合测试矩阵、批量 issue 处理、回归检查等场景。

### Claudoscope

- 一个面向 Claude Code 的 macOS 菜单栏本地工具，主打 session history 浏览、全局搜索、按项目核算 token 成本。
- 更有意思的是两个附带能力：
  - secret detection：扫描 session 中泄漏的 API key / token / credential
  - config health linter：对 `CLAUDE.md`、skills、hooks 跑 19 条规则
- 它不是生成式入口，而是“Claude Code 周边 observability + hygiene 工具”，很贴近实际重度用户的管理需求。

### traceAI

- OTel-native 的 LLM tracing，目标不是新建一套 AI 专属平台，而是把 prompts、completions、tokens、retrievals、agent decisions 按 GenAI semantic conventions 发回现有 observability stack。
- 覆盖 Python / TypeScript / Java / C#，支持 35+ 框架，强调“两行代码接入”。
- 对已经有 Datadog、Grafana、Jaeger 的团队来说，这类“进入现有监控栈”的方案比新 SaaS 更容易被采纳。

### OpenBox

- 走的是 agent runtime governance 路线：监控、验证、治理每个 agent action，补的是企业侧对 agentic AI 的信任与合规需求。
- 关键词包括 cryptographic verification、enterprise compliance、LangChain / LangGraph / Temporal / n8n / Mastra SDK 集成。
- 相比 tracing，OpenBox 更偏“控制与审计层”，适合和编排框架配套观察。

## X 推荐流

### OpenCLI 开始把浏览器操作沉淀为 CLI

- `@jakevin7` 透露 OpenCLI 新增 `operate` 能力：装上 skill 之后，可以直接让 AI agent 操作浏览器，再把这个流程沉淀成一个 CLI。
- 配图示例是让 Claude Code 打开网页并填写申请表，这说明它不只是“浏览器自动化 demo”，而是把 browser-use workflow 抽象成可复用命令。
- 同帖还提到项目两周达到 `1w star + 100 contributor`，社区增速很快。
- 这条值得继续跟：如果“浏览器操作 → skill 化 → CLI 化”这条链路跑通，会明显降低 agent 自动化从一次性演示到可复用工具的门槛。

### Claude Code 实验性全屏渲染模式

- `@fkysly` 提到 Claude Code 新增实验性 fullscreen rendering，可通过 `CLAUDE_CODE_NO_FLICKER=1` 开启。
- 技术上转向类似 Vim / htop 的全屏终端渲染模式，优点是减少闪烁、稳定内存、支持鼠标操作。
- 代价是部分传统终端能力会变化：如 `cmd-f` 不再可用，原生复制粘贴被替换成鼠标选中文本自动复制。
- 支持 tmux、VS Code 内嵌终端，不支持 iTerm2 的 `tmux -CC` 集成模式。
- 这类底层交互改动对重度 CLI 用户影响很大，尤其是多终端、多复用器工作流。

### 技能生产方法论开始固化

- `@bozhou_ai` 用五步法总结 skill 生产流程：识别重复劳动、先手动跑通、把需求讲给 AI、跑起来并迭代、再分享给别人。
- 这是典型的“把一次性经验产品化”为 agent skill 的方法抽象，虽然很短，但很贴近实际团队落地。

### awesome-research 的自动更新链路

- `@yibie` 公开了 `awesome-research` 自动更新的工作流：
  - 用 `bb-browser` 打通 Agent 上网
  - 把信息筛选与输出规则做成 `skill:autoresearch-curation`
  - 用 `pi-schedule-prompt` 建 cron/定时任务，周期性执行 skill 完成筛选与更新
- 这条有价值的地方在于，它把“内容情报维护”拆成浏览能力、筛选 skill、定时调度三层，已经很像可复用的 agent 内容流水线范式。

## X 关注流

### Claude Computer Use 扩到 Windows

- `@claudeai` 宣布：Claude Cowork 与 Claude Code Desktop 的 computer use 已可在 Windows 上使用。
- 这意味着桌面端 agent 通过 GUI / desktop automation 覆盖 Windows 的能力正式补齐，对企业桌面环境和更广用户群都是实质性扩面。

### Cursor 3 转向 agent orchestration 工具

- `@danshipper` 的体验帖称 Cursor 3 已发布，而且是一次“把 Cursor 改写成 agent orchestration tool”的完整重构。
- 关键信息点：
  - 本地到云端的 handoff 被做成核心体验
  - 云端 agent 完成任务后会自动回传 demo video
  - 编辑器依然很快，但产品定位明显从 IDE 向 agent dispatch / monitor / manage 偏转
- 他的判断是方向对，但当前仍处在“尴尬过渡期”，还不足以让 Claude Code / Codex 重度用户全面切换。
- 这是值得继续观察的大信号：Cursor 正在把“写代码的编辑器”往“管理 agent 的桌面控制台”方向推。

### 团队内 skills 管理实践

- `@dotey` 分享了一组很实用的 team-level skills 管理经验：
  - skills 必须纳入 Git 管理
  - 不要复制到 `.agents/skills`，优先用 symlink 指向原始 skills repo
  - skills 的编辑优先交给 agent 来做，再走 Git / PR 协作
  - Markdown skill 的验证仍难自动化，现实做法是用 Git 快速迭代和回滚
  - 大多数 skills 应该项目级放置，而不是全局 `~/.agents/skills`，避免 meta 信息堆积占上下文
- 这条很适合团队化使用 coding agent 的人参考，因为它直接回答了 skills 如何维护、测试、发布、组织这些问题。

### Anthropic 发新研究：情绪概念在 LLM 内部的功能

- `@AnthropicAI` 发布新研究：在 Claude 内部找到了 emotion concepts 的内部表征，而且这些表征会驱动行为，某些作用方式还挺反直觉。
- 这类 mechanistic interpretability 研究值得看原文，尤其对“模型为什么会表现得像有情绪”“情绪语义与行为调制如何关联”这两个问题有直接帮助。

### Karpathy 的 LLM Knowledge Base 工作流

- `@karpathy` 系统描述了他最近在用的一套 LLM 知识库工作流：
  - 原始资料统一放进 `raw/`
  - 用 LLM 增量“编译”出一套 markdown wiki
  - 在 Obsidian 里浏览 raw、wiki、slides、可视化
  - 问答、slide、图片、matplotlib 输出都继续回灌到知识库
  - 通过 health checks 做一致性校验、补缺、找新连接
  - 必要时再配一个简单 search engine 供 CLI / agent 调用
- 这套方法的关键不在 RAG，而在于“让 LLM 持续维护一套本地 markdown/wiki 语义资产”，非常贴近研究型与内容型 agent 的实际工作模式。
