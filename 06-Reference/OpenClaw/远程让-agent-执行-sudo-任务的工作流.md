# 远程让 agent 执行 sudo 任务的工作流

## 结论

对当前这套协作方式，最稳的默认流程是：

> **交互式 CLI / 选择型 TUI / sudo 任务，优先用 tmux 承载；如果预计会用到 sudo，优先预制 sudo 票据。**

不要默认设计成“等人回到电脑前再输入密码”。这不适合远程协作场景。

## 什么时候优先用 tmux

### 适合 tmux 的场景

- 菜单选择型 CLI
- 需要方向键 / Tab / Enter / Space 的 TUI
- 会反复确认、翻页、回退的安装器
- 长时间运行但中途还可能交互的命令
- 需要 sudo 的一串系统操作

### 不一定需要 tmux 的场景

- 只问一次 `y/n`
- 只输入一次 token
- 一次性非常短的 prompt

这类可以用 `exec + pty`，但如果不确定，默认用 tmux 更稳。

## sudo 的默认策略

### 当前更适合的默认

如果任务预计需要 sudo：

1. 先预制 sudo 票据
2. 再让 agent 在 tmux 里继续执行

这样能避免执行到一半卡在密码提示。

## 如何预制 sudo 票据

最直接：

```bash
sudo -v
```

作用：

- 验证一次 sudo 密码
- 在短时间内缓存 sudo 凭据
- 后续 `sudo xxx` 通常无需再次输入密码

## 如何让票据续命

如果预计是一段较长时间的系统操作，可以先执行：

```bash
sudo -v
( while true; do sudo -n true; sleep 30; done ) &
```

说明：

- `sudo -v`：先拿到票据
- `sudo -n true`：无交互刷新票据
- `sleep 30`：每 30 秒续一次

这适合：

- 一段连续的 agent 运维操作
- 长安装过程
- 多条 sudo 命令串行执行

## 如何停止续命

```bash
kill %1
```

或者关闭对应 shell / pane。

## 如何手动让票据失效

```bash
sudo -k
sudo -K
```

通常：

- `sudo -k`：让当前票据失效
- `sudo -K`：更彻底地清理

## 推荐工作流

### 工作流 A：短时间 sudo 任务

适合：

- 改几个系统文件
- 跑少量 root 命令
- 短安装流程

流程：

1. 先执行 `sudo -v`
2. agent 在 tmux 中执行后续命令
3. 完成后自然超时或手动 `sudo -k`

### 工作流 B：长时间 sudo 任务

适合：

- 系统级安装
- 一串需要 root 的运维操作
- 长流程排障

流程：

1. 先执行 `sudo -v`
2. 后台启动续命循环
3. agent 在 tmux 中完成整段操作
4. 结束后停止续命并按需 `sudo -k`

## 为什么这样最稳

因为远程 agent 场景下，真正的问题不是“tmux 能不能 sudo”，而是：

- 会不会中途卡住等密码
- 人是不是正好在电脑前
- 交互状态能不能持续接管
- 操作中断后能不能恢复上下文

tmux + sudo 票据 的组合，解决的是：

- 会话可持续
- 菜单 / 交互可接管
- sudo 不容易在中途打断流程

## 一个简单决策规则

可以收口成三条：

- **输入型短交互** → `exec + pty`
- **选择型 / TUI / 长交互** → `tmux`
- **需要 sudo 的连续任务** → `tmux + 预制 sudo 票据`

## 边界提醒

虽然技术上 agent 可以在 tmux 里执行 sudo，但仍应区分风险：

### 低风险

- 安装工具
- 改非核心配置
- 查看系统状态
- 启停服务

### 高风险

- 批量删改系统文件
- 改 SSH / 防火墙 / 安全配置
- 替换关键服务配置
- 可能导致机器失联的操作

高风险任务不应该因为“sudo 可用”就省略边界确认。

## 对当前协作方式的实际结论

对贵平这种经常不在电脑前、希望 agent 远程推进的场景：

> **不要把“sudo 等人工回来输入密码”当默认方案。**

更好的默认是：

- 预计要 sudo 时，先预制票据
- 复杂交互放进 tmux
- agent 在边界明确后继续推进

这样远程协作的中断最少，效率最高。
