# 如何让你的 Claude 活得更久：网络篇｜共读笔记（2026-03-30）

## 1. DNS 泄漏为什么危险

这篇文章的一个核心判断是：

**“能连上”不等于“风控安全”。**

如果一个用户的网络画像是：

- HTTP / HTTPS 出口 IP 看起来在美国
- 但 DNS 查询仍然从中国本地或公司网络发出

那么这条链路对风控系统来说就是不自洽的。

### 我们对这一点的共识

1. **DNS 泄漏是真问题。**
   - 代理了业务流量，不代表 DNS 解析也被代理了。
   - 如果 DNS 仍在本地网络解析，就会暴露访问意图和地区特征。

2. **风控看的是整条网络画像是否一致。**
   - 真正危险的不是单一出口 IP，而是“美国出口 + 中国 DNS”这种矛盾画像。

3. **`socks5h` 很关键。**
   - `socks5://` 常常是本地先解析域名，再通过 SOCKS 转发连接。
   - `socks5h://` 更接近把域名解析也交给代理端完成。

### 我们的保留意见

文章里把 `dns: enable: false` 称为“终极解法”，这个判断在作者环境里可能成立，但不一定适合直接推广到所有环境。

原因是实际效果还取决于：

- 系统 DNS 如何配置
- VPN 是否劫持 DNS
- Clash / Mihomo 在当前模式下如何处理解析
- 应用是浏览器、Electron、CLI、Node，还是 curl
- 代理协议到底是 HTTP、SOCKS，还是 `socks5h`

所以更准确的表述应该是：

> **DNS 泄漏是关键风险点；而“关掉 Clash DNS”只是特定环境中的一种有效规避方案。**

## 2. 家用网络下，系统代理或 TUN 能不能规避这个问题？

### 先说结论

在家用网络里，这个问题会比“公司 VPN + 企业 DNS”简单很多，但：

- **开系统代理 ≠ 一定没有 DNS 泄漏**
- **开 TUN 通常更容易统一整条链路，包括 DNS**
- 真正关键的不是“有没有开 TUN”，而是**DNS 最终在哪解析**

### A. 只开系统代理

系统代理主要解决的是：

- 应用连接如何走代理

但它不必然自动解决：

- 域名解析在哪里做

这取决于：

- 应用是否读系统代理
- 应用自己怎么实现网络请求
- 是 HTTP 代理、SOCKS 代理，还是 `socks5h`
- 系统和代理软件怎么处理 DNS

所以系统代理能改善很多问题，但**不能天然保证 DNS 一致**。

### B. 开 TUN

TUN 更像是在网络层接管流量：

- 整机更多流量进入代理栈
- 如果 DNS 规则也配置正确，通常更容易做到流量和 DNS 一起统一

因此在家用网络下，从“减少 DNS 泄漏”的角度看：

**TUN 通常比单纯系统代理更彻底。**

### 为什么原文反对 TUN？

原文反对的不是 TUN 这项技术本身，而是它在特定场景下的副作用。

文章讨论的是：

- 公司 VPN
- AI 代理
- 两套网络体系共存

在这个约束下，TUN 的问题是它太强了，容易：

- 接管整机流量
- 和公司 VPN 抢底层控制权
- 把不该代理的办公流量也卷进去

所以文章真正想表达的是：

> **在“公司 VPN 共存”这个约束下，TUN 不是最稳的解法。**

而不是：

> “TUN 技术上比系统代理更差。”

### 我们最后的总结

#### 家用网络
- 如果目标是**整机统一、尽量减少 DNS 不一致**：
  - **TUN + 正确 DNS 配置** 更容易一步到位
- 如果目标是**更克制、只代理特定应用**：
  - **系统代理 + 规则代理 + CLI 单独配 proxy** 也可以，但要格外注意 DNS 是否仍在本地解析

#### 公司网络 / 公司 VPN
- 不应该简单照搬家用网络那套“全局接管”思路
- 文章推荐的“不启用 TUN、只代理必要 AI 域名”在这个场景下更有现实意义

---

## 3. 当前阶段最实用的 takeaway

如果只记最关键的几句：

1. **代理不等于 DNS 也被代理了。**
2. **风控看的是整条网络画像是否自洽。**
3. **家用网络下，TUN 更容易统一链路；公司 VPN 共存时，TUN 不一定是最稳的选择。**
4. **真正需要确认的，不是“开了什么模式”，而是“DNS 最终在哪解析”。**
