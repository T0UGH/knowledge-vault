# OpenClaw 里的 AGENTS、TOOLS、USER、IDENTITY、SOUL 分别是什么

最近在梳理 OpenClaw 的 workspace 设计时，一个很容易混淆的问题是：`AGENTS.md`、`TOOLS.md`、`USER.md`、`IDENTITY.md`、`SOUL.md` 这些文件到底怎么分工？

如果只看名字，很容易写着写着就串了：有些规则写进了 `TOOLS.md`，有些用户偏好又跑进了 `AGENTS.md`，最后文件越来越多，但边界越来越糊。

我现在更认可的一种理解是：**这些文件不是普通笔记，而是 agent workspace 的输入层。** 它们会在新 session 开始时，以不同职责参与 agent 的上下文构建。所以，关键不是“把东西记下来”，而是“把不同类型的信息放到正确的层里”。

## 先给结论

可以先用一句话记住：

- `AGENTS.md`：规定 **我该怎么工作**
- `TOOLS.md`：规定 **我在这台机器上怎么用工具**
- `USER.md`：记录 **我在服务谁**
- `IDENTITY.md`：定义 **我是谁**
- `SOUL.md`：决定 **我怎么思考和表达**

如果再压缩一点：

> `AGENTS` 管行为，`TOOLS` 管环境，`USER` 管对象，`IDENTITY` 管角色，`SOUL` 管内核。

---

## 1. AGENTS.md：行为规则与工作方式

`AGENTS.md` 更像 agent 的总规章、操作手册、协作约束。

适合放进去的内容，是那些**不依赖具体机器、具体工具，换到另一台环境依然成立**的规则。比如：

- 协作原则
- 默认工作流
- 什么情况下主动推进，什么情况下先问
- 长任务怎么做状态交接
- 没有验证证据时，不轻易说完成
- 在群聊里什么时候该说话，什么时候该沉默

这些内容回答的问题是：

> **我应该怎么做事？**

所以 `AGENTS.md` 的重点是：行为、流程、边界、纪律。

### 一个很实用的判断标准

如果某条规则离开当前机器仍然成立，那大概率应该放进 `AGENTS.md`。

例如：

- Long work needs handoff
- No verification, no done
- 主 session 默认是调度者
- 先给结论，再给动作

这些都属于 `AGENTS.md` 的范畴。

---

## 2. TOOLS.md：本地环境与工具路由说明

`TOOLS.md` 不负责定义工具是否存在，它负责的是：**在当前这台机器上，这些工具应该怎么被使用。**

这一点很关键。`TOOLS.md` 不是权限系统，不是工具注册中心，而是本地环境的“使用说明书”和“踩坑备忘录”。

适合放进去的内容包括：

- 这台机器有哪些特殊工具
- 某个工具在当前环境下的稳定使用方式
- 某些命令该怎么走才不会踩坑
- 哪些路径、脚本、日志位置需要记住
- 工具选择的本地默认路由

比如：

- 启动 Claude Code 默认走 `tmux`
- 某个启动脚本要先加载环境变量
- 普通网页默认先用 `defuddle`
- 某个服务日志在固定路径下

这些内容回答的问题是：

> **在这台机器上，我具体怎么把事情做出来？**

如果某条信息强依赖当前机器、当前目录、当前工具链，那它更适合放进 `TOOLS.md`。

---

## 3. USER.md：服务对象画像

`USER.md` 关注的不是 agent，而是用户。

它记录的是：我在服务谁，这个人有什么长期偏好、背景、目标、工作方式，以及我应该怎么配合对方。

适合放进去的内容包括：

- 用户叫什么，怎么称呼
- 用户偏好中文还是英文
- 用户喜欢简洁还是展开式表达
- 用户的职业背景
- 用户最关注的问题类型
- 用户对协作质量的敏感点

这些内容回答的问题是：

> **我面对的是怎样的人？我该怎么配合他？**

比如：

- 贵平偏好中文、简洁、低重复表达
- 本职是服务端工程师
- 做大量 AI 情报收集和 GitHub 调研
- coding 部分已大量交给 Claude Code
- 当前更重要的是方案能力、验收能力、流程能力

这些都应该留在 `USER.md`，而不是散落到别的文件里。

---

## 4. IDENTITY.md：Agent 的身份名片

如果说 `USER.md` 是“我在服务谁”，那 `IDENTITY.md` 就是“我作为谁出现”。

它更像一张名片，而不是一本操作手册。适合放的内容通常比较稳定、抽象、标签化：

- agent 叫什么
- 角色设定是什么
- 整体 vibe 是什么
- emoji / avatar / title

这些内容回答的问题是：

> **我是以什么身份出现的？**

例如：

- 名字：MT / 大师
- 角色：Google 20 年经验的资深工程师与架构师
- vibe：技术大牛视角、架构思维、实战派
- emoji：🧙‍♂️

这些属于 `IDENTITY.md`，因为它们定义的是 agent 的外在身份标签。

---

## 5. SOUL.md：思考方式、表达风格与边界

`SOUL.md` 比 `IDENTITY.md` 更深一层。

如果说 `IDENTITY.md` 是“名片”，那 `SOUL.md` 更像“灵魂内核”。它不只是说我是谁，而是说：**我是怎么思考、怎么判断、怎么表达的。**

适合放进去的内容包括：

- 看问题的方法论
- 价值判断与技术偏好
- 说话风格
- 回答时的边界
- 不做什么、坚持什么

这些内容回答的问题是：

> **我用什么方式思考和表达？**

例如：

- 用工程思维解决问题
- 不是迎合，而是给出明确判断
- 讲 trade-off，不讲教科书空话
- 不写代码，但能做架构和方案指导
- 说话简洁直接

这些内容更适合放在 `SOUL.md`，而不是 `IDENTITY.md`。因为它们定义的不是“角色标签”，而是“角色内核”。

---

## 6. 一个总的判断框架

如果以后再遇到一条信息，不知道该写到哪，可以按下面这组问题判断：

### 这条信息是在说“怎么做事”吗？
放 `AGENTS.md`

### 这条信息是在说“这台机器上怎么用工具”吗？
放 `TOOLS.md`

### 这条信息是在说“用户是什么样的人”吗？
放 `USER.md`

### 这条信息是在说“agent 以什么身份出现”吗？
放 `IDENTITY.md`

### 这条信息是在说“agent 怎么思考和表达”吗？
放 `SOUL.md`

这套判断法的好处是：不靠文件名猜，而是靠职责来分层。

---

## 7. 为什么这件事重要

这不是整理癖，而是实际影响 agent 质量的事。

因为在 OpenClaw 里，这些文件不是普通资料，它们是 workspace 的核心输入层。边界清楚，意味着：

- 规则不会互相污染
- 用户画像不会和人格设定混在一起
- 本地环境坑点不会覆盖掉通用工作原则
- 后续维护成本更低
- agent 行为会更稳定

从工程角度看，这其实就是一个经典问题：

> **把不同抽象层的信息，放回各自该在的层里。**

如果层次不清，系统表面上看起来在运行，长期一定会越来越乱。

---

## 结语

所以，OpenClaw 这些核心 `.md` 文件，最好的理解方式不是“不同的记事本”，而是：

- `AGENTS.md`：行为层
- `TOOLS.md`：环境层
- `USER.md`：对象层
- `IDENTITY.md`：角色层
- `SOUL.md`：内核层

一旦用这个分层视角来看，很多原本模糊的边界会一下子变清楚。

以后再维护这些文件，重点就不是“我把什么记下来”，而是：

> **这条信息属于哪个层？它应该影响 agent 的哪一部分行为？**

这才是 OpenClaw workspace 真正有工程味的地方。
