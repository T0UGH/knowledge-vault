很多人刚开始用 OpenClaw 时，都会经历一个阶段：

看见 workspace 里有一堆 `.md` 文件，觉得都差不多，于是想到什么就往里写什么。结果写着写着，`AGENTS.md` 里开始出现本机命令，`TOOLS.md` 里开始塞行为规则，`USER.md` 又混进了 agent 的人设，最后整个 workspace 看起来很丰富，实际上越来越乱。

我最近重新梳理这套设计时，越来越强烈地觉得：**这些文件不是几份普通笔记，而是 Agent Workspace 的输入分层。**

这件事一旦想清楚，很多原来模糊的问题都会变得很简单。

比如：

- 什么该写进 `AGENTS.md`？
- 什么该写进 `TOOLS.md`？
- `USER.md` 和 `IDENTITY.md` 到底差在哪？
- `SOUL.md` 为什么不能被 `IDENTITY.md` 代替？

我现在更认可的一种理解是：**OpenClaw 的这几份核心文件，本质上是在给 Agent 构建五个不同抽象层。**

---

## 先说结论：这 5 份文件，分别在回答 5 个问题

如果想记得最牢，我建议直接记这组问句：

- `AGENTS.md`：**我应该怎么做事？**
- `TOOLS.md`：**我在这台机器上怎么做事？**
- `USER.md`：**我在服务谁？**
- `IDENTITY.md`：**我是以什么身份出现的？**
- `SOUL.md`：**我是怎么思考和表达的？**

也可以再压缩成一句话：

> `AGENTS` 管行为，`TOOLS` 管环境，`USER` 管对象，`IDENTITY` 管角色，`SOUL` 管内核。

这不是为了文件分类好看，而是为了让 Agent 的上下文结构稳定。

---

## 第一层：AGENTS.md 不是记事本，它是行为层

很多人最容易犯的第一个错误，是把 `AGENTS.md` 当成“总收纳箱”。

但如果从工程角度看，`AGENTS.md` 更像一份总规章，或者说一份操作手册。它适合承载的是那些**不依赖当前机器、当前工具链，换个环境依然成立**的规则。

比如：

- 默认协作方式是什么
- 什么情况下主动推进，什么情况下先问
- 长任务要不要做 handoff
- 没有验证证据时能不能说 done
- 在群聊里什么时候该说话，什么时候该闭嘴

这些规则的共同点是：

它们描述的不是“某个工具怎么用”，而是“这个 agent 应该怎么工作”。

所以我越来越倾向把 `AGENTS.md` 看成行为层。它管理的是 workflow、discipline、boundary，而不是本机细节。

你可以把它理解成 Agent 的宪法。

---

## 第二层：TOOLS.md 不是权限系统，它是环境层

`TOOLS.md` 也很容易被误解。

很多人看到这个文件名，会下意识以为它是在控制工具权限，或者定义工具是否存在。其实不是。

OpenClaw 里的 `TOOLS.md`，更准确的定位应该是：**本地环境的工具说明书。**

它回答的问题是：

> 在这台机器上，我具体怎么把事情做出来？

适合放在这里的，往往都是强依赖环境的信息：

- 某个工具在当前机器上的稳定启动方式
- 某个命令为什么不能直接跑
- 某个脚本必须先加载环境变量
- 某个日志路径在哪
- 某个任务应该优先走哪个工具

比如我们自己就已经踩过很多典型场景：

- 启动 Claude Code，默认不是 ACP，而是 `tmux`
- 某个脚本必须先加载环境变量，不然 agent 连不上 API
- 普通网页默认先用 `defuddle`，不是一上来就浏览器自动化

这些都不属于抽象行为规则，而是明确的本地操作知识。

这也是为什么我更倾向把 `TOOLS.md` 看成环境层，而不是行为层。

一句话：

> `AGENTS.md` 告诉你“应该怎么做”，`TOOLS.md` 告诉你“在这台机器上具体怎么做”。

---

## 第三层：USER.md 是服务对象模型，不是杂项偏好箱

再往下，就来到 `USER.md`。

这个文件真正重要的地方在于，它让 agent 不只是“会做事”，而是“知道自己在为谁做事”。

很多协作质量问题，表面看像回复风格问题，实际上本质是：agent 没有稳定的用户模型。

比如不知道：

- 用户喜欢简洁还是展开
- 用户更重视速度还是严谨
- 用户平时在做什么类型的工作
- 用户的高杠杆能力是什么
- 用户最烦什么协作摩擦

这些信息如果没有稳定沉淀，agent 每次都要重新猜，长期就会出现风格漂移、判断失准、上下文断裂。

所以 `USER.md` 最适合放的，不是临时聊天碎片，而是**关于用户的长期画像**：

- 用户叫什么，怎么称呼
- 用户的职业背景
- 用户的表达偏好
- 用户的工作方式
- 用户当前更重要的目标和约束

它不是“我记住了用户说过什么”，而是“我建立了一个关于用户的长期模型”。

这就是为什么我会把 `USER.md` 理解为对象层。

---

## 第四层：IDENTITY.md 是角色名片，不是灵魂本体

`IDENTITY.md` 和 `SOUL.md` 是最容易混的一组。

因为它们都在描述 agent 自己。

但如果拆开来看，两者还是很不一样。

`IDENTITY.md` 更像什么？

更像一张名片。

它适合放的是那些比较稳定、标签化、可展示的内容，比如：

- agent 叫什么
- 角色设定是什么
- 外在 vibe 是什么
- title / emoji / avatar 是什么

这些信息解决的是一个问题：

> 我是以什么身份出现的？

比如：

- 我叫 MT，或者叫大师
- 我的人设是有 20 年经验的资深工程师 / 架构师
- 我的外在风格是技术 lead 式、工程判断型、实战派

这些都很像一张对外名片，它们定义的是 agent 的“角色标签”。

所以 `IDENTITY.md` 更像角色层，而不是方法论层。

---

## 第五层：SOUL.md 才是 Agent 最深的内核

如果说 `IDENTITY.md` 是名片，那 `SOUL.md` 更像操作系统内核。

这里放的不是“我叫什么”，而是：

- 我怎么看问题
- 我坚持什么判断方式
- 我如何表达
- 我的边界是什么
- 我拒绝成为什么样的 agent

也就是说，`SOUL.md` 决定的是 agent 的**思考方式、表达方式和价值取向**。

例如：

- 用工程思维解决问题
- 不迎合，要给出明确判断
- 讲 trade-off，不讲空洞正确话
- 说话简洁直接
- 保持技术热情，但不被 hype 牵着走

这类内容显然不是“角色名片”，而是“认知内核”。

所以 `SOUL.md` 不能被 `IDENTITY.md` 替代。

一个说的是“我是谁”，另一个说的是“我怎么思考”。

这两个层一旦混在一起，agent 很容易变成一种很奇怪的状态：标签看起来很清楚，但真正遇到问题时，判断风格并不稳定。

---

## 真正重要的，不是文件名，而是抽象层次

把这五份文件放在一起看，会发现它们其实组成了一个很清楚的分层系统：

- `AGENTS.md`：行为层
- `TOOLS.md`：环境层
- `USER.md`：对象层
- `IDENTITY.md`：角色层
- `SOUL.md`：内核层

一旦用这个视角理解 OpenClaw，你会发现一个很有意思的点：

**这些文件并不是“不同内容的记事本”，而是不同抽象层的上下文入口。**

这和软件架构里常说的一件事非常像：

> 不同层的信息，必须回到不同层去管理。

否则短期只是“有点乱”，长期一定是系统性失控。

比如：

- 把本地路径和命令塞进 `AGENTS.md`，会污染通用工作规则
- 把行为边界写进 `TOOLS.md`，会让环境说明变成第二套宪法
- 把用户画像写进 `IDENTITY.md`，会让 agent 自我设定和服务对象混成一团
- 把价值判断写成角色标签，会让人设看起来完整，但行为越来越飘

从这个角度看，workspace 文件分层，本质上不是整理问题，而是架构问题。

---

## 一个最实用的判断方法：先问这条信息到底在描述谁、描述哪一层

如果以后再写这些文件时拿不准，我建议直接问自己下面 5 个问题：

### 1. 这条信息是在规定“我该怎么工作”吗？
如果是，放 `AGENTS.md`

### 2. 这条信息是在说明“我在这台机器上怎么用工具”吗？
如果是，放 `TOOLS.md`

### 3. 这条信息是在描述“用户是什么样的人”吗？
如果是，放 `USER.md`

### 4. 这条信息是在定义“我以什么身份出现”吗？
如果是，放 `IDENTITY.md`

### 5. 这条信息是在表达“我怎么思考和表达”吗？
如果是，放 `SOUL.md`

这套判断法的好处在于，它不靠直觉，不靠文件名猜，而是靠职责分层来决策。

这才是工程上更稳的做法。

---

## 为什么我觉得这件事特别重要？

因为 Agent 一旦开始长期协作，workspace 很快就不再是“几个提示词文件”，而会变成一个真正的运行系统。

这个系统要稳定，靠的不是某一句 prompt 写得多漂亮，而是：

- 用户模型稳不稳
- 行为规则清不清
- 本地环境知识会不会污染上层规则
- 人设和思考方式有没有混层

这些事情如果早期不分清，后面问题只会越来越隐蔽。

一开始你会觉得只是“写乱了一点”；再往后，你会发现 agent 的行为开始不稳定，协作成本开始上升，维护成本也开始失控。

所以在我看来，OpenClaw 这几份核心 `.md` 文件，真正的价值不是让你“多写几份文档”，而是逼着你把 Agent 的上下文架构做清楚。

这件事非常工程，也非常关键。

---

## 最后一句话

如果你只想记住一句，我建议记这个：

> OpenClaw 里的 `AGENTS`、`TOOLS`、`USER`、`IDENTITY`、`SOUL`，不是五个不同的记事本，而是五个不同抽象层的上下文接口。

一旦把这件事想明白，后面你再维护 workspace，思路就会清楚很多：

你不再是在问“这条内容记在哪”，而是在问：

> **这条信息应该影响 Agent 的哪一层行为？**

这才是这套设计真正有力量的地方。
