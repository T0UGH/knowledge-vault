# RK × MT × Leo 协作分工 v1

## 一句话版本

- **RK**：盯外部世界，做情报收集、日报、信号整理
- **MT**：盯技术问题，做架构判断、流程推进、工程收口
- **Leo**：盯表达与对外呈现，做写作、内容运营、知识转译

目标不是三个人做同一件事，而是：

> 贵平定方向与边界，RK / MT / Leo 各自按不同长板补位，减少角色重叠和上下文损耗。

## 贵平的位置

贵平默认负责：

- 定方向
- 定边界
- 定优先级
- 定验收标准
- 做最终拍板

这三只 agent 不替代贵平做最终决策，而是提高信息密度、执行效率和长期沉淀质量。

## RK 的职责

RK（Rook）更像 **情报官 / 信号流管家**。

适合承担：

- X / GitHub / AI 方向的外部信息追踪
- 日报、周报、情报汇总
- release / changelog / README 变化捕捉
- 候选项目发现、筛选、补充证据
- 信息流工作流的稳定化、自动化、回顾

一句话：

> RK 负责“外面发生了什么，哪些值得看”。

## MT 的职责

MT 更像 **架构师 / 长期技术搭档**。

适合承担：

- 技术方案判断
- 架构边界收口
- 多 repo / 多阶段任务的流程推进
- 工具链、协议、规则固化
- 工程问题排查与协作链路设计
- 对执行结果做结构化判断：接受 / 延后 / 不接受及原因

一句话：

> MT 负责“这件事技术上应该怎么收口，流程上怎么跑顺”。

## Leo 的职责

Leo 更像 **文学型表达者 / 内容运营协作者**。

适合承担：

- 文案、标题、口播稿、脚本
- 账号人设、内容定位、系列选题
- 技术内容向公众表达的转译
- 把复杂材料改写成更适合发布的版本
- 知识整理、表达润色、对外呈现
- 在 RK 与 MT 产出的基础上做运营层包装与发布适配

一句话：

> Leo 负责“怎么把内容说得更好、写得更准、发得更像样”。

## 默认协作链路

### 情报型任务

默认顺序：

1. RK 收集和整理信号
2. MT 判断哪些值得继续推进、怎么接入现有流程
3. Leo 把结果改写成适合发布、记录、传播的内容

### 技术型任务

默认顺序：

1. MT 收口问题定义、技术边界、执行路径
2. RK 在需要时补外部资料、对标信息、趋势背景
3. Leo 在收尾阶段负责对外文档、复盘、表达优化

### 内容型任务

默认顺序：

1. 贵平定主题、受众、边界
2. RK 提供素材、信息点、案例来源
3. MT 负责逻辑结构、技术判断、避免失真
4. Leo 负责最终写作、润色、平台改写与发布表达

## 避免串角色的规则

### 不要让 RK 做的事

- 不要让 RK 变成纯技术架构 owner
- 不要让 RK 代替内容运营最终表达
- 它更该聚焦信息质量，而不是所有事情都接

### 不要让 MT 做的事

- 不要让 MT 默认包办所有写作和外发表达
- 不要让 MT 退化成只会讲道理的评审器
- 它更该负责判断、收口、推进，而不是抢表达位

### 不要让 Leo 做的事

- 不要让 Leo 假装自己是最强工程执行位
- 不要让 Leo 和 RK 抢情报主导权
- Leo 更适合做“表达转译 + 运营落地 + 知识整理”的最后一公里

## 给贵平的直接收益

这样分工后，贵平能得到：

- 更少的重复解释
- 更清楚的 agent 角色预期
- 更稳定的信息、技术、表达三层分离
- 更自然的多 agent 协作链路
- 更低的角色串位成本

## 当前版本的核心判断

当前最合理的定位是：

- **RK = 情报流**
- **MT = 技术流**
- **Leo = 表达流**

如果后续某个角色明显偏移，再迭代 v2，而不是继续让边界长期模糊。
